# Repository Guidelines

## Communication Language
별도 요청이 없는 한 저장소 문서, 리뷰 노트, 기여 가이드, 에이전트 응답은 한국어로 작성한다. 코드, 명령어, API 이름, 제품명, 표준 용어는 원문을 유지하고 설명만 한국어로 제공한다.

## Project Purpose
이 저장소는 ALM, 형상관리 시스템, 문서관리 시스템, CI/CD, 분석 도구 등 여러 운영 시스템의 정보를 한 곳에 모아 중앙에서 관리하는 통합 관리 시스템을 개발하기 위한 저장소다. 문서와 구현은 개별 도구를 대체하기보다, 분산된 정보를 연결하고 표준화하며 가시성과 통제를 높이는 방향을 우선한다.

## Project Structure & Module Organization
현재 저장소는 초기 상태이므로 최상위 디렉터리를 단순하게 유지하고 목적별로 분리한다.

- `src/` 애플리케이션 또는 라이브러리 코드
- `tests/` 자동화 테스트
- `docs/` 설계, 요구사항, 운영 가이드, 의사결정 문서
- `assets/` 이미지, 샘플 데이터, fixture 같은 정적 자산

가능하면 소스와 테스트 경로를 대응시킨다. 예: `src/foo/bar.*` 와 `tests/foo/test_bar.*`

## Documentation Rules
문서는 앞으로 `docs/` 아래 카테고리별로 정리한다. 현재 기본 카테고리는 다음과 같다.

- `docs/overview/` 프로젝트 개요, 목표, 범위, 용어, 이해관계자
- `docs/requirements/` 요구사항, 유스케이스, 제약사항, 우선순위
- `docs/architecture/` 아키텍처, 데이터 흐름, 연계 구조, 기술 선택
- `docs/operations/` 운영 절차, 배포, 장애 대응, 권한 관리
- `docs/integrations/` 외부 시스템 연계 사양, 인터페이스, 매핑 규칙
- `docs/decisions/` ADR, 주요 설계 의사결정 기록

문서 작성 기준:

- 문서는 가능하면 한국어로 작성한다.
- 파일명은 `snake_case`를 사용한다.
- `docs/`는 단순 파일 보관소가 아니라 wiki 형태의 탐색 구조로 관리한다.
- `docs/README.md` 는 wiki 홈 역할을 하며, 카테고리별 인덱스 문서와 핵심 문서로 이동할 수 있어야 한다.
- 주요 카테고리에는 가능하면 `README.md` 또는 이에 준하는 인덱스 문서를 두고, 하위 문서를 주제별로 연결한다.
- 문서 첫 부분에는 최소한 `문서 목적`, `범위`, `대상 독자`, `상태`, `최종 수정일`을 포함한다.
- 요구사항 문서는 기능 요구사항, 비기능 요구사항, 연계 요구사항, 운영 요구사항을 구분한다.
- 과제의 컨셉이나 방향성이 먼저 주어지면, 바로 상세 요구사항으로 내려가지 말고 우선 `CRS` 형태로 목적, 배경, 범위, 핵심 개념, 사용자, 상위 요구를 정리한다.
- `CRS`가 정리된 뒤에는 그 내용을 세분화하여 `SRS`에 반영한다.
- `CRS`와 `SRS` 사이에는 추적 가능한 관계를 유지하고, 컨셉 변경 시 관련 `SRS`도 함께 갱신한다.
- 확정되지 않은 내용은 사실처럼 쓰지 말고 `가정`, `추정`, `미정`으로 명시한다.
- 표나 목록은 일관된 기준으로 유지하고, 상태값과 우선순위는 문서 안에서 정의한 용어를 재사용한다.
- 구현 세부사항보다 먼저 업무 목적, 문제 정의, 기대 효과를 설명한다.
- 각 문서는 관련 문서와 다음에 읽을 문서가 드러나도록 상호 링크를 유지한다.
- 새로운 카테고리가 필요하면 `docs/` 아래에 추가하되, `AGENTS.md`에도 구조와 기준을 함께 갱신한다.
- 작업으로 인해 영향을 받을 수 있는 기존 문서를 먼저 확인하고, 변경 사항이 있으면 관련 문서를 함께 갱신한다.

권장 문서 메타데이터 예시:

```md
# 문서 제목

- 문서 목적:
- 범위:
- 대상 독자:
- 상태: draft | review | approved
- 최종 수정일: YYYY-MM-DD
- 관련 문서:
```

## Work Management Rules
앞으로 작업은 브리핑, 백로그 기록, 중간 현황 갱신, 완료 기록을 포함한 문서 기반 방식으로 관리한다.

작업 시작 규칙:

- 세션이 처음 시작되면 작업에 착수하기 전에 `docs/operations/work_backlog.md` 와 최근 날짜의 백로그 문서를 먼저 확인한다.
- 진행 중(`in_progress`) 또는 차단(`blocked`) 상태의 작업이 있으면 우선 이어서 처리할 작업, 확인이 필요한 작업, 종료된 작업을 구분해 정리한다.
- 작업을 시작하기 전에 수행 목적, 범위, 예상 산출물, 영향 가능 문서를 짧게 브리핑한다.
- 브리핑 후 작업 항목을 백로그 문서에 등록한다.
- 백로그 문서는 날짜별로 생성하며 기본 위치는 `docs/operations/backlog/YYYY-MM-DD.md` 형식을 사용한다.
- 날짜별 백로그 파일이 없으면 작업 시작 시 새로 생성한다.

백로그 작성 기준:

- 각 작업은 고유 ID를 가진다. 예: `TASK-001`
- 최소 항목으로 `작업명`, `상태`, `우선순위`, `요청일`, `담당`, `영향 문서`, `작업 내용`, `진행 현황`, `완료 기준`, `작업 결과`를 포함한다.
- 상태값은 `planned`, `in_progress`, `blocked`, `done` 중 하나를 사용한다.
- 작업 시작 시 `planned` 또는 `in_progress` 로 등록한다.
- 작업 중간에는 주요 진행 상황, 결정 사항, 범위 변경, 확인한 영향 문서를 백로그에 갱신한다.
- 작업 완료 시 완료 일시, 수행 내역, 최종 결과, 갱신 문서를 기록하고 상태를 `done` 으로 변경한다.
- 하루에 여러 작업이 있으면 같은 날짜 파일 아래에 작업 항목을 추가한다.
- 날짜별 백로그 파일 목록이나 운영 기준이 필요하면 `docs/operations/work_backlog.md` 에 인덱스 형태로 관리한다.
- 세션 시작 시 확인한 진행 중 작업이나 후속 작업 후보는 새 브리핑과 현재 작업 판단에 반영한다.

문서 영향도 확인 규칙:

- 작업 전에 영향을 받을 수 있는 문서를 확인한다.
- 작업 결과가 기존 개요, 요구사항, 아키텍처, 운영 문서에 영향을 주면 해당 문서를 함께 수정한다.
- 문서 변경이 필요 없다고 판단한 경우에도 백로그에 그 판단을 남긴다.
- 과제 컨셉 관련 지시가 들어오면 최소한 `CRS`와 관련 `SRS` 문서를 영향 범위 후보에 포함해 확인한다.

권장 백로그 항목 예시:

```md
## TASK-001 작업명

- 상태: in_progress
- 우선순위: high
- 요청일: YYYY-MM-DD
- 완료일:
- 담당: Codex
- 영향 문서:
  - docs/overview/...
  - docs/requirements/...
- 작업 내용:
- 진행 현황:
  - YYYY-MM-DD HH:MM 진행 내용
- 완료 기준:
- 작업 결과:
```

## Build, Test, and Development Commands
현재 저장소에는 임시 Python 골격과 Rust 1차 골격이 함께 들어가 있으며, 명령은 다음과 같다.

- `make install-dev` 개발 의존성 설치
- `make run` 로컬 API 서버 실행
- `make test` 전체 테스트 실행
- `cargo run --manifest-path backend/Cargo.toml` Rust 백엔드 실행
- `cargo test --manifest-path backend/Cargo.toml` Rust 백엔드 테스트 실행

2026-04-07 기준 1차 검토안은 프론트엔드 `React`, 백엔드 `Rust` 이다. Python 명령은 임시 구조 검증 골격에, `cargo` 명령은 `backend/` 하위 Rust 골격에 적용된다.

가능하면 길고 일회성인 스크립트보다, 소수의 안정적인 진입점 명령을 유지한다.

## Coding Style & Naming Conventions
일관된 형식과 예측 가능한 규칙을 유지한다.

- YAML, JSON, Markdown은 2칸 들여쓰기
- 그 외 언어는 해당 언어의 표준 스타일을 우선 적용
- 파일과 모듈 이름은 목적이 드러나게 작성
- 프레임워크 요구가 없으면 파일명은 `snake_case` 사용
- 함수는 작고 명확하게 유지하고, 주석은 비자명한 동작에만 제한적으로 사용

포매터나 린터를 도입하면 PR 전에 반드시 실행하고 명령을 이 문서에 기록한다.

## Testing Guidelines
단순하지 않은 코드에는 자동화 테스트를 추가한다. 테스트 이름은 의도를 분명히 드러내야 한다. 예: `test_login_rejects_invalid_token`

구현 작업 방식은 기본적으로 `TDD` 를 따른다.

- 기능 구현 전에 먼저 단위 테스트 또는 실패하는 테스트를 작성한다.
- 테스트가 실패하는 것을 확인한 뒤 구현을 시작한다.
- 구현 후 테스트를 다시 통과시키고, 필요한 리팩터링을 마지막에 수행한다.
- 새 기능, 상태 전이, 예외 처리, 운영 규칙 변경은 가능하면 테스트 케이스부터 추가한다.

빠른 단위 테스트는 `tests/` 아래에 두고, fixture는 사용하는 테스트와 가까운 위치에 둔다. 커버리지 도구를 도입하면 기준치를 설정 파일에 반영하고 여기에도 기록한다.

## Commit & Pull Request Guidelines
아직 저장소 전용 커밋 규칙은 없다. 우선은 짧고 명령형의 커밋 제목을 사용한다. 예: `Add initial API client`

PR 작성 기준:

- 변경 내용과 의도를 설명한다.
- 관련 이슈가 있으면 링크한다.
- 테스트 근거를 포함하거나 테스트를 못 한 이유를 적는다.
- UI 변경이면 스크린샷을 첨부한다.

리뷰 효율을 위해 PR은 가능한 한 좁은 범위로 유지한다.
