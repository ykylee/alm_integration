## 1. 분류 원칙

- 시스템 분류는 제품명보다 업무 기능과 관리 책임을 기준으로 한다.
- 하나의 제품이 여러 카테고리에 걸칠 수 있으므로 주 카테고리와 보조 카테고리를 함께 가질 수 있다.
- 초기 제품 전략은 외부 전문 도구의 강점을 활용하되, 통합 시스템은 조회, 연결, 일부 관리 기능을 내부에서 제공하는 방향을 따른다.
- `SCM`, `CI/CD`, 정적분석 실행 엔진은 초기 내재화 대상에서 제외하고 연계 중심으로 관리한다.
- 코드리뷰 맥락, 빌드/산출물 메타데이터, 정적분석 결과와 품질 상태는 통합 시스템 내부에서 조회하거나 관리할 수 있도록 설계한다.

## 2. 권장 카테고리

### 2.1 요구사항 관리

- 카테고리 ID: `requirements_management`
- 정의: 요구사항, 기능 요청, 변경 요구, 상위 과제 정의를 관리하는 영역
- 대표 데이터: 요구사항 항목, 우선순위, 상태, 변경 이력

### 2.2 업무 관리

- 카테고리 ID: `work_management`
- 정의: 작업, 일정, 담당자, 스프린트, 진행 상태를 관리하는 영역
- 대표 데이터: Task, 담당자, 일정, 보드 상태, 이슈

### 2.3 문서 및 지식 관리

- 카테고리 ID: `knowledge_management`
- 정의: 위키, 설계서, 운영 문서, 회의 기록을 관리하는 영역
- 대표 데이터: 문서 메타데이터, 버전, 작성자, 링크 관계

### 2.4 형상관리

- 카테고리 ID: `scm`
- 정의: 저장소, 브랜치, 커밋, 태그, 병합 이력을 관리하는 영역
- 대표 데이터: Repository, Branch, Commit, Pull Request 또는 Merge Request
- 전략: 외부 전문 도구를 연계해 사용한다. 초기 후보는 `Gitea` 다.

### 2.5 코드 리뷰 및 개발 품질 보조

- 카테고리 ID: `code_review`
- 정의: 코드 변경 검토, 리뷰 의견, 품질 신호를 관리하는 영역
- 대표 데이터: 리뷰 요청, 코멘트, 승인 상태, 품질 체크 결과
- 전략: 원본 리뷰 이벤트는 외부 `SCM` 또는 연계 도구에서 수집할 수 있으며, 통합 시스템은 이를 프로젝트 문맥 안에서 조회하고 관리할 수 있어야 한다. 또한 `AI` 연계 기반 자동 코드리뷰 초안 작성 기능을 제공해 변경 요약, 잠재 이슈, 확인 포인트를 제안할 수 있어야 하며, 프로젝트 또는 개발팀이 지정한 코딩룰 기준으로 준수 여부를 판단할 수 있어야 한다.

### 2.6 빌드 및 지속적 통합

- 카테고리 ID: `ci_build`
- 정의: 빌드 파이프라인, 자동 검증, 패키징을 수행하는 영역
- 대표 데이터: Job, Build, Pipeline, 상태, 로그 메타데이터
- 전략: 빌드 실행과 파이프라인 오케스트레이션은 외부 전문 도구를 연계해 사용한다. 초기 연계 대상은 `Jenkins` 와 `Bamboo` 를 함께 고려한다. `Bamboo` 의 경우 `Bamboo Specs` 기반 관리 가능성을 포함한다. 통합 시스템은 미리 정의된 `stage`/`job` 템플릿과 프로젝트 소스 특성을 바탕으로 `AI` 기반 `CI` 플랜 초안을 자동 생성하고, 대상 엔진에 맞는 설정 초안으로 제안할 수 있어야 한다.

### 2.7 배포 및 릴리스 관리

- 카테고리 ID: `cd_release_deploy`
- 정의: 배포 실행, 릴리스, 환경 반영 상태를 관리하는 영역
- 대표 데이터: Release, Deployment, Environment, 승인 이력
- 전략: 실제 배포 실행은 외부 도구와 연계할 수 있으나, 배포 현황과 이력 조회는 통합 시스템에서 일관되게 제공한다.

### 2.8 빌드 산출물 관리

- 카테고리 ID: `artifact_management`
- 정의: 빌드 결과물과 패키지 메타데이터를 관리하는 영역
- 대표 데이터: Artifact, 버전, 생성 빌드, 저장 위치, 배포 대상
- 전략: 원본 저장소가 외부에 있더라도 통합 시스템은 산출물 메타데이터와 추적 관계를 관리할 수 있어야 한다.

### 2.9 테스트 관리

- 카테고리 ID: `test_management`
- 정의: 테스트 케이스, 계획, 실행, 결과를 관리하는 영역
- 대표 데이터: Test Case, Test Run, 결과, 결함 연계
- 전략: 통합 시스템은 단위테스트 결과를 수집하는 수준을 넘어, `PR` 변경 범위 또는 사용자 지정 범위를 기준으로 `AI` 기반 단위테스트 초안 작성과 테스트 실행 보조 기능을 제공할 수 있어야 한다.

### 2.10 정적 분석 및 품질 분석

- 카테고리 ID: `quality_analysis`
- 정의: 정적 분석, 코딩 룰 점검, 보안성/품질성 분석을 수행하거나 관리하는 영역
- 대표 데이터: 분석 실행, 이슈, 심각도, 품질 게이트, 추세
- 전략: 분석 실행 엔진은 현행 도구를 유지해 연계한다. 초기 기준 제품은 `Coverity` 다. 통합 시스템은 분석 결과와 조치 상태를 조회하거나 관리할 수 있어야 한다.

### 2.11 리포트 및 대시보드

- 카테고리 ID: `reporting_dashboard`
- 정의: 프로젝트, 품질, 배포, 운영 상태를 요약해 보여주는 영역
- 대표 데이터: KPI, 집계 지표, 추세, 경보

### 2.12 운영 지원

- 카테고리 ID: `operations_support`
- 정의: `ITSM`, `CMDB`, 장애/변경 운영 지원을 다루는 영역
- 대표 데이터: 구성 항목, 장애, 요청, 변경 기록

## 3. 초기 제품 매핑 예시

- `Jira`: `requirements_management`, `work_management`
- `Confluence`: `knowledge_management`
- `Gitea`: `scm`, 일부 `code_review`
- `Jenkins`: `ci_build`, 일부 `cd_release_deploy`
- `Bamboo`: `ci_build`, 일부 `cd_release_deploy`
- `Coverity`: `quality_analysis`
- 테스트 관리 도구: `test_management`
- 단위테스트 리포트 수집 체계: `test_management`, `reporting_dashboard`
- 내부 품질/현황 화면: `code_review`, `artifact_management`, `quality_analysis`, `reporting_dashboard`

## 4. 내부 제공과 외부 연계의 경계

### 4.1 외부 연계 유지 대상

- `SCM` 본체와 저장소 운영
- `CI/CD` 파이프라인 실행 엔진
- 정적분석 실행 엔진과 원본 분석 저장소

### 4.2 내부 제공 우선 대상

- 프로젝트 기준 통합 조회와 추적성 탐색
- 코드리뷰 맥락 조회와 리뷰 상태 관리
- `AI` 기반 `CI` 플랜 초안 작성과 템플릿 선택 보조
- `AI` 기반 자동 코드리뷰 초안 작성과 리뷰 보조
- 팀별 코딩룰 기준 준수 여부 판단
- `AI` 기반 단위테스트 작성 및 실행 보조
- 빌드 결과와 산출물 메타데이터 관리
- 정적분석 결과 조회, 필터링, 조치 상태 관리
- 품질/테스트/배포 현황 대시보드

## 5. 설계 시사점

- 외부 도구를 대체하지 않는 카테고리와 내부 제품 기능으로 점진 내재화 가능한 카테고리를 구분해야 한다.
- 요구사항 관리와 업무 관리는 분리하되, 동일 제품 안에서 함께 운영될 수 있음을 고려한 데이터 모델이 필요하다.
- `SCM`, `CI/CD`, 정적분석은 실행 계층보다 메타데이터, 상태, 추적 관계, 결과 활용 방식을 통합 시스템의 핵심 가치로 삼아야 한다.
- `CI` 영역에서는 실행 엔진 내재화보다 템플릿 기반 플랜 초안 생성, 프로젝트 특성 분석, 설정 보조 경험을 우선 가치로 삼아야 한다.
- 코드리뷰, 산출물, 분석 결과 관리는 단순 복제보다 프로젝트 문맥 안에서 연결 조회와 운영 통제가 가능한 구조로 설계해야 한다.
- `AI` 자동 코드리뷰는 사람 리뷰를 대체하기보다 초안 작성과 우선순위 식별을 보조하는 방식으로 설계해야 한다.
- 코딩룰 기반 `AI` 리뷰는 팀별 규칙셋 차이를 수용할 수 있어야 하며, 어떤 규칙을 근거로 판단했는지 드러나야 한다.
- 테스트 영역에서는 변경 범위 기반 점검과 범위 지정 기반 생성/실행 보조를 함께 고려해야 한다.

## 6. 가정 및 미정 사항

- `Gitea`, `Jenkins`, `Bamboo`, `Coverity` 를 초기 기준 제품으로 가정하나 최종 확정은 추후 검토가 필요하다.
- 산출물 원본 저장소를 별도 제품으로 둘지, 빌드 도구 또는 내부 저장 구조로 관리할지는 미정이다.
- 코드리뷰를 외부 `SCM`에 완전히 의존할지, 내부 보조 관리 기능을 어느 수준까지 둘지는 추가 결정이 필요하다.
- `AI` 자동 코드리뷰 결과를 단순 제안으로 둘지, 리뷰 워크플로우의 공식 입력으로 활용할지는 추가 결정이 필요하다.
- 코딩룰 규칙셋을 전역, 조직, 프로젝트, 저장소 중 어떤 범위 단위로 관리할지는 추가 결정이 필요하다.
- `AI` 기반 `CI` 플랜 초안 생성을 어떤 템플릿 구조와 승인 흐름으로 운영할지는 추가 결정이 필요하다.
- `CI` 템플릿의 공통 모델과 `Jenkins`/`Bamboo Specs` 출력 모델을 어떻게 분리할지는 추가 결정이 필요하다.
- `AI` 기반 단위테스트 생성 결과를 초안으로만 둘지, 저장소 변경안으로 직접 연결할지는 추가 결정이 필요하다.
