## 1. 목적

본 문서는 역할 기반 UX를 실제 응용 서비스 경계와 운영 통제 구조로 내리기 위한 초안이다. 같은 데이터를 보더라도 역할별 기본 진입점, 액션 권한, 관리자 전용 행위를 다른 계층으로 분리하는 것이 목적이다.

## 2. 역할 기반 응용 서비스 경계 초안

### 2.1 공통 사용자 응용 계층

공통 사용자 응용 계층은 같은 원천 데이터를 역할별로 다른 뷰 모델로 조합한다.

- 역할 홈 서비스
- 과제 워크스페이스 서비스
- 프로젝트 운영 서비스
- 조직 운영 서비스
- 품질 검증 서비스

이 계층은 권한 정책을 직접 소유하지 않고 운영 통제 계층의 판정을 받아 읽기/기여/관리 가능 범위를 결정한다.

### 2.2 관리자 응용 계층

관리자 응용 계층은 일반 사용자 응용 계층과 분리된 진입점을 가져야 한다.

- 연계 상태 모니터링 서비스
- 조직 변경 반영 센터
- 권한/감사 콘솔
- 예외 승인 및 운영 정책 관리

관리자 응용 계층은 변경 실행 전에 검토 단계와 영향 범위 확인 단계를 반드시 거치도록 설계해야 한다.

### 2.3 역할별 기본 진입 규칙

- 개발자: 과제 워크스페이스와 코드/품질 신호 중심
- 프로젝트리더: 프로젝트 운영과 릴리스 준비도 중심
- 조직장: 조직 운영과 승인/가용성 중심
- 시스템 관리자: 관리자 콘솔과 연계/마이그레이션 중심
- `SE` 담당자: 품질 검증 내 환경/산출물/배포 준비도 중심
- 테스트 담당자: 품질 검증 내 테스트/결함/게이트 중심

같은 응용 서비스라도 기본 필터, 기본 정렬, 기본 경고 규칙은 역할별로 다르게 제공하는 방향이 적절하다.

## 3. 운영/관리 경계와 감사 구조 초안

### 3.1 일반 운영과 관리자 운영의 분리

다음 행위는 일반 업무 흐름과 분리해야 한다.

- 연계 재실행
- 조직 변경 반영 승인
- 프로젝트/과제 참조 정보 일괄 갱신
- 권한 정책 변경
- `AI` 정책 예외 승인

이 기능들은 모두 관리자 콘솔에서만 실행되도록 두는 것이 안전하다.

### 3.2 감사 대상 행위

다음 행위는 최소 감사 단위로 남겨야 한다.

- 동기화 실행, 실패, 재처리
- 조직/인력 마스터 수동 보정
- 조직 변경 영향 분석 결과 확인
- 마이그레이션 승인/반려/실행
- 권한 정책 변경
- `AI` 초안 수용/반려/예외 승인

감사 로그는 사용자 행위와 시스템 자동 행위를 구분할 수 있어야 하며, 입력 근거와 결과 상태를 함께 보관해야 한다.

### 3.3 예외 처리 구조

예외는 업무 도메인 안에서 흩어 관리하지 않고 운영 통제 계층에서 공통 모델로 다루는 편이 적절하다.

- 품질 예외
- 코딩룰 오탐 예외
- `AI` 실행 정책 예외
- 계정계 연계 실패 후속 조치
- 조직 참조 갱신 제외 항목

공통 예외 모델을 쓰면 승인 주체, 만료 시각, 재검토 조건, 감사 이력을 일관되게 관리할 수 있다.

## 4. 후속 상세화 후보

- 역할별 액션 가능 상태 정의
- 관리자 콘솔 서브플로우 정의
- 권한/감사 모델 상세화
- 예외 승인 라이프사이클 정의
