# 세션 인계 문서

- 문서 목적: 새로운 세션이나 새로운 호스트 환경에서 작업을 시작할 때 이전 세션의 진행 상태, 완료 항목, 잔여 작업, 환경 차이를 빠르게 파악할 수 있도록 현재 기준 상태를 요약한다.
- 범위: 진행 중 작업, 차단 작업, 최근 완료 작업, 잔여 작업, 환경별 검증 현황, 세션 시작 절차
- 대상 독자: 개발자, 운영자, 리뷰어, 문서 작성자
- 상태: draft
- 최종 수정일: 2026-04-16
- 관련 문서: `docs/operations/work_backlog.md`, `docs/operations/standard_workflow_draft.md`, `docs/operations/environments/README.md`

## 1. 문서 개요

이 문서는 날짜별 백로그를 대체하지 않는다. 날짜별 백로그는 상세 이력의 근거 문서이고, 이 문서는 새로운 세션이 시작될 때 가장 먼저 읽는 현재 상태 요약 문서다.

목표는 다음과 같다.

- 새 세션이 5분 안에 현재 작업 기준선을 재구성할 수 있어야 한다.
- 최근 백로그 전체를 다시 읽지 않아도 `진행 중`, `차단`, `최근 완료`, `잔여 작업` 을 파악할 수 있어야 한다.
- 작업 상태와 환경 상태를 분리해서 읽을 수 있어야 한다.

## 2. 현재 작업 요약

- 현재 기준선:
  - 개발 도구 복구 완료
  - 로컬 Python/Rust 테스트 통과
  - `podman` 기반 격리 테스트 통과
  - 운영 UI `admin.html`, `organization.html` 실제 렌더링 확인 완료
- 현재 주 작업 축:
  - 운영 규칙과 작업 방식의 문서화
  - 세션 간 인계 가능한 작업 흐름 확립
- 최근 핵심 기준 문서:
  - [표준 작업 워크플로우 초안](./standard_workflow_draft.md)
  - [개발 환경 및 테스트 환경 가이드](./development_environment.md)
  - [bazzite 환경 기록](./environments/bazzite-192.168.0.122/README.md)

## 3. 진행 중 작업

- `TASK-106` 워크플로우 체계화 계획 수립 및 세부 문서 구조 정의 진행 중

세션 시작 시 해야 할 일:

- `workflow_development_plan.md` 를 먼저 읽고 현재 단계가 `계획 확정` 인지, `세부 문서 작성` 단계로 넘어갔는지 확인
- 최신 백로그에서 `TASK-106` 의 진행 현황과 후속 작업을 확인

## 4. 차단 작업

- 현재 `blocked` 로 유지 중인 작업 없음

세션 시작 시 확인할 것:

- 새 호스트에서 도구 미설치 또는 네트워크 제약이 다시 발생하는지
- `docs/operations/environments/<hostname>-<ip>/README.md` 에 현재 호스트 특이사항이 이미 있는지

## 5. 최근 완료 작업

- `TASK-102` 개발 도구 설치 및 로컬/격리 테스트 환경 검증 완료
- `TASK-103` 운영 UI 실행 및 렌더링 확인 완료
- `TASK-104` 표준 작업 워크플로우 초안 작성 완료
- `TASK-105` 세션 인계 중심 워크플로우 구축 완료

최근 완료 작업의 의미:

- 이제 저장소는 새 호스트에서 환경 복구와 테스트 검증을 위한 기준 문서를 갖고 있다.
- 운영 UI는 문서상 연결 가능 상태가 아니라 실제 실행과 렌더링 증적까지 확보된 상태다.
- 다음 단계는 “어떻게 일할 것인가”를 세션 인계 중심으로 세분화하는 작업이다.

## 6. 잔여 작업 우선순위

### 우선순위 1

- 워크플로우 개발 계획을 기준으로 세부 문서 구조를 확정
- 작업 진행 워크플로우 상세 문서 작성
- 문서화 및 문서 동기화 워크플로우 상세 문서 작성

### 우선순위 2

- 코드 색인 전략 문서 작성
- AI agent 빠른 참조 문서 작성
- 브랜치 병합 시 문서/색인 견고성 정책 작성

### 우선순위 3

- 자동화 계획 문서 작성
- 백로그/환경/문서 동기화 보조 자동화 후보 설계
- `build_router` 미사용 경고 정리
- Python 캐시/`egg-info` 정리 기준과 무시 규칙 보강

## 8. 새 세션 시작 절차

1. [session_handoff.md](./session_handoff.md) 를 먼저 읽는다.
2. [work_backlog.md](./work_backlog.md) 와 가장 최근 날짜 백로그를 확인한다.
3. `in_progress`, `blocked`, `done but follow-up needed` 항목을 재분류한다.
4. 현재 호스트의 환경 문서가 있으면 먼저 읽고, 없으면 새로 만든다.
5. 필요한 도구, 포트, DB, 컨테이너 상태를 확인한다.
6. 오늘 작업 목적, 범위, 산출물, 영향 문서를 짧게 브리핑한다.
7. 오늘자 백로그에 작업을 등록하거나 기존 작업을 이어받는다.

## 9. 세션 종료 절차

1. 오늘 수행한 작업의 상태를 `planned`, `in_progress`, `blocked`, `done` 으로 정리한다.
2. 백로그에 `다음 세션 시작 포인트`, `남은 리스크`, `후속 작업` 을 적는다.
3. 환경 변경이 있었으면 호스트별 환경 문서를 갱신한다.
4. 현재 상태를 `session_handoff.md` 의 `진행 중`, `차단`, `최근 완료`, `잔여 작업` 에 반영한다.
5. 다음 세션이 가장 먼저 읽어야 할 문서와 명령을 남긴다.

## 10. 인계 문서 갱신 규칙

- 세션 시작 시:
  - 이전 문서가 최신인지 확인하고, 실제 상태와 다르면 즉시 갱신한다.
- 세션 종료 시:
  - 반드시 한 번은 갱신한다.
- 작업 완료 시:
  - 영향이 큰 완료 작업은 `최근 완료 작업` 에 반영한다.
- 작업 차단 시:
  - 차단 원인을 `차단 작업` 섹션에 즉시 반영한다.
