# 검증 워크플로우

- 문서 목적: 코드 또는 문서 변경 후 로컬 검증, 격리 검증, 실행 확인을 어떤 순서와 기준으로 수행할지 표준화한다.
- 범위: 빠른 로컬 검증, 컨테이너 기반 격리 검증, API/UI 실행 확인, 증적 기록, 실패 처리
- 대상 독자: 개발자, 운영자, 리뷰어, AI agent
- 상태: draft
- 최종 수정일: 2026-04-16
- 관련 문서: `docs/operations/standard_workflow_draft.md`, `docs/operations/development_environment.md`, `docs/operations/workflow_task_execution.md`, `docs/operations/environments/README.md`, `Makefile`, `docker-compose.yml`

## 1. 문서 개요

이 문서는 변경 이후 무엇을 얼마나 검증해야 하는지, 어떤 순서로 검증을 올려야 하는지, 결과를 어디에 남겨야 하는지를 다룬다. 환경 설치 자체는 `development_environment.md` 가 담당하고, 이 문서는 검증 실행 순서와 기록 기준에 집중한다.

핵심 목표는 아래와 같다.

- 가장 빠른 실패를 먼저 잡는다.
- 로컬 성공과 격리 성공을 구분해서 기록한다.
- 실행 결과가 중요한 작업은 실제 프로세스를 띄워 확인한다.
- 검증 결과가 다음 세션과 리뷰어에게 재사용 가능해야 한다.

## 2. 검증 단계의 기본 원칙

- 검증은 빠른 것부터 무거운 순서로 올린다.
- 로컬 검증 성공이 격리 검증을 대체하지 않는다.
- 격리 검증 성공이 실행 확인을 대체하지 않는다.
- UI/API/배치처럼 런타임 결과가 중요한 변경은 실제 기동 증적을 남긴다.
- 검증하지 못한 항목은 명시적으로 남긴다.

## 3. 검증 단계 구조

표준 검증 흐름은 아래 4단계다.

1. 빠른 로컬 검증
2. 컨테이너 기반 격리 검증
3. 실행 확인
4. 결과 기록

모든 작업이 4단계를 다 수행해야 하는 것은 아니다. 하지만 어떤 단계를 생략했는지는 문서로 남겨야 한다.

## 4. 빠른 로컬 검증

수정 직후 가장 먼저 수행하는 단계다. 목적은 구현이나 문서 변경이 즉시 깨졌는지 빠르게 확인하는 것이다.

대표 명령:

- Python 테스트: `python3 -m pytest`
- Rust 테스트: `cargo test --manifest-path backend/Cargo.toml`
- 정적 diff 검사: `git diff --check`

선택 기준:

- Python 코드나 Python 관련 설정을 건드렸으면 Python 테스트를 우선 수행한다.
- Rust 백엔드 코드를 건드렸으면 Rust 테스트를 수행한다.
- 문서와 설정 변경만 있어도 `git diff --check` 는 기본으로 본다.

기록 방식:

- 성공 시 어떤 명령이 통과했는지 남긴다.
- 실패 시 어느 명령이 어느 이유로 실패했는지 남긴다.

## 5. 컨테이너 기반 격리 검증

로컬 검증이 끝나면 가능한 경우 컨테이너 기반 경로를 수행한다. 목적은 로컬 개발기 우연성에 기대지 않고 저장소 기본 경로가 재현 가능한지 확인하는 것이다.

대표 명령:

- 인프라 기동: `CONTAINER_RUNTIME=podman make infra-up`
- DB readiness 확인: `CONTAINER_RUNTIME=podman make infra-wait`
- Python 격리 테스트: `CONTAINER_RUNTIME=podman make container-test-python`
- Rust 격리 테스트: `CONTAINER_RUNTIME=podman make container-test-rust`
- 일괄 실행: `CONTAINER_RUNTIME=podman make container-test`

현재 저장소 기준 유의사항:

- 기본 `PostgreSQL` 이미지는 `docker.io/library/postgres:17`
- SELinux 환경 바인드 마운트는 `:Z` 필요
- Rust 테스트 이미지는 `docker.io/library/rust:1.94-bookworm`
- `podman` 과 `docker` 모두 가능하지만, 현재 호스트 검증 기준은 `podman`

생략 가능 조건:

- 문서 수정만 있고 실행/테스트 경로에 영향을 주지 않는 경우
- 네트워크 또는 런타임 제약으로 격리 검증이 불가능한 경우

단, 생략 시 이유를 백로그에 기록한다.

## 6. 실행 확인

실행 결과가 중요한 작업이면 실제 프로세스를 띄워 확인한다.

대표 대상:

- API 응답 변화
- UI 렌더링 변화
- 백엔드/정적 서버 기동 경로 변화
- 환경 연동 결과 변화

대표 흐름:

1. 필요한 DB 또는 컨테이너를 기동한다.
2. 백엔드 또는 서버를 기동한다.
3. `curl` 로 1차 상태를 확인한다.
4. 브라우저 또는 Playwright 로 UI/API 결과를 확인한다.
5. 필요 시 스크린샷, 로그, 응답을 저장한다.

현재 저장소에서 실제 사용한 예:

- 백엔드 헬스 체크: `curl http://127.0.0.1:8080/api/v1/health`
- 정적 UI 확인: `python3 -m http.server 8000`
- 브라우저 렌더 증적: `output/playwright/ui_prototype/runtime_check_2026-04-16/`

## 7. 검증 수준 선택 기준

### 7.1 문서 변경만 있는 경우

- 기본적으로 `git diff --check`
- 링크 구조나 운영 절차가 바뀌면 관련 문서 간 링크 확인
- 실행 검증은 보통 생략 가능

### 7.2 코드 로직 변경이 있는 경우

- 관련 로컬 테스트 수행
- 가능한 경우 격리 검증 수행
- API 또는 UI 동작이 바뀌면 실행 확인까지 고려

### 7.3 환경 또는 컨테이너 설정 변경이 있는 경우

- 격리 검증 우선
- 필요 시 실제 서버 기동 확인
- 호스트별 환경 문서 갱신

### 7.4 UI 관련 변경이 있는 경우

- 정적 또는 동적 화면 실제 렌더링 확인
- 상태 칩, 오류 문구, 콘솔 오류 여부 확인
- 가능하면 스크린샷 저장

## 8. 증적 기록 기준

검증 결과는 아래 수준 중 필요한 만큼 남긴다.

- 통과한 명령
- 실패한 명령과 원인
- API 응답 요약
- UI 렌더링 결과 요약
- 스크린샷 경로
- 미실행 또는 생략 사유

좋은 기록 예:

- `python3 -m pytest 통과`
- `CONTAINER_RUNTIME=podman make container-test-rust 통과`
- `GET /api/v1/health => 200 {"status":"ok"}`
- `admin.html, organization.html 브라우저 기준 연결 정상 확인`

## 9. 실패 처리 원칙

### 9.1 로컬에서 실패할 때

- 먼저 가장 작은 단위에서 재현 가능한지 본다.
- 환경 문제인지 기대값 문제인지 구분한다.
- 기대값 문제면 테스트 또는 기준 문서 보정이 필요한지 확인한다.

### 9.2 격리 검증에서만 실패할 때

- 저장소 기본값 문제인지 먼저 본다.
- 이미지, 마운트, 환경 파일, 네트워크, 런타임 호환성 순으로 확인한다.
- 특정 호스트 이슈면 공통 문서보다 호스트 문서에 남긴다.

### 9.3 실행 확인에서 실패할 때

- API, 프로세스 기동, 자산 경로, 브라우저 렌더링 중 어디서 끊기는지 분리한다.
- 상태 코드는 간단히 요약하고, 상세는 백로그 또는 증적으로 남긴다.

## 10. 검증 결과와 백로그 연결

작업 종료 전에는 검증 결과가 백로그에 반영돼 있어야 한다.

최소 기록 항목:

- 어떤 검증을 했는지
- 무엇이 통과했는지
- 무엇을 못 했는지
- 남은 리스크가 무엇인지

세션 인계 문서에 반영할 때는 요약만 남긴다.

예:

- `로컬 Python/Rust 테스트 통과`
- `podman 기반 격리 테스트 통과`
- `운영 UI 렌더링 확인 완료`

## 11. 빠른 체크리스트

검증 시작 체크:

- 변경 유형에 맞는 검증 범위를 정했는가
- 가장 빠른 로컬 검증부터 시작했는가
- 격리 검증 필요 여부를 판단했는가
- 실행 확인 필요 여부를 판단했는가

검증 종료 체크:

- 통과한 검증을 기록했는가
- 생략한 검증과 이유를 기록했는가
- 필요 시 스크린샷이나 응답 요약을 남겼는가
- 남은 리스크를 백로그에 반영했는가
