# 코드 색인 전략

- 문서 목적: 코드 상세 분석 문서, 빠른 탐색용 색인 문서, AI agent용 참조 문서가 어떤 구조와 책임으로 운영되어야 하는지 정의한다.
- 범위: 색인 문서 계층, 관리 단위, 필수 메타데이터, 검색 키 설계, 갱신 규칙, 병합 안정성 기준
- 대상 독자: 개발자, 아키텍트, 문서 작성자, AI agent
- 상태: draft
- 최종 수정일: 2026-04-16
- 관련 문서: `docs/operations/workflow_development_plan.md`, `docs/operations/workflow_documentation_sync.md`, `docs/operations/workflow_task_execution.md`, `docs/operations/standard_workflow_draft.md`, `docs/architecture/current_backend_implementation_status_summary.md`

## 1. 문서 개요

이 문서는 “코드를 빨리 찾고, 바뀐 사실을 빨리 추적하며, AI agent 가 긴 문서를 다 읽지 않고도 필요한 문맥을 복원할 수 있게 하려면 문서를 어떻게 구성해야 하는가”를 다룬다.

현재 저장소에는 구현 설명 문서와 운영 문서가 있지만, 앞으로 코드베이스가 커지면 아래 문제가 커질 가능성이 높다.

- 특정 기능이 어디에 구현돼 있는지 찾기 어렵다.
- 설명 문서는 있는데 빠른 탐색용 키가 없다.
- 사람에게 좋은 문서가 AI 에게는 너무 길거나 느리다.
- 브랜치 병합 후 문서 링크나 색인 정합성이 깨질 수 있다.

이 문서는 위 문제를 줄이기 위한 색인 전략의 기준 문서다.

## 2. 목표

- 사람은 설명 문서를 따라가며 구조를 이해할 수 있어야 한다.
- AI agent 는 최소 문서 집합으로 필요한 파일과 관련 문서를 찾을 수 있어야 한다.
- 코드 변경 시 어떤 색인 문서를 같이 갱신해야 하는지 판단 가능해야 한다.
- 여러 브랜치에서 작업해도 색인 충돌을 줄일 수 있어야 한다.

## 3. 색인 문서 계층

코드 참조 문서는 아래 3계층으로 나눈다.

### 3.1 설명 문서

역할:

- 맥락, 배경, 구조 설명
- 설계 이유와 운영 의미 설명

예시:

- [current_backend_implementation_status_summary.md](../architecture/current_backend_implementation_status_summary.md)
- 아키텍처 초안 문서
- 요구사항 문서

특징:

- 사람이 읽기 좋다.
- 설명이 길어질 수 있다.
- AI 에게는 기준 사실 문서 역할을 하되, 빠른 탐색에는 비효율적일 수 있다.

### 3.2 색인 문서

역할:

- 경로, 책임, 엔트리포인트, 주요 테스트, 관련 문서를 빠르게 찾게 한다.

예시 후보:

- 도메인 색인
- 모듈 색인
- 운영 UI 색인
- 백엔드 API 색인

특징:

- 설명보다 탐색 키를 우선한다.
- 링크 허브 역할을 한다.
- 문서 길이는 짧게 유지한다.

### 3.3 AI agent 빠른 참조 문서

역할:

- AI agent 가 최소 문서와 파일 경로로 작업을 시작하게 한다.

예시 후보:

- `ai_agent_quickstart.md`
- 도메인별 quick reference

특징:

- 탐색 순서가 명확해야 한다.
- 기준 문서 링크를 반드시 포함해야 한다.
- 독립 사실 문서가 아니라 빠른 라우팅 문서다.

## 4. 권장 문서 구조

향후 색인 문서는 아래 구조를 따른다.

### 4.1 저장소 최상위 빠른 진입

- `docs/README.md`
- `docs/operations/README.md`
- `docs/operations/ai_agent_quickstart.md`

역할:

- 카테고리 진입
- 우선 읽을 문서 안내

### 4.2 도메인 또는 기능 단위 색인

문서 예시 후보:

- `docs/architecture/backend_code_index.md`
- `docs/architecture/admin_ui_code_index.md`
- `docs/integrations/integration_mapping_index.md`

역할:

- 해당 영역의 주요 파일 경로
- 책임 모듈
- 엔트리포인트
- 주요 테스트
- 기준 문서 링크

### 4.3 AI 용 요약 진입

문서 예시 후보:

- `docs/operations/ai_agent_quickstart.md`

역할:

- 새 세션 AI 가 먼저 읽을 문서
- 질문 유형별 진입 순서
- 빠른 파일 탐색 키

## 5. 색인 문서의 필수 항목

색인 문서는 최소한 아래 항목을 포함하는 것이 좋다.

- 문서 목적
- 범위
- 대상 독자
- 상태
- 최종 수정일
- 관련 문서
- 대상 영역 요약
- 주요 경로
- 엔트리포인트
- 관련 테스트
- 관련 운영 문서
- 최근 주의사항 또는 리스크

권장 메타데이터 예시:

```md
# 문서 제목

- 문서 목적:
- 범위:
- 대상 독자:
- 상태:
- 최종 수정일:
- 관련 문서:
```

## 6. 관리 단위 설계 기준

색인 문서는 너무 크거나 너무 작으면 둘 다 쓸모가 떨어진다.

### 6.1 도메인 기준이 적합한 경우

- 기능 경계가 비교적 안정적일 때
- 여러 모듈이 하나의 업무 흐름을 구성할 때

예:

- 조직/인력 관리
- 프로젝트/업무 항목 관리
- 수집/정규화 파이프라인

### 6.2 모듈 기준이 적합한 경우

- 기술 경계가 명확할 때
- 특정 디렉터리 자체가 명확한 책임을 가질 때

예:

- `backend/`
- `src/ui_prototype/`

### 6.3 기능 기준이 적합한 경우

- 한 기능이 여러 디렉터리에 걸쳐 있지만 추적이 중요할 때

예:

- 운영 UI와 운영 API 연결 흐름
- 동기화 실행과 결과 조회 흐름

기본 권장:

- 처음에는 `도메인 또는 기능` 단위로 시작하고, 필요할 때만 모듈 상세 색인을 추가한다.

## 7. 검색 키 설계 원칙

AI 와 사람이 빠르게 찾으려면 문서에 검색 가능한 키가 명확해야 한다.

권장 포함 항목:

- 디렉터리 경로
- 핵심 파일명
- 라우트 또는 API prefix
- 도메인 엔터티명
- 핵심 서비스명
- 주요 테스트 파일명
- 관련 운영 화면명

좋은 예:

- `backend/src/routes/admin`
- `GET /api/v1/admin/projects`
- `work_item`
- `sync_runs`
- `organization.html`

## 8. 갱신 규칙

색인 문서는 아래 상황에서 갱신 대상이 된다.

- 경로가 바뀔 때
- 엔트리포인트가 바뀔 때
- 주요 책임 모듈이 이동할 때
- 관련 테스트 위치가 바뀔 때
- 기준 문서 링크가 바뀔 때

갱신 순서:

1. 기준 사실 문서를 갱신한다.
2. 색인 문서를 갱신한다.
3. 허브와 AI 빠른 참조 문서를 갱신한다.
4. 백로그에 영향 문서를 반영한다.

## 9. 병합 안정성 원칙

색인 문서는 병합 충돌에 약할 수 있으므로 아래 원칙을 따른다.

- 한 문서에 전체 저장소 색인을 몰아넣지 않는다.
- 도메인 또는 기능 단위로 분리해 충돌 범위를 줄인다.
- 정렬 기준은 고정한다.
- 경로 표기 방식은 일관되게 유지한다.
- 생성 가능한 정보와 설명이 필요한 정보를 분리한다.

예:

- 경로 목록, 엔트리포인트 목록은 자동화 가능 후보
- 책임 설명, 주의사항, 운영 메모는 수동 유지 항목

## 10. AI 친화 운영 원칙

AI agent 가 색인 문서를 쓸 때는 아래가 보장돼야 한다.

- 첫 문서 하나만 읽어도 다음 문서를 알 수 있다.
- 경로와 책임이 한눈에 보인다.
- 관련 테스트와 관련 문서를 함께 찾을 수 있다.
- 긴 설명은 기준 문서로 넘기고, 색인 문서 자체는 짧게 유지한다.

실행 방향:

- 색인 문서는 “무엇을 읽어야 하는가”를 안내한다.
- AI quickstart 는 “어떤 순서로 읽어야 하는가”를 안내한다.
- 설명 문서는 “왜 그런가”를 설명한다.

## 11. 초기 도입 제안

현재 저장소에서는 아래 순서로 도입하는 것이 적절하다.

1. `ai_agent_quickstart.md` 작성
2. 백엔드 영역 색인 초안 작성
3. 운영 UI 영역 색인 초안 작성
4. 병합 정책 문서 작성
5. 자동화 계획에 색인 검사/재생성 후보 반영

이 순서를 권장하는 이유:

- 먼저 AI 와 사람이 같은 진입점을 가져야 한다.
- 그 다음 실제 코드 영역별 색인을 붙여야 효과가 생긴다.

## 12. 현재 결정

현재 기준 권장 방향은 아래와 같다.

- 사람용 설명 문서와 AI용 빠른 참조 문서는 분리한다.
- 색인 문서는 설명 문서와 별도 계층으로 둔다.
- 초기 색인 단위는 `도메인 또는 기능` 기준으로 시작한다.
- 전체 저장소 단일 초대형 색인은 만들지 않는다.
