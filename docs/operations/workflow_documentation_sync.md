# 문서화 및 동기화 워크플로우

- 문서 목적: 코드 변경, 설계 변경, 운영 규칙 변경이 발생할 때 어떤 문서를 함께 확인하고 갱신해야 하는지 표준 절차를 정의한다.
- 범위: 문서 영향도 판단, 문서 갱신 순서, 문서 간 책임 분리, 사람용/AI용 문서 동기화, 예외 처리
- 대상 독자: 개발자, 문서 작성자, 아키텍트, 운영자, AI agent
- 상태: draft
- 최종 수정일: 2026-04-16
- 관련 문서: `docs/operations/standard_workflow_draft.md`, `docs/operations/workflow_task_execution.md`, `docs/requirements/system_crs.md`, `docs/requirements/system_srs.md`, `docs/architecture/system_architecture_draft.md`, `AGENTS.md`

## 1. 문서 개요

이 문서는 “코드를 바꿨으면 문서도 바꿔야 한다”는 원칙을 실제 작업 절차로 구체화한다. 핵심은 문서를 많이 쓰는 것이 아니라, 변경의 성격에 맞는 기준 문서를 정확히 갱신하고 탐색 경로가 끊기지 않게 유지하는 것이다.

이 문서가 해결하려는 문제는 아래와 같다.

- 코드 변경 후 어떤 문서를 갱신해야 하는지 사람마다 판단이 달라지는 문제
- 운영 문서, 요구사항 문서, 아키텍처 문서가 코드와 어긋나는 문제
- 나중에 색인 문서와 AI 참조 문서가 생겼을 때 기준 사실이 흔들리는 문제
- 여러 브랜치에서 문서가 따로 수정돼 병합 후 정합성이 깨지는 문제

## 2. 핵심 원칙

- 문서의 기준 사실은 한 곳에서만 확정한다.
- 요약 문서는 상세 문서를 대체하지 않는다.
- 코드 변경이 기능, 구조, 운영 규칙을 바꾸면 관련 기준 문서를 함께 확인한다.
- 문서 링크 구조와 메타데이터도 변경의 일부로 본다.
- 사람용 설명 문서와 AI용 빠른 참조 문서는 깊이가 달라도 같은 기준 사실을 가리켜야 한다.

## 3. 문서 역할 구분

문서 간 역할을 먼저 분리해야 동기화 기준도 명확해진다.

### 3.1 개요 문서

대상:

- `docs/overview/*`

역할:

- 프로젝트 목적, 방향, 사용자 관점, 상위 배경 설명

갱신이 필요한 경우:

- 프로젝트 방향, 목표, 범위, 사용자 관점이 바뀔 때

### 3.2 요구사항 문서

대상:

- [system_crs.md](../requirements/system_crs.md)
- [system_srs.md](../requirements/system_srs.md)

역할:

- 무엇을 왜 만들어야 하는지, 어떤 요구와 제약이 있는지 정의

갱신이 필요한 경우:

- 기능 요구가 추가되거나 삭제될 때
- 비기능 요구, 운영 요구, 연계 요구가 바뀔 때
- 상위 개념 또는 정책이 바뀔 때

### 3.3 아키텍처 문서

대상:

- [system_architecture_draft.md](../architecture/system_architecture_draft.md)
- 그 하위 분리 문서

역할:

- 시스템 경계, 책임 분리, 데이터 흐름, 컴포넌트 구조 설명

갱신이 필요한 경우:

- 구현 구조가 설계 가정과 달라졌을 때
- 컴포넌트 책임, 연계 구조, 데이터 흐름이 바뀔 때

### 3.4 운영 문서

대상:

- `docs/operations/*`

역할:

- 실제 작업 절차, 운영 규칙, 환경 차이, 검증 절차, 작업 기록 관리

갱신이 필요한 경우:

- 개발/검증/기록 방식이 바뀔 때
- 새 절차 문서가 추가되거나 이동할 때
- 세션 인계 방식, 백로그 방식, 환경 기록 방식이 바뀔 때

### 3.5 색인 및 AI 참조 문서

대상:

- [code_index_strategy.md](./code_index_strategy.md)
- [ai_agent_quickstart.md](./ai_agent_quickstart.md)
- 향후 모듈/도메인 단위 색인 문서

역할:

- 빠른 탐색
- 변경 영향 확인
- AI agent 의 최소 경로 참조

갱신이 필요한 경우:

- 경로, 책임, 엔트리포인트, 주요 테스트 위치가 바뀔 때
- 기준 문서 링크가 바뀔 때

## 4. 문서 영향도 판단 절차

코드나 문서를 수정하기 전에는 아래 순서로 영향도를 판단한다.

1. 변경의 성격을 분류한다.
2. 기준 사실이 무엇인지 찾는다.
3. 기준 사실을 담는 문서를 확인한다.
4. 요약/허브/색인 문서 중 따라 수정해야 할 문서를 찾는다.
5. 백로그의 `영향 문서` 에 반영한다.

변경 성격 분류 예시:

- 기능 변화
- 구조 변화
- 운영 절차 변화
- 환경 변화
- 용어/정책 변화
- 탐색 경로 또는 문서 구조 변화

## 5. 변경 유형별 기본 동기화 규칙

### 5.1 기능 변경

예시:

- API 추가
- UI 동작 변경
- 도메인 규칙 변경
- 상태 전이 변경

기본 확인 문서:

- 요구사항 문서
- 관련 아키텍처 문서
- 운영 문서 중 검증/작업 절차 문서

판단 기준:

- 기존 요구를 구현한 것인지, 요구 자체를 바꾸는 것인지 먼저 구분한다.
- 요구 자체가 바뀌면 `CRS` 또는 `SRS` 를 갱신한다.
- 구현 구조나 데이터 흐름이 바뀌면 아키텍처 문서를 갱신한다.

### 5.2 구조 변경

예시:

- 모듈 분리
- 디렉터리 이동
- 서비스 책임 이동
- 연계 구조 재구성

기본 확인 문서:

- 아키텍처 문서
- 운영 문서 중 개발/검증 절차
- 색인 문서 및 AI 빠른 참조 문서

판단 기준:

- 사용자 기능은 같아도 책임 경계가 바뀌면 아키텍처 문서를 갱신한다.
- 경로와 진입점이 바뀌면 색인 문서도 같이 갱신 대상이다.

### 5.3 운영 절차 변경

예시:

- 테스트 명령 변경
- 세션 인계 방식 변경
- 백로그 항목 규칙 변경
- 환경 검증 절차 변경

기본 확인 문서:

- `docs/operations/*`
- `AGENTS.md`
- 관련 README

판단 기준:

- 실제 일하는 방식이 바뀌면 운영 문서가 기준 문서다.
- 단순 설명 추가가 아니라 규칙 변경이면 허브 문서와 상세 절차 문서를 함께 본다.

### 5.4 환경 변경

예시:

- 런타임 버전 변경
- 컨테이너 기준 변경
- 특정 호스트 제약 추가

기본 확인 문서:

- `development_environment.md`
- 호스트별 환경 문서
- 검증 관련 운영 문서

판단 기준:

- 공통 절차는 공통 문서에 적고, 호스트 특이사항은 호스트 문서에 적는다.
- 특정 세션에서만 발생한 문제를 공통 규칙처럼 쓰지 않는다.

### 5.5 문서 구조 변경

예시:

- 문서 이동
- 새 인덱스 문서 추가
- 허브/상세 문서 분리

기본 확인 문서:

- `docs/README.md`
- 카테고리별 `README.md`
- 허브 문서
- 세션 인계 문서 또는 운영 문서 중 진입점을 안내하는 문서

판단 기준:

- 문서를 만들기만 하고 인덱스와 허브에 연결하지 않으면 동기화가 끝난 것이 아니다.

## 6. 문서 갱신 순서

문서 갱신은 아래 순서를 기본으로 한다.

1. 기준 사실 문서를 갱신한다.
2. 관련 상세 문서를 갱신한다.
3. 허브 문서와 인덱스 문서를 갱신한다.
4. 세션 인계나 백로그에 결과를 반영한다.
5. 색인/AI 문서가 있으면 빠른 참조 문서를 갱신한다.

이 순서를 쓰는 이유:

- 기준 사실이 먼저 바뀌어야 파생 문서가 무엇을 따라가야 하는지 분명해진다.
- 허브 문서와 인덱스는 마지막에 맞추는 편이 링크 누락을 줄인다.

## 7. 사람용 문서와 AI용 문서 동기화 원칙

AI 전용 문서를 운영할 때는 아래 원칙을 따른다.

- 사람용 문서는 설명과 맥락을 담는다.
- AI용 문서는 탐색 경로와 핵심 식별자를 담는다.
- 동일 사실을 두 문서에서 각각 새로 정의하지 않는다.
- AI용 문서는 반드시 사람용 기준 문서 또는 원문 기준 문서로 링크해야 한다.
- 브랜치 병합 후에는 quickstart 와 색인 문서를 현재 기준선으로 다시 맞춘다.

예시 구조:

- 사람용 기준 문서
  - 요구사항 문서
  - 아키텍처 문서
  - 운영 절차 문서
- AI용 빠른 참조 문서
  - 관련 경로
  - 엔트리포인트
  - 주요 테스트
  - 기준 문서 링크

## 8. 백로그와의 연결 규칙

백로그의 `영향 문서` 는 선택사항이 아니라 동기화 체크포인트다.

작업 시작 시:

- 예상 영향 문서를 적는다.

작업 중간:

- 실제 영향 범위가 바뀌면 목록을 갱신한다.

작업 종료 시:

- 실제 수정한 문서가 영향 문서 목록과 맞는지 확인한다.
- 문서 변경이 필요 없었다면 그 판단을 `작업 결과` 또는 `진행 현황` 에 남긴다.

## 9. 자주 발생하는 누락 패턴

- 코드만 바꾸고 요구사항 문서를 안 보는 경우
- 절차 문서를 만들고 운영 위키 링크를 안 다는 경우
- 공통 문서에 호스트 특이사항을 섞어 쓰는 경우
- 상세 문서를 갱신했지만 허브 문서는 옛 상태로 두는 경우
- AI용 요약 문서가 기준 문서와 다른 표현을 갖기 시작하는 경우

## 10. 검토 체크리스트

코드 변경 후 체크:

- 이 변경이 요구사항 해석을 바꾸는가
- 이 변경이 구조 책임을 바꾸는가
- 이 변경이 운영 절차를 바꾸는가
- 이 변경이 환경 의존 정보를 바꾸는가
- 이 변경으로 탐색 경로나 링크 구조가 바뀌는가

문서 변경 후 체크:

- 기준 사실 문서가 먼저 갱신됐는가
- 허브 문서와 인덱스 링크가 갱신됐는가
- 세션 인계 또는 백로그에 반영할 내용이 생겼는가
- 사람용 문서와 AI용 문서 사이에 모순이 없는가

## 11. 예외 처리

### 11.1 문서 갱신이 당장 어렵지만 코드 수정이 급한 경우

- 문서 갱신을 생략하지 말고 백로그에 미반영 사실과 이유를 남긴다.
- 세션 종료 전에 최소한 영향 문서 후보와 남은 작업을 기록한다.

### 11.2 어떤 문서를 기준으로 봐야 할지 모를 때

- 먼저 요구사항, 아키텍처, 운영 중 어느 축의 변경인지 분류한다.
- 불명확하면 상위 문서에서 시작하고, 확정되지 않은 내용은 `가정` 으로 표시한다.

### 11.3 기존 문서가 이미 실제와 어긋나 있을 때

- 이번 변경의 범위를 넘더라도 기준 사실과 직접 충돌하는 부분은 함께 바로잡는다.
- 범위가 너무 커지면 별도 백로그 작업으로 분리한다.
