# 워크플로우 개발 계획

- 문서 목적: 저장소의 작업 워크플로우, 문서화 워크플로우, 색인 구조, 자동화 범위를 체계적으로 정리하기 위한 단계별 개발 계획을 정의한다.
- 범위: 운영 워크플로우 분해, AI/사람 공용 문서 구조, 자동화 후보, 브랜치 병합 안정성, 단계별 산출물
- 대상 독자: 개발자, 문서 작성자, 운영자, AI agent
- 상태: draft
- 최종 수정일: 2026-04-16
- 관련 문서: `docs/operations/standard_workflow_draft.md`, `docs/operations/session_handoff.md`, `docs/operations/work_backlog.md`, `docs/README.md`, `AGENTS.md`

## 1. 배경과 의도

현재 저장소에는 작업 방식의 큰 흐름을 설명하는 운영 문서가 생겼지만, 실제로는 아래 문제가 남아 있다.

- 작업 유형별 절차가 한 문서에 뭉쳐 있어 빠른 참조성이 떨어진다.
- 코드 구현과 문서 갱신, 색인 갱신, 세션 인계가 아직 하나의 체계로 묶여 있지 않다.
- 사람이 읽기 좋은 문서와 AI agent 가 빠르게 탐색하기 좋은 문서의 목적이 아직 분리되지 않았다.
- 여러 브랜치에서 문서와 코드가 함께 바뀔 때 병합 후 문서 정확성이 깨질 위험이 있다.

사용자 의도는 단순한 문서 정리가 아니라 다음 구조를 만드는 것이다.

- 새 세션과 새 환경에서 즉시 현재 상태를 복원할 수 있는 작업 체계
- 코드 변경 시 관련 분석 문서와 색인 문서가 함께 동기화되는 문서 체계
- 사람과 AI agent 모두가 불필요한 탐색 비용 없이 참조할 수 있는 구조
- 브랜치 병합 이후에도 문서, 색인, 인계 정보가 견고하게 유지되는 운영 방식

## 2. 목표

### 2.1 1차 목표

- 표준 워크플로우를 카테고리별 문서로 분해한다.
- 작업 진행, 문서화, 세션 인계, 검증, 코드 색인 관리의 책임과 순서를 명시한다.
- AI agent 가 우선 읽어야 할 문서와 사람이 우선 읽어야 할 문서를 분리 설계한다.
- 자동화 가능한 갱신 지점을 식별하고 문서에 반영한다.

### 2.2 2차 목표

- 문서와 색인이 코드 변경과 함께 누락 없이 갱신되도록 체크 구조를 정의한다.
- 브랜치별 변경이 병합된 뒤에도 문서 링크, 색인, 상태 요약이 망가지지 않도록 병합 규칙을 정의한다.
- 이후 스크립트 또는 CI 로 확장 가능한 최소 자동화 인터페이스를 설계한다.

## 4. 워크플로우 카테고리 설계 방향

### 4.1 작업 진행 워크플로우

다룰 내용:

- 세션 시작 확인
- 작업 브리핑
- 백로그 등록
- 영향 문서 식별
- 구현 및 수정
- 검증 실행
- 결과 기록
- 세션 종료 인계

핵심 질문:

- 어떤 시점에 어떤 문서를 반드시 확인해야 하는가
- 작업 상태와 문서 상태를 어떻게 함께 관리할 것인가

### 4.2 문서화 워크플로우

다룰 내용:

- 개요 문서, 요구사항 문서, 아키텍처 문서, 운영 문서 간 연계
- 코드 수정 시 관련 문서 갱신 규칙
- CRS/SRS/아키텍처/운영 문서의 영향 범위 판단 기준
- 문서 메타데이터와 상호 링크 규칙

핵심 질문:

- 코드가 바뀌었을 때 어떤 문서가 반드시 동기화돼야 하는가
- 문서 누락 여부를 사람이든 AI 든 빠르게 판단할 수 있는가

### 4.3 코드 분석 및 색인 워크플로우

다룰 내용:

- 코드 상세 분석 문서의 작성 단위
- 빠른 탐색용 색인 문서의 작성 단위
- 사람용 설명 문서와 AI 용 빠른 참조 문서의 분리 여부
- 코드 기준 식별자, 경로, 책임 모듈, 연계 지점을 어떻게 인덱싱할 것인가

핵심 질문:

- AI agent 가 긴 문서를 다 읽지 않고도 필요한 파일과 문맥을 찾을 수 있는가
- 브랜치 병합 후에도 색인 충돌 없이 유지할 수 있는가

### 4.4 검증 워크플로우

다룰 내용:

- 빠른 로컬 검증
- 컨테이너 기반 격리 검증
- UI/API 실행 검증
- 문서 갱신 검증

핵심 질문:

- 코드가 통과했더라도 문서/색인 동기화 누락을 잡아낼 수 있는가
- 검증 결과를 어떤 형식으로 백로그와 인계 문서에 남길 것인가

### 4.5 브랜치 병합 견고성 워크플로우

다룰 내용:

- 브랜치별 문서 수정 충돌 패턴
- 병합 전 확인할 기준 문서
- 병합 후 재생성 또는 재검증이 필요한 색인
- 공통 문서와 호스트별 문서를 분리 관리하는 원칙

핵심 질문:

- 어떤 문서는 수동 병합, 어떤 문서는 자동 재생성이 적절한가
- 병합 이후 신뢰 가능한 기준 문서를 어떻게 재확정할 것인가

## 5. AI 친화 구조 설계 원칙

AI agent 가 불필요한 탐색 없이 빠르게 참조하려면 다음 원칙이 필요하다.

- 문서마다 목적이 단일해야 한다.
- 요약 문서와 상세 문서를 분리한다.
- 색인 문서는 긴 설명보다 탐색 키와 링크를 우선한다.
- 상태 문서와 사실 문서를 분리한다.
- 환경별 내용은 공통 문서에 섞지 않는다.
- 경로, 책임, 주요 엔트리포인트, 관련 테스트, 관련 문서를 한 번에 찾을 수 있어야 한다.

실행 방향:

- 사람용 문서는 설명과 맥락을 유지한다.
- AI 용 문서는 짧은 규칙, 경로 맵, 탐색 순서, 변경 영향 매트릭스를 중심으로 구성한다.
- 동일 내용을 중복 복제하기보다, 사람용 문서와 AI 용 문서가 서로 다른 깊이의 링크 허브 역할을 하게 한다.

## 6. 자동화 범위 초안

자동화는 문서를 사람이 대신 쓰는 방향보다, 누락을 줄이고 일관성을 검사하는 방향으로 먼저 설계한다.

### 6.1 우선 자동화 후보

- 백로그 템플릿 생성 자동화
- 호스트별 환경 폴더 생성 자동화
- 코드 변경 경로 기반 영향 문서 후보 출력
- 문서 메타데이터 누락 검사
- 색인 문서 링크 유효성 검사
- 세션 종료 시 갱신 누락 체크리스트 생성

### 6.2 후순위 자동화 후보

- 코드 색인 초안 생성
- 병합 후 색인 재생성
- PR 전 문서 동기화 점검 리포트 생성
- AI quickstart 문서의 변경 영향 자동 갱신

### 6.3 자동화 원칙

- 먼저 사람이 이해할 수 있는 규칙을 문서화한다.
- 규칙이 안정된 뒤 스크립트와 CI 로 옮긴다.
- 자동 생성 결과는 사람이 검토 가능한 형태로 남긴다.
- 호스트별 기록과 세션별 상태는 완전 자동화보다 반자동 보조를 우선한다.

## 7. 단계별 수행 계획

### 단계 1. 계획 확정

- 목표, 범위, 산출물, 우선순위를 이 문서로 확정한다.
- 오늘 세션에서는 이 단계까지 수행한다.

완료 기준:

- 워크플로우 카테고리와 후속 문서 목록이 합의 가능 수준으로 정리돼 있어야 한다.
- 다음 단계에서 어떤 문서를 먼저 만들지 결정할 수 있어야 한다.

### 단계 2. 기준 문서 분해

- 기존 `standard_workflow_draft.md` 를 기준 문서와 상세 문서로 분리한다.
- 작업 진행, 문서화, 세션 인계, 검증 워크플로우 문서를 작성한다.

완료 기준:

- 운영 위키에서 카테고리별 진입이 가능해야 한다.
- 각 문서의 책임 범위가 겹치지 않아야 한다.

### 단계 3. 문서 동기화 및 색인 정책 수립

- 코드 변경 시 확인할 문서 매핑 규칙을 정의한다.
- 색인 문서 구조와 AI quickstart 구조를 정의한다.

완료 기준:

- 코드 수정 시 어떤 문서를 같이 갱신할지 판단 기준이 문서화돼 있어야 한다.
- AI agent 가 최소 경로로 참조할 문서 집합이 정리돼 있어야 한다.

### 단계 4. 병합 견고성 및 자동화 계획 수립

- 병합 충돌에 강한 문서 운영 규칙을 정의한다.
- 자동화 우선순위와 구현 단위를 문서화한다.

완료 기준:

- 병합 후 깨질 가능성이 높은 문서와 재검증 절차가 정리돼 있어야 한다.
- 자동화 착수 가능한 작업 목록이 준비돼 있어야 한다.

### 단계 5. 자동화 구현

- 우선순위가 높은 스크립트 또는 CI 보조를 구현한다.
- 운영 문서에 사용법과 실패 시 대응 절차를 추가한다.

완료 기준:

- 최소 1개 이상 자동화가 실제 저장소에서 재사용 가능해야 한다.
- 문서와 검증 절차가 함께 정리돼 있어야 한다.

## 8. 제안하는 다음 실행 순서

다음 단계에서 우선 수행할 후보는 아래 순서가 적절하다.

1. `작업 진행 워크플로우` 상세 문서 작성
2. `문서화 및 동기화 워크플로우` 상세 문서 작성
3. `코드 색인 전략` 문서 작성
4. `AI agent quickstart` 문서 작성
5. `자동화 계획` 문서 작성

이 순서를 권장하는 이유는, 자동화보다 먼저 수동 규칙과 문서 책임 경계를 분명히 해야 이후 충돌과 중복을 줄일 수 있기 때문이다.

## 9. 현재 결정이 필요한 항목

다음 단계에 들어가기 전에 확인하면 좋은 쟁점은 아래와 같다.

- 색인 문서를 `사람용 + AI 용 분리`로 갈지, 한 문서 안에 `요약/상세` 구조로 갈지
- 코드 분석 문서의 기준 단위를 `도메인`, `모듈`, `디렉터리`, `기능` 중 무엇으로 잡을지
- 자동화 범위를 로컬 스크립트 중심으로 시작할지, CI 점검까지 바로 포함할지
