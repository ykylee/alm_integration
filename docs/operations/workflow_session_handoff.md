# 세션 인계 워크플로우

- 문서 목적: 새 세션 시작과 기존 세션 종료 시 어떤 순서로 상태를 확인하고 기록해야 하는지 표준 절차를 정의한다.
- 범위: 세션 시작 절차, 세션 종료 절차, 상태 문서 갱신 규칙, 호스트 전환 시 확인 항목, 예외 처리
- 대상 독자: 개발자, 운영자, 문서 작성자, AI agent
- 상태: draft
- 최종 수정일: 2026-04-16
- 관련 문서: `docs/operations/session_handoff.md`, `docs/operations/workflow_task_execution.md`, `docs/operations/work_backlog.md`, `docs/operations/environments/README.md`, `AGENTS.md`

## 1. 문서 개요

이 문서는 세션 인계의 절차만 다룬다. 현재 상태 요약은 [session_handoff.md](./session_handoff.md) 가 담당하고, 이 문서는 그 상태 문서를 언제 어떻게 읽고 갱신하는지를 정의한다.

핵심 목적은 아래와 같다.

- 새 세션이 이전 맥락 복원에 시간을 과도하게 쓰지 않게 한다.
- 세션 종료 시 다음 세션이 바로 이어받을 수 있는 최소 정보를 남긴다.
- 작업 상태와 환경 상태를 같은 순서로 확인하게 만든다.

## 2. 문서 역할 구분

이 문서가 담당하는 것:

- 세션 시작 체크 순서
- 세션 종료 체크 순서
- 상태 문서 갱신 시점
- 호스트 변경 시 추가 확인 항목

이 문서가 담당하지 않는 것:

- 현재 진행 중인 실제 작업 목록 관리
- 작업별 세부 기록
- 환경 설치 절차의 상세 설명

관련 문서:

- 현재 상태 요약: [session_handoff.md](./session_handoff.md)
- 작업 실행 절차: [workflow_task_execution.md](./workflow_task_execution.md)
- 백로그 운영 기준: [work_backlog.md](./work_backlog.md)
- 환경 기록 위키: [environments/README.md](./environments/README.md)

## 3. 세션 시작 절차

세션 시작 시에는 아래 순서를 기본으로 따른다.

1. [session_handoff.md](./session_handoff.md) 를 먼저 읽는다.
2. `현재 작업 요약`, `진행 중 작업`, `차단 작업`, `잔여 작업 우선순위` 를 확인한다.
3. [work_backlog.md](./work_backlog.md) 와 최신 날짜 백로그를 읽는다.
4. `in_progress`, `blocked`, `done but follow-up needed` 성격의 항목을 다시 분류한다.
5. 현재 호스트 환경 문서가 있는지 확인하고 읽는다.
6. 이번 세션에서 이어받을 작업과 새로 시작할 작업을 결정한다.
7. 브리핑 후 오늘 작업을 백로그에 등록하거나 기존 항목을 이어받는다.

세션 시작의 최소 산출물:

- 이번 세션의 우선 작업 1개 이상
- 확인이 필요한 리스크 목록
- 현재 호스트 기준 환경 확인 여부

## 4. 새 호스트 또는 새 환경에서 시작할 때

새 환경에서 시작할 때는 일반 세션 시작 절차에 아래를 추가한다.

1. `hostname`, `hostname -I` 로 현재 호스트 식별 정보를 확인한다.
2. `docs/operations/environments/<hostname>-<ip>/` 문서가 있는지 확인한다.
3. 없으면 새 폴더와 문서를 만들고 현재 환경 상태를 기록한다.
4. 기존 공통 절차와 현재 호스트 특이사항을 구분한다.
5. 이전 세션의 호스트와 현재 호스트가 다르면 차이를 짧게 남긴다.

기록 원칙:

- 공통 규칙은 공통 문서에 쓴다.
- 현재 호스트에서만 유효한 제약은 호스트 문서에 쓴다.
- 호스트가 불명확한 과거 기록은 `env_old` 로 유지한다.

## 5. 세션 종료 절차

세션 종료 전에는 아래 순서를 따른다.

1. 오늘 작업한 백로그 항목의 상태를 `planned`, `in_progress`, `blocked`, `done` 으로 확정한다.
2. 각 항목의 `작업 결과`, `다음 세션 시작 포인트`, `남은 리스크`, `후속 작업` 을 채운다.
3. 환경 변경이 있었으면 호스트별 환경 문서를 갱신한다.
4. [session_handoff.md](./session_handoff.md) 의 `진행 중 작업`, `차단 작업`, `최근 완료 작업`, `잔여 작업 우선순위` 를 갱신한다.
5. 다음 세션이 먼저 읽어야 할 문서나 명령이 있으면 인계 문서에 반영한다.

세션 종료의 최소 기준:

- `in_progress` 또는 `blocked` 상태 작업이 왜 남았는지 문서로 설명돼 있어야 한다.
- 다음 세션이 무엇부터 할지 한 문장으로 알 수 있어야 한다.
- 상태 문서와 백로그가 서로 모순되지 않아야 한다.

## 6. 상태 문서 갱신 규칙

### 6.1 언제 갱신할지

- 세션 시작 직후 실제 상태와 다르면 즉시 갱신
- 영향이 큰 작업 완료 직후 갱신
- 차단 상태 발생 직후 갱신
- 세션 종료 직전 반드시 한 번 갱신

### 6.2 무엇을 갱신할지

- `현재 작업 요약`
- `진행 중 작업`
- `차단 작업`
- `최근 완료 작업`
- `잔여 작업 우선순위`
- 필요 시 `환경별 검증 현황`

### 6.3 무엇을 넣지 말아야 하는지

- 작업별 상세 로그 전부
- 긴 명령 실행 내역 전체
- 특정 세션에서만 의미 있는 임시 메모

이런 상세 정보는 날짜별 백로그가 담당한다.

## 7. 세션 인계 품질 기준

좋은 인계 문서는 아래 조건을 만족해야 한다.

- 5분 안에 현재 기준선을 복원할 수 있다.
- 다음 세션의 첫 행동이 드러난다.
- 현재 막힌 이유가 분명하다.
- 호스트 차이로 인한 리스크가 분리돼 있다.

좋지 않은 인계 문서의 예:

- 최근 완료만 있고 남은 작업 우선순위가 없음
- `진행 중` 이라고 써 있지만 무엇이 남았는지 없음
- 환경 문제가 있었는데 호스트 문서가 없음
- 상세 로그를 그대로 복붙해서 핵심 상태가 가려짐

## 8. 자주 발생하는 실패 패턴

- 세션 종료 전에 백로그만 갱신하고 상태 문서를 안 갱신하는 경우
- 새 호스트에서 시작했는데 환경 문서를 새로 만들지 않는 경우
- 완료 작업만 적고 후속 작업을 남기지 않는 경우
- `blocked` 원인을 추상적으로만 적는 경우
- 상태 문서와 최신 백로그가 서로 다른 상태를 말하는 경우

## 9. 빠른 체크리스트

세션 시작 체크:

- `session_handoff.md` 를 먼저 읽었는가
- 최신 백로그를 읽었는가
- 현재 호스트 환경 문서를 확인했는가
- 이어받을 작업과 신규 작업을 구분했는가

세션 종료 체크:

- 백로그 상태를 확정했는가
- 다음 세션 시작 포인트를 적었는가
- 남은 리스크와 후속 작업을 적었는가
- `session_handoff.md` 를 갱신했는가
- 환경 변화가 있었다면 호스트 문서를 갱신했는가

## 10. 예외 처리

### 10.1 세션 종료 시간이 부족할 때

- 최소한 백로그의 상태와 다음 세션 시작 포인트는 갱신한다.
- `session_handoff.md` 는 진행 중/차단/최근 완료 중 핵심 항목만 먼저 맞춘다.

### 10.2 상태가 불명확할 때

- 확정처럼 쓰지 말고 `가정`, `미검증`, `추정` 으로 표시한다.
- 상세 근거는 백로그에 남기고 상태 문서에는 짧게 요약한다.

### 10.3 여러 작업이 동시에 열려 있을 때

- 우선순위가 높은 항목만 `진행 중 작업` 에 올리고, 나머지는 최신 백로그에서 근거를 찾을 수 있게 한다.
- 상태 문서가 장황해지면 요약 역할을 잃으므로 항목 수를 제어한다.
