# 작업 진행 워크플로우

- 문서 목적: 세션 시작 이후 작업 브리핑, 백로그 등록, 진행 중 갱신, 종료 기록까지의 작업 실행 절차를 표준화한다.
- 범위: 작업 착수, 작업 단위 정의, 백로그 기록, 진행 중 갱신, 종료 정리, 예외 처리
- 대상 독자: 개발자, 문서 작성자, 운영자, AI agent
- 상태: draft
- 최종 수정일: 2026-04-16
- 관련 문서: `docs/operations/standard_workflow_draft.md`, `docs/operations/work_backlog.md`, `docs/operations/session_handoff.md`, `docs/operations/backlog/2026-04-16.md`, `AGENTS.md`

## 1. 문서 개요

이 문서는 실제 작업을 어떻게 시작하고, 어떻게 기록하며, 어떤 시점에 상태를 갱신하고, 어떻게 마무리할지를 다룬다. 환경 셋업 절차, 검증 절차, 문서 동기화 기준처럼 별도 책임이 필요한 내용은 이 문서에서 모두 설명하지 않고 연결만 제공한다.

이 문서의 목적은 아래 세 가지다.

- 세션마다 작업 시작 방식이 흔들리지 않게 한다.
- 백로그와 세션 인계 문서가 실제 작업 상태를 따라가게 만든다.
- 사람과 AI agent 가 같은 작업 단위를 기준으로 협업하게 만든다.

## 2. 이 문서가 다루는 것과 다루지 않는 것

이 문서가 다루는 것:

- 작업 시작 전 확인 순서
- 작업 브리핑 작성 기준
- 백로그 항목 생성과 상태 전이
- 작업 중간 갱신 시점
- 세션 종료 전 정리 절차

이 문서가 자세히 다루지 않는 것:

- 호스트별 도구 설치와 환경 복구 상세 절차
- 로컬/격리/실행 검증 상세 절차
- 코드 변경 시 문서 동기화 판단 기준
- 코드 색인 작성 기준

관련 상세 문서:

- 상위 허브: [standard_workflow_draft.md](./standard_workflow_draft.md)
- 세션 상태 요약: [session_handoff.md](./session_handoff.md)
- 백로그 운영 기준: [work_backlog.md](./work_backlog.md)
- 환경 공통 절차: [development_environment.md](./development_environment.md)

## 3. 핵심 원칙

- 하나의 작업은 하나의 백로그 항목으로 추적 가능해야 한다.
- 작업 시작 전에 목적, 범위, 산출물, 영향 문서를 먼저 적는다.
- 진행 상태는 기억이 아니라 문서로 남긴다.
- `in_progress` 와 `blocked` 는 현재 상태를 반영해야 하며 방치하면 안 된다.
- 세션 종료 시 다음 세션 시작 포인트가 남아 있어야 한다.
- 호스트 의존 정보는 작업 기록과 분리해 환경 문서에 남긴다.

## 4. 작업 실행 표준 흐름

### 4.1 세션 시작 직후 확인

작업에 착수하기 전에 아래 순서를 따른다.

1. [session_handoff.md](./session_handoff.md) 를 읽는다.
2. [work_backlog.md](./work_backlog.md) 와 최신 날짜 백로그를 읽는다.
3. `in_progress`, `blocked`, `done but follow-up needed` 성격의 항목을 구분한다.
4. 현재 호스트 환경 문서가 있으면 확인하고, 없으면 생성 필요 여부를 판단한다.
5. 이번 세션에서 이어받을 작업과 새로 시작할 작업을 구분한다.

이 단계의 산출물:

- 이번 세션에 처리할 작업 후보
- 이어받을 작업과 신규 작업의 구분
- 확인이 필요한 리스크 목록

### 4.2 작업 브리핑 작성

작업을 시작하기 전에는 짧은 브리핑을 먼저 정리한다.

브리핑에 들어가야 할 최소 항목:

- 수행 목적
- 범위
- 예상 산출물
- 영향 가능 문서
- 현재 알고 있는 제약 또는 리스크

브리핑 원칙:

- 구현 방안을 길게 설계하기보다 이번 세션의 작업 단위를 명확히 적는다.
- 무엇을 하지 않을지도 범위에 포함해 두면 이후 범위 팽창을 줄일 수 있다.
- 문서 작업이어도 영향 코드나 연계 문서를 같이 적는다.

### 4.3 백로그 항목 생성 또는 이어받기

브리핑 후에는 백로그 문서에 작업 항목을 생성하거나 기존 항목을 이어받는다.

새 작업 생성 기준:

- 이전 백로그 항목으로 설명하기 어려운 새 목적이 생겼을 때
- 결과물과 완료 기준이 분리되어야 할 때
- 같은 날이라도 독립적으로 추적해야 할 작업일 때

기존 작업 이어받기 기준:

- 이미 `in_progress` 또는 `blocked` 상태로 존재하는 작업을 계속할 때
- 완료 기준과 산출물이 기존 항목과 동일할 때

백로그 필수 항목:

- 작업명
- 상태
- 우선순위
- 요청일
- 완료일
- 담당
- 호스트명
- 호스트 IP
- 영향 문서
- 작업 내용
- 진행 현황
- 완료 기준
- 작업 결과
- 다음 세션 시작 포인트
- 남은 리스크
- 후속 작업

### 4.4 작업 상태 전이 기준

상태는 아래 기준으로 관리한다.

- `planned`
  - 아직 착수하지 않았지만 이번 세션 또는 가까운 후속 세션에서 수행할 작업
- `in_progress`
  - 현재 세션에서 실제로 진행 중이며, 중단 시 다음 세션 인계가 필요한 작업
- `blocked`
  - 진행하려 했지만 외부 조건, 의사결정 부재, 환경 문제 등으로 멈춘 작업
- `done`
  - 완료 기준을 충족했고 작업 결과와 후속 작업이 기록된 작업

상태 관리 원칙:

- 실제로 손을 대기 전까지는 `planned` 로 둔다.
- 착수 후 결과를 아직 확정하지 못했으면 `in_progress` 로 바꾼다.
- 중단 사유가 명확하면 바로 `blocked` 로 전환하고 원인을 기록한다.
- 완료 시 `done` 으로 바꾸고 결과를 비워두지 않는다.

## 5. 작업 중간 갱신 규칙

다음 상황에서는 백로그를 중간 갱신한다.

- 중요한 결정이 내려졌을 때
- 범위가 바뀌었을 때
- 영향 문서가 추가 또는 제거됐을 때
- 예상과 다른 기술적 문제를 발견했을 때
- 테스트 또는 실행 검증 결과가 나왔을 때

진행 현황에는 아래 내용을 짧게 남긴다.

- 언제 갱신했는지
- 무엇을 확인했는지
- 무엇이 바뀌었는지
- 다음으로 무엇을 할지

좋은 예:

- `2026-04-16 작업 허브 문서를 재정의하고 분리 대상 문서를 확정했다. 다음으로 작업 진행 상세 문서를 작성한다.`

나쁜 예:

- `문서 수정 중`

## 6. 세션 종료 전 정리 절차

세션 종료 전에는 아래 순서를 최소 기준으로 따른다.

1. 오늘 작업한 항목의 상태를 확정한다.
2. `작업 결과` 를 비워두지 않는다.
3. `다음 세션 시작 포인트` 를 실제 다음 행동 기준으로 작성한다.
4. `남은 리스크` 와 `후속 작업` 을 분리해서 적는다.
5. 세션 인계 문서에 영향이 큰 변경이면 [session_handoff.md](./session_handoff.md) 를 갱신한다.
6. 환경 변화가 있었으면 호스트별 환경 문서를 갱신한다.

구분 기준:

- `남은 리스크`
  - 아직 해결되지 않았고 다음 작업 품질이나 일정에 영향을 줄 수 있는 요소
- `후속 작업`
  - 이어서 수행해야 할 구체 작업
- `다음 세션 시작 포인트`
  - 다음 세션이 가장 먼저 해야 할 한 문장 행동 지시

## 7. 작업 단위 설계 기준

작업 단위를 너무 크게 잡으면 인계가 어려워지고, 너무 작게 잡으면 문서가 과도하게 쪼개진다. 아래 기준으로 균형을 맞춘다.

하나의 작업으로 묶기 좋은 경우:

- 하나의 명확한 목표를 향해 움직일 때
- 완료 기준이 한 문장으로 정의될 때
- 영향 문서 집합이 크게 흔들리지 않을 때

작업을 쪼개야 하는 경우:

- 구현과 문서화, 자동화처럼 책임이 다를 때
- 하나의 작업 안에 검토/구현/배포 수준이 섞일 때
- 다음 세션에서 맥락 복원 비용이 지나치게 커질 때

## 8. 예외 처리

### 8.1 작업 범위가 커졌을 때

- 기존 작업의 완료 기준을 다시 쓰지 말고, 새 작업으로 분리 가능한지 먼저 본다.
- 분리가 적절하면 새 `TASK` 를 만들고 관계를 명시한다.

### 8.2 차단 상태가 발생했을 때

- 차단 원인을 한 줄로 적지 말고, 무엇이 없어서 막혔는지 구체적으로 적는다.
- 환경 문제인지, 의사결정 문제인지, 외부 의존 문제인지 구분한다.
- 차단 해소를 위해 필요한 다음 행동을 적는다.

### 8.3 같은 작업을 여러 세션에서 이어갈 때

- 매 세션 종료 시 `다음 세션 시작 포인트` 를 갱신한다.
- 실제 상태가 바뀌었으면 같은 작업 안의 `진행 현황` 을 누적한다.
- 완전히 다른 목표로 변했으면 새 작업으로 분리한다.

## 9. 빠른 체크리스트

작업 시작 체크:

- `session_handoff.md` 를 읽었는가
- 최신 백로그를 읽었는가
- 이번 작업 목적과 범위를 한 문장 이상으로 정리했는가
- 영향 문서를 적었는가
- 백로그 항목을 만들었거나 이어받았는가

작업 종료 체크:

- 상태를 최신으로 바꿨는가
- 작업 결과를 적었는가
- 다음 세션 시작 포인트를 적었는가
- 남은 리스크와 후속 작업을 분리했는가
- 필요하면 `session_handoff.md` 를 갱신했는가
