# 브랜치 병합 문서 정책

- 문서 목적: 여러 브랜치의 변경을 병합할 때 문서, 색인, 세션 인계 정보, 환경 기록이 깨지지 않도록 병합 기준과 재확정 절차를 정의한다.
- 범위: 병합 전 확인 항목, 병합 충돌 패턴, 문서 유형별 병합 원칙, 병합 후 재확정 절차, 재생성 가능 항목 구분
- 대상 독자: 개발자, 문서 작성자, 리뷰어, AI agent
- 상태: draft
- 최종 수정일: 2026-04-16
- 관련 문서: `docs/operations/standard_workflow_draft.md`, `docs/operations/workflow_documentation_sync.md`, `docs/operations/code_index_strategy.md`, `docs/operations/ai_agent_quickstart.md`, `docs/operations/session_handoff.md`, `docs/operations/work_backlog.md`, `AGENTS.md`

## 1. 문서 개요

이 문서는 브랜치 병합이 단순히 Git 충돌을 해소하는 작업이 아니라, 병합 이후에도 문서 기준선과 탐색 경로가 신뢰 가능해야 한다는 점을 전제로 한다.

병합 후 자주 생기는 문제는 아래와 같다.

- 문서 링크는 병합됐지만 실제 기준 사실이 서로 다르게 남는 문제
- `session_handoff.md` 와 날짜별 백로그가 서로 다른 상태를 가리키는 문제
- 색인 문서가 옛 경로나 옛 엔트리포인트를 계속 가리키는 문제
- 호스트별 환경 문서를 공통 문서처럼 병합해 환경 차이가 흐려지는 문제

이 문서의 목표는 충돌을 줄이는 것뿐 아니라, 병합 후 어떤 문서를 다시 확인하고 어떤 문서를 재생성 또는 재정리해야 하는지 표준화하는 것이다.

## 2. 핵심 원칙

- 기준 사실 문서는 병합 후 하나의 최신 상태만 남아야 한다.
- 상태 문서와 사실 문서는 같은 방식으로 병합하지 않는다.
- 환경별 문서는 공통 문서와 분리해서 유지한다.
- 재생성 가능한 정보와 수동 판단이 필요한 정보를 분리한다.
- 병합 완료는 충돌 해결이 아니라 기준 문서 재확정까지 끝난 상태를 의미한다.

## 3. 문서 유형별 병합 우선순위

### 3.1 기준 사실 문서

대상:

- 요구사항 문서
- 아키텍처 문서
- 운영 규칙 문서

병합 원칙:

- 단순 양쪽 내용 병합보다 현재 기준 사실이 무엇인지 먼저 재판단한다.
- 두 브랜치가 서로 다른 정책을 주장하면 병합 커밋에서 기준 사실을 하나로 재정리한다.
- 애매한 상태를 그대로 두지 말고 `가정`, `미정`, `추정` 으로 명시한다.

### 3.2 상태 문서

대상:

- `session_handoff.md`
- 날짜별 백로그 문서

병합 원칙:

- 두 브랜치의 상태 문서를 기계적으로 합치지 않는다.
- 병합 시점의 최신 상태를 기준으로 다시 쓴다.
- 완료 작업, 진행 중 작업, 잔여 작업 우선순위를 다시 정리한다.

### 3.3 색인 및 AI 참조 문서

대상:

- `code_index_strategy.md`
- `ai_agent_quickstart.md`
- 향후 도메인/기능 색인 문서

병합 원칙:

- 링크와 경로가 여전히 유효한지 다시 확인한다.
- 기준 사실 문서가 바뀌면 색인 문서가 그 문서를 정확히 가리키는지 재검토한다.
- 빠른 참조 문서는 병합 후 탐색 순서가 끊기지 않는지 우선 확인한다.

### 3.4 환경 문서

대상:

- `docs/operations/environments/*`

병합 원칙:

- 호스트별 기록은 공통 문서처럼 합치지 않는다.
- 서로 다른 호스트 기록은 독립 유지한다.
- 같은 호스트 문서를 병합할 때만 설치 상태, 검증 시점, 제약 사항을 시간순으로 정리한다.

## 4. 자주 발생하는 충돌 패턴

### 4.1 허브 문서 링크 충돌

예:

- 한 브랜치는 새 상세 문서를 추가했고, 다른 브랜치는 기존 링크 구조를 수정한 경우

해결 원칙:

- 링크 목록만 합치지 말고, 실제 진입 순서가 자연스러운지 확인한다.
- `docs/README.md`, 카테고리 `README.md`, 허브 문서가 같은 기준선으로 연결되는지 본다.

### 4.2 상태 요약 충돌

예:

- 두 브랜치가 각각 다른 작업을 완료하고 각자 `session_handoff.md` 를 갱신한 경우

해결 원칙:

- 병합 후 실제 최근 완료 작업을 다시 추려 적는다.
- `in_progress`, `blocked`, `우선순위` 섹션은 현재 시점 기준으로 재작성한다.

### 4.3 색인 경로 충돌

예:

- 한 브랜치는 파일 이동을 했고, 다른 브랜치는 기존 경로를 전제로 색인 문서를 확장한 경우

해결 원칙:

- 실제 최종 경로를 기준으로 색인 문서를 다시 정리한다.
- 파일 경로, 엔트리포인트, 테스트 경로를 병합 후 재검증한다.

### 4.4 공통 문서와 환경 문서 혼합 충돌

예:

- 특정 호스트에서만 유효한 설치 메모가 공통 운영 문서에 들어간 경우

해결 원칙:

- 공통 규칙은 공통 문서에 남기고, 호스트 특이사항은 환경 문서로 옮긴다.
- 공통 문서에 남길 내용은 재현 가능한 기준 절차만 유지한다.

## 5. 병합 전 체크리스트

- 두 브랜치가 각각 수정한 기준 사실 문서가 무엇인지 확인했는가
- `session_handoff.md` 와 날짜별 백로그가 양쪽에서 모두 바뀌었는가
- 경로 이동, 명령 변경, 문서 이동이 있었는가
- 색인 또는 quickstart 가 참조하는 링크가 깨질 가능성이 있는가
- 호스트별 환경 문서가 공통 문서로 잘못 흡수되지 않았는가

## 6. 병합 중 처리 원칙

1. 기준 사실 문서를 먼저 해결한다.
2. 그 기준을 따라 상세 절차 문서를 맞춘다.
3. 허브 문서와 인덱스를 맞춘다.
4. 상태 문서를 현재 시점 기준으로 다시 쓴다.
5. 색인과 quickstart 를 마지막에 갱신한다.

이 순서를 쓰는 이유:

- 허브와 색인은 기준 사실을 따라가야 하므로 먼저 고칠 대상이 아니다.
- 상태 문서는 병합 전 각 브랜치의 상태가 아니라 병합 후 기준 상태를 반영해야 한다.

## 7. 수동 병합과 재생성의 경계

### 7.1 수동 판단이 필요한 항목

- 정책 설명
- 책임 분리 설명
- 우선순위
- 남은 리스크
- 세션 인계 요약

### 7.2 재생성 또는 자동 점검이 가능한 항목

- 링크 유효성 확인
- 문서 메타데이터 누락 확인
- 정렬 가능한 경로 목록
- 색인 문서의 파일 경로 존재 여부

원칙:

- 설명 문장은 사람이 최종 판단한다.
- 경로/링크/메타데이터 검사는 자동화 후보로 분리한다.

## 8. 병합 후 재확정 절차

병합이 끝나면 아래 순서로 다시 확인한다.

1. `session_handoff.md` 를 현재 기준으로 갱신한다.
2. 관련 날짜 백로그에 병합 결과와 영향 문서를 반영한다.
3. 허브 문서와 인덱스 문서 링크를 확인한다.
4. 색인 문서와 quickstart 를 갱신한다.
5. 필요 시 검증 문서 또는 환경 문서를 갱신한다.

병합 후 반드시 확인할 문서:

- `docs/README.md`
- `docs/operations/README.md`
- `docs/operations/session_handoff.md`
- 관련 날짜 백로그
- `docs/operations/ai_agent_quickstart.md`
- 관련 색인 문서

## 9. 병합 완료 기준

아래 조건을 만족해야 문서 기준 병합이 완료된 것으로 본다.

- 기준 사실 문서가 하나의 최신 상태로 정리됐다.
- 허브와 인덱스 링크가 깨지지 않는다.
- `session_handoff.md` 와 백로그가 현재 상태를 반영한다.
- 색인과 quickstart 가 최종 경로를 가리킨다.
- 공통 문서와 환경 문서의 경계가 유지된다.

## 10. AI agent 관점의 주의사항

- 병합 충돌 마커를 지우는 것만으로 끝났다고 판단하지 않는다.
- 병합 후에는 항상 `session_handoff.md` 와 최신 백로그를 다시 읽는다.
- quickstart 의 탐색 순서가 바뀌었으면 즉시 갱신한다.
- 경로 이동이 있었으면 색인 문서와 테스트 경로를 우선 확인한다.
