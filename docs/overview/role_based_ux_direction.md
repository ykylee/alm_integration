## 1. 배경

현재 정적 프로토타입은 기안서의 핵심 기능을 빠르게 보여주는 데에는 유효하지만, 실제 사용 관점에서는 역할별 업무면이 섞여 있다. 특히 다음 문제가 보인다.

- 개발자가 자주 보는 과제 진행 화면과 관리자성 운영 화면이 같은 탐색 레벨에 있다.
- 프로젝트 운영 판단, 조직 승인, 시스템 운영, 품질 검증처럼 결재와 책임 주체가 다른 기능이 같은 화면 문맥에 섞여 있다.
- 조직 변경 반영, 연계 설정, 권한 관리처럼 시스템 관리자만 수행해야 하는 기능이 일반 사용자 흐름에서 너무 가깝게 노출된다.
- 과제 상세가 메인 화면 내부 탭처럼 보이는데, 실제 업무에서는 과제별 고유 맥락과 깊은 탐색이 필요해 독립 워크스페이스가 더 자연스럽다.

## 2. 역할별 핵심 관심사

### 2.1 개발자

개발자는 과제나 변경 단위로 일을 본다. 시스템 설정이나 조직 마이그레이션 자체보다, 지금 담당 중인 과제가 어떤 상태인지와 다음 액션이 무엇인지가 더 중요하다.

- 오늘 내가 봐야 할 과제, `PR`, 빌드, 테스트, 리뷰 대기
- 연결된 `Jira`, `Bitbucket`, `Bamboo` 상태와 이상 신호
- `AI` 리뷰 초안, 코딩룰 위반 의심, 단위테스트 보완 필요 항목
- 리뷰어 가용성, 부재로 인한 일정 영향

개발자 UX는 "내 작업 우선", "과제 맥락 우선", "즉시 액션 가능"이 핵심이다.

### 2.2 프로젝트리더

프로젝트리더는 개인 작업보다 과제 또는 프로젝트의 납기, 위험, 의사결정 대기 항목을 본다. 핵심 관심사는 범위, 일정, 품질 위험, 승인 병목, 외부 시스템 상태가 일정에 미치는 영향이다.

- 내 담당 과제의 진행률, 주요 마일스톤, 위험 신호
- 요구사항 변경, `PR` 정체, 빌드 실패, 테스트 미통과, 배포 지연
- 조직장 승인 필요 항목, 타 조직 협업 지연, 리소스 부족
- 과제별 산출물 완결도와 대외 보고용 상태 요약

프로젝트리더 UX는 "진행 통제", "위험 우선", "의사결정 대기 최소화"가 핵심이다.

### 2.3 조직장

조직장은 개인 과제보다 조직 전체의 자원 배치, 승인 책임, 인력 가용성, 조직 단위 리스크를 본다. 핵심 관심사는 어떤 과제를 승인하거나 조정해야 하는지, 조직의 병목이 어디서 발생하는지다.

- 조직별 활성 과제 수, 중요도별 분포, 지연 과제
- 승인 대기 항목, 관리 외 프로젝트 전환 요청, 조직 변경 영향
- 공유 캘린더, 휴가/출장, 핵심 인력 부재로 인한 일정 영향
- 조직장 승인 이후 발생할 비용, 리소스, 일정 영향

조직장 UX는 "조직 단위 관찰", "승인과 우선순위 조정", "가용성 관리"가 핵심이다.

### 2.4 시스템 관리자

시스템 관리자는 실제 업무 과제보다 시스템 운영 안정성과 데이터 거버넌스를 본다. 이 역할은 일반 사용자 화면과 분리된 관리 콘솔을 사용하는 편이 맞다.

- 외부 연계 설정, 동기화 상태, 실패 재처리
- 조직/인력 원천 반영, 매핑 충돌, 데이터 품질 문제
- 권한, 역할, 메뉴 접근 제어
- 조직 변경 반영, 계정 연계, 마이그레이션 실행

시스템 관리자 UX는 "안전한 운영", "명시적 승인", "감사 가능한 변경 실행"이 핵심이다.

### 2.5 `SE` 담당자

`SE` 담당자는 설계 표준, 환경 구성, 빌드/배포 흐름, 운영 반영 가능성을 본다. 이 역할은 단순 개발자 화면보다 환경과 기술 제약 정보를 더 깊게 봐야 한다.

- 과제별 아키텍처 결정, 연계 구조, 환경 제약, 배포 경로
- `CI` 플랜, 산출물 메타데이터, 배포 패키지, 인프라 의존성
- 표준 템플릿 준수 여부, 예외 구성, 운영 반영 위험
- 빌드/배포 실패 원인, 환경별 차이, 이행 계획

`SE` 담당자 UX는 "구성 가시성", "환경 차이 식별", "반영 가능성 검증"이 핵심이다.

### 2.6 테스트 담당자

테스트 담당자는 기능 개발 진행 자체보다 테스트 계획, 커버리지, 결함 상태, 릴리스 가능성을 본다. 이 역할은 요구사항과 변경을 품질 검증 문맥으로 재구성해서 봐야 한다.

- 요구사항 대비 테스트 케이스 준비율, 실행률, 실패 추세
- 결함 심각도, 재현 상태, 수정 재검증 대기
- 단위테스트, 통합테스트, 수동 테스트, 품질 지표 연결 상태
- 릴리스 후보 버전의 품질 게이트 충족 여부

테스트 담당자 UX는 "검증 기준 우선", "결함 흐름 추적", "출시 가능성 판단"이 핵심이다.

## 3. 역할별 뷰 분리 원칙

### 3.1 메인 화면은 공통 랜딩 + 역할별 진입점이어야 한다

메인 화면은 모든 세부 정보를 다 담는 대시보드가 아니라, 로그인 사용자가 어떤 역할인지에 따라 적절한 업무면으로 보내는 관문이 되어야 한다.

- 개발자: 내 과제, 내 리뷰, 내 빌드 이상, 내 테스트 보완
- 프로젝트리더: 담당 과제 현황, 위험 과제, 승인 대기, 마일스톤 지연
- 조직장: 조직 현황, 승인 대기, 인력 가용성, 과제 분포
- 시스템 관리자: 관리 콘솔 진입, 동기화 이상, 반영 대기 변경, 보안/권한 경고
- `SE` 담당자: 환경 차이, 배포 준비 상태, `CI`/산출물 이상, 구조적 위험
- 테스트 담당자: 테스트 실행 현황, 결함 추세, 릴리스 게이트 미충족, 재검증 대기

같은 데이터라도 역할에 따라 첫 문맥이 달라야 한다. 공통 메인은 존재할 수 있지만, 역할별 카드와 우선순위가 달라져야 한다.

### 3.2 시스템 관리자 화면은 일반 운영 화면과 격리해야 한다

조직 변경 반영, 계정 마이그레이션, 외부 연계 설정, 권한 변경은 단순 조회가 아니라 운영 행위다. 이런 기능은 일반 사용자 메뉴 안에 섞지 말고 별도 관리 영역으로 분리해야 한다.

권장 구조:

- 일반 업무 영역: 과제, 조직 운영, 캘린더, 보고
- 관리자 콘솔: 연계 관리, 동기화 모니터링, 조직 변경 반영, 권한 관리, 감사 로그

분리 이유:

- 실수로 실행하면 안 되는 기능이다.
- 화면의 정보 밀도와 용어 체계가 일반 사용자와 다르다.
- 승인, 재처리, 마이그레이션은 확인 단계와 이력 보존이 필수다.

### 3.3 프로젝트 운영 역할과 조직 승인 역할을 분리해야 한다

프로젝트리더와 조직장은 모두 "승인"을 보지만 승인 대상과 판단 기준이 다르다. 이 둘을 하나의 관리자 역할로 합치면 화면이 빠르게 비대해진다.

- 프로젝트리더는 과제/프로젝트 단위 일정과 범위 조정에 집중한다.
- 조직장은 조직 자원 배치, 승인 책임, 인력 가용성에 집중한다.
- 같은 승인 요청이라도 프로젝트리더는 과제 적합성을, 조직장은 조직 수용 가능성을 본다.

따라서 승인함은 공통으로 둘 수 있어도, 기본 홈과 요약 카드, 필터 기본값은 분리해야 한다.

### 3.4 품질 역할은 개발 흐름의 보조가 아니라 독립된 작업면이 필요하다

`SE` 담당자와 테스트 담당자는 모두 품질/반영 영역을 보지만 관점이 다르다.

- `SE` 담당자는 구조, 환경, 배포 가능성을 본다.
- 테스트 담당자는 검증 범위, 결함, 릴리스 가능성을 본다.

두 역할을 개발자 화면의 보조 탭으로만 두면 필요한 데이터 밀도와 상태 추적 수준이 부족해질 가능성이 크다. 과제 상세 안에 공통 데이터는 공유하되, 전용 작업면은 별도로 두는 편이 맞다.

## 4. 권장 정보구조

### 4.1 1차 탐색 구조

- 홈
- 내 작업
- 과제
- 프로젝트 운영
- 조직 운영
- 품질/검증
- 보고/검색
- 관리자 콘솔

`관리자 콘솔`은 시스템 관리자에게만 노출한다. 프로젝트리더는 프로젝트 운영, 조직장은 조직 운영, `SE` 담당자와 테스트 담당자는 품질/검증 영역을 우선 사용하고, 개발자는 주로 내 작업과 과제를 본다.

### 4.2 과제 상세는 독립 서브페이지로 분리

과제 상세는 메인 페이지 안의 한 섹션이 아니라 과제별 독립 워크스페이스여야 한다. 이유는 과제마다 연결된 요구사항, 저장소, 빌드, 테스트, 참여자, 일정, 문서가 깊게 쌓이기 때문이다.

권장 예시:

- `/tasks`
- `/tasks/TASK-2419`
- `/tasks/TASK-2419/overview`
- `/tasks/TASK-2419/requirements`
- `/tasks/TASK-2419/code-review`
- `/tasks/TASK-2419/builds`
- `/tasks/TASK-2419/release-readiness`
- `/tasks/TASK-2419/test-quality`
- `/tasks/TASK-2419/defects`
- `/tasks/TASK-2419/environment`
- `/tasks/TASK-2419/people-calendar`

이 구조가 되면 메인 화면은 "과제를 찾는 곳"이 되고, 실제 작업은 과제 워크스페이스에서 이뤄진다.

권장 배치 기준:

- 개발자: `overview`, `code-review`, `builds`, `test-quality`
- 프로젝트리더: `overview`, `requirements`, `release-readiness`, `people-calendar`
- 조직장: `overview`, `people-calendar`, 승인 관련 요약 패널
- `SE` 담당자: `environment`, `builds`, `release-readiness`
- 테스트 담당자: `test-quality`, `defects`, `release-readiness`

### 4.3 조직 변경 반영은 관리자 콘솔의 서브플로우로 이동

현재 프로토타입의 "조직 변경 검토" 화면은 기능적으로 맞지만 배치가 잘못됐다. 이 화면은 일반 탐색에 노출할 것이 아니라 다음 구조가 더 적절하다.

- `/admin`
- `/admin/hr-sync`
- `/admin/hr-sync/change-requests`
- `/admin/hr-sync/change-requests/:id`

이 안에서만 변경 확인, 영향 분석, 내부 반영, `Crowd` 후속 연계, 감사 이력을 다뤄야 한다.

### 4.4 품질/검증 영역은 역할별 서브플로우를 허용해야 한다

품질/검증 영역은 하나의 메뉴로 묶되, 역할별 시작점은 달라야 한다.

- `/quality/se-overview`
- `/quality/test-overview`
- `/quality/releases`
- `/quality/defects`
- `/quality/ci-plans`

이 구조를 쓰면 공통 품질 데이터를 공유하면서도 `SE` 담당자와 테스트 담당자의 기본 흐름을 다르게 설계할 수 있다.

## 5. 역할별 대표 화면 제안

### 5.1 개발자용

- 내 작업 홈: 내가 참여 중인 과제, 리뷰 대기, 실패 빌드, 테스트 보완 필요 항목
- 과제 목록: 담당 과제와 관심 과제 검색
- 과제 상세 워크스페이스: 요구사항, `PR`, 빌드, 테스트, 품질, 참여자, 일정
- 리뷰 작업면: `AI` 리뷰 초안, 코딩룰 판단, 리뷰 우선순위

### 5.2 프로젝트리더용

- 프로젝트 운영 홈: 마일스톤, 위험 과제, 승인 병목, 품질 게이트 미충족
- 과제/프로젝트 조정 화면: 범위 변경, 일정 조정, 우선순위 갱신
- 보고용 상태 보드: 경영/조직 보고용 요약, 주요 위험, 조치 현황
- 승인 요청 허브: 조직장 또는 시스템 관리자 판단이 필요한 항목 전달

### 5.3 조직장용

- 조직 운영 홈: 조직별 활성 과제, 승인 대기, 위험 신호
- 과제 등록/전환 승인: 일반 과제 등록, 관리 외 프로젝트 전환 요청
- 조직 캘린더: 부재 일정, 리뷰 가능 인력, 승인 일정 충돌
- 조직 영향 보기: 조직 변경 시 영향 받는 과제와 승인 루트

### 5.4 시스템 관리자용

- 운영 콘솔 홈: 동기화 실패, 데이터 불일치, 반영 대기 변경
- 연계 관리: 시스템별 연결, 인증, 동기화 주기, 상태
- 기준정보 관리: 조직/인력 원천 매핑, 내부 식별자, 충돌 해결
- 변경 반영 센터: 조직 변경 검토, 마이그레이션 실행, 후속 계정 연계
- 권한/감사: 역할, 접근 범위, 감사 로그

### 5.5 `SE` 담당자용

- `SE` 개요 홈: 환경별 반영 준비 상태, 빌드/배포 실패, 구성 예외
- `CI`/배포 작업면: 플랜, 산출물, 패키지, 환경 차이 비교
- 구조 검토 화면: 연계 구조, 표준 템플릿 준수, 반영 제약
- 릴리스 준비도 화면: 운영 반영 조건 충족 여부와 차단 항목

### 5.6 테스트 담당자용

- 테스트 개요 홈: 테스트 계획 대비 실행률, 실패 추세, 결함 집중 영역
- 결함 작업면: 재현, 상태 변경, 재검증 대기, 릴리스 차단 결함
- 테스트 설계/실행 화면: 테스트 케이스, 실행 결과, 누락 범위
- 품질 게이트 화면: 릴리스 후보의 테스트 통과 기준 충족 여부

## 6. 역할 충돌과 UX 리스크

- 프로젝트리더와 조직장의 승인함을 같은 화면으로 합치면 책임 경계가 흐려진다.
- 개발자와 테스트 담당자가 같은 "테스트" 탭을 보더라도 필요한 기본 필터와 상태 정의가 다르다.
- `SE` 담당자와 시스템 관리자를 모두 "운영"으로 묶으면 환경 설계 판단과 시스템 관리 행위가 혼동된다.
- 시스템 관리자 기능을 프로젝트리더나 조직장 홈에서 노출하면 승인 화면이 과도하게 복잡해진다.
- 한 사용자가 복수 역할을 갖는 경우 첫 랜딩은 대표 역할 기준으로 두되, 역할 전환 스위처를 상단에 두는 편이 안전하다.

## 7. 프로토타입 수정 방향

현재 프로토타입 기준으로 다음 수정이 적절하다.

- `통합 현황`은 6개 역할 중 현재 역할을 선택하거나 전환할 수 있는 랜딩으로 바꾼다.
- `과제 상세`는 메인 내 화면에서 제거하고 과제 목록에서 진입하는 독립 페이지로 바꾼다.
- `프로젝트 운영`과 `조직 운영`을 분리해 프로젝트리더와 조직장의 홈을 나눈다.
- `품질/검증` 아래에 `SE` 담당자와 테스트 담당자 전용 시작 화면을 둔다.
- `조직 변경 검토`는 메인 탐색에서 제거하고 관리자 콘솔 안으로 이동한다.
- `공유 캘린더`는 개발자 공통 화면이 아니라 조직 운영, 프로젝트 운영, 과제 상세의 하위 문맥으로 재배치한다.

## 8. 우선 적용 권장안

프로토타입을 다음 순서로 개편하는 것이 효율적이다.

1. 랜딩 페이지를 6개 역할 기준 홈 전환 구조로 재설계한다.
2. 과제 목록과 과제 상세 서브페이지 구조를 분리한다.
3. 프로젝트리더용 프로젝트 운영 화면과 조직장용 조직 운영 화면을 분리한다.
4. `SE` 담당자와 테스트 담당자용 품질/검증 작업면을 분리한다.
5. 시스템 관리자 전용 관리자 콘솔과 조직 변경 반영 플로우를 격리한다.

## 9. 현재 프로토타입 검토 메모

현재 정적 프로토타입은 초기 방향 설명용 데모로서는 충분하지만, 6개 역할 체계 기준으로는 화면 문맥이 아직 넓게 섞여 있다.

### 9.1 확인된 장점

- 과제 중심이라는 핵심 메시지가 랜딩과 과제 상세 화면에 일관되게 드러난다.
- `Jira`, `Bitbucket`, `Bamboo`, 조직/인력, 공유 캘린더를 한 제품 안의 운영 표면으로 설명하는 데는 성공적이다.
- 등록, 과제 추적, 조직 변경, 부재 관리처럼 서로 다른 운영 흐름을 빠르게 시연하기에는 단순한 화면 전환 구조가 유효하다.

### 9.2 주요 문제

- 왼쪽 탐색에 `과제 상세`, `조직 변경 검토`, `공유 캘린더`가 같은 레벨로 배치돼 역할 경계가 흐려진다.
- 랜딩 화면이 역할별 진입점이 아니라 기능 샘플 나열에 가깝다.
- 과제 상세 화면은 추적성 조회는 보여주지만 프로젝트리더, `SE` 담당자, 테스트 담당자가 각자 써야 하는 서브페이지 구조가 없다.
- 조직 변경 검토 화면은 시스템 관리자 전용 서브플로우인데 일반 메인 탐색에 드러난다.
- 공유 캘린더 화면은 조직장, 프로젝트리더, 테스트 담당자 맥락으로 재배치할 수 있는 여지가 큰데 독립 메인 메뉴로 노출돼 있다.
- 등록 화면의 승인 루트가 `PM -> 조직장 -> 운영관리자` 한 줄로 고정돼 있어 프로젝트리더와 시스템 관리자 역할 구분이 약하다.

### 9.3 개선 우선순위

1. 랜딩을 역할 선택 또는 역할 자동 진입 홈으로 바꾼다.
2. 1차 메뉴를 `과제`, `프로젝트 운영`, `조직 운영`, `품질/검증`, `관리자 콘솔` 구조로 재편한다.
3. 과제 상세를 독립 워크스페이스로 만들고 `overview`, `code-review`, `release-readiness`, `environment`, `test-quality`, `defects` 서브페이지를 둔다.
4. 조직 변경 검토는 `관리자 콘솔` 하위로 이동한다.
5. 공유 캘린더는 조직 운영과 과제 상세 안의 보조 문맥으로 재배치한다.

## 10. 가정 및 미정 사항

- 한 사용자가 복수 역할을 가질 때 기본 랜딩 규칙은 추가 정의가 필요하다.
- 프로젝트리더와 조직장의 승인 권한 분리 기준, 대행 승인 허용 범위는 추가 검토가 필요하다.
- `SE` 담당자의 범위가 배포/환경 설계 중심인지, 아키텍처 검토까지 포함하는지는 추가 정의가 필요하다.
- 테스트 담당자가 단위테스트 보조 기능까지 직접 사용하는지, 통합/시스템 테스트 중심인지 추가 정의가 필요하다.
- 시스템 관리자 콘솔을 완전히 별도 `URL` 공간으로 둘지, 같은 제품 안의 숨겨진 영역으로 둘지는 보안 정책에 따라 확정해야 한다.
- 과제 상세 서브페이지의 실제 탭 구성은 현행 `Jira`, `Bitbucket`, `Bamboo` 사용 패턴 분석 후 세분화할 필요가 있다.
