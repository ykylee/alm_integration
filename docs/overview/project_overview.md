## 1. 배경

현재 운영 환경에는 ALM을 비롯한 `SDLC` 인프라가 이미 존재하며, 대표적으로 `Jira`, `Confluence`, 형상관리 계열, `CI/CD` 계열, 정적분석 계열 도구가 함께 사용되고 있다. 현재 구상 기준으로 핵심 우선 연계 대상은 요구사항/이슈 관리의 `Jira`, 형상관리/리뷰 계열의 `Bitbucket`, `CI/CD` 계열의 `Bamboo` 다. `Bamboo` 는 `Bamboo Specs` 활용 가능성을 함께 고려한다. 다음 우선순위는 `Confluence`, 테스트 관리 도구, `KPI`/지표 관리 계열이며, 정적분석은 현행 `Coverity` 유지 방향을 우선 검토한다. 여기에 코딩 룰 검사 및 리뷰 보조 도구, 언어별 단위테스트 결과 수집 체계가 함께 운영된다.

문제는 이러한 시스템이 기능별로 나뉘어 운영되면서도 실제 업무에서는 서로 강하게 연결되어 있다는 점이다. 특히 실제 운영의 중심 단위가 되는 과제를 기준으로 보면, 요구사항, 변경 이력, 문서 상태, 빌드/배포 결과, 정적분석 결과, 테스트 결과, 품질 지표가 여러 도구에 분산되어 있어 전체 흐름을 종합적으로 파악하기 어렵다.

이 프로젝트는 초기에는 이러한 분산 정보를 중앙에서 통합 관리하고, 이후에는 축적된 정보와 운영 경험을 바탕으로 자체 `ALM` 프로토타입으로 확장해 현재 사용 중인 일부 기능을 단계적으로 내재화하는 것을 목표로 한다. 이때 우리 시스템의 중심 엔터티는 프로젝트 자체보다 과제다. 일반적인 관리 대상 업무는 과제 등록 프로세스를 통해 등록되고, 등록된 과제가 요구사항, 소스 변경, 빌드, 배포, 테스트, 문서, 품질 지표를 연결하는 기준점이 된다. 반면 아직 관리 범위 밖에 있는 프로젝트도 간소한 별도 등록 프로세스를 통해 최소 정보만 등록할 수 있어야 하며, 진행 중 필요성이 커질 경우 관리 외 프로젝트를 관리 대상 프로젝트로 승격하는 전환 경로가 제공되어야 한다. 다만 `SCM`, `CI/CD`, 정적분석 실행 계층은 초기 판단상 외부 전문 도구를 연계하는 방향을 우선하며, 통합 시스템은 코드리뷰, 빌드/산출물 메타데이터, 정적분석 결과와 같은 개발 맥락을 내부에서 일관되게 조회하고 관리하는 데 중점을 둔다. 코드리뷰 영역에는 `AI` 연계 기반 자동 리뷰 초안 작성 기능을 포함해 개발자가 변경 영향과 잠재 위험을 더 빠르게 파악할 수 있도록 하는 방향을 고려한다. 이 기능은 개발팀이 지정한 코딩룰을 기준으로 준수 여부를 함께 판단할 수 있어야 한다. 테스트 영역에는 `PR` 변경 범위를 기준으로 단위테스트 존재 여부를 점검하고, 필요 시 `AI` 가 단위테스트 초안을 작성하고 실행을 보조하는 기능과, 사용자가 지정한 파일 또는 범위를 기준으로 단위테스트를 작성하고 수행하는 기능을 함께 고려한다. `CI` 영역에는 프로젝트별 소스 언어, 컴파일러, 구성 특성과 표준 템플릿을 바탕으로 `AI` 가 `stage`/`job` 초안을 자동 제안하고, 대상 환경에 따라 `Jenkins` 또는 `Bamboo Specs` 형태의 설정 초안으로 연결될 수 있도록 하는 방향을 고려한다.

## 2. 문제 정의

현재와 같은 분산 운영 환경에서는 다음 문제가 발생한다.

- 프로젝트 상태를 종합적으로 파악하기 어렵다.
- 시스템별 데이터 기준과 식별자가 달라 추적성이 약해진다.
- 변경, 승인, 배포, 품질 상태를 연결해서 보기 어렵다.
- 운영 보고서와 감사 대응을 위해 수작업 데이터 수집이 발생한다.
- 권한 관리와 표준 프로세스를 일관되게 적용하기 어렵다.
- 단위테스트, 정적분석, 테스트 관리, 리뷰 품질과 같은 지표를 통합 기준으로 관리하기 어렵다.
- 여러 도구를 오가야 하므로 개발자 경험이 분절되고 생산성이 저하된다.

## 3. 프로젝트 목표

본 프로젝트는 단계적으로 추진된다.

### Phase 1 목표

여러 운영 시스템의 핵심 정보를 수집, 정규화, 연결, 조회할 수 있는 중앙 관리 플랫폼을 제공한다.

### Phase 2 목표

Phase 1에서 축적한 연계 구조, 표준 데이터 모델, 사용자 흐름을 바탕으로 자체 `ALM`의 프로토타입을 구축한다. 이 단계에서는 현재 사용 중인 도구의 `UX`를 참고하되, 개발자 편의성을 최우선 기준으로 삼는다.

세부 목표:

- 프로젝트, 애플리케이션, 산출물, 변경, 배포, 품질 정보를 통합 조회한다.
- 과제를 중심으로 요구사항, 형상, 빌드, 테스트, 문서, 품질 정보를 연결 관리한다.
- 외부 시스템별로 흩어진 식별자와 메타데이터를 표준 모델로 정규화한다.
- 변경에서 빌드, 배포, 품질, 문서까지 추적 가능한 연결 구조를 제공한다.
- 대시보드와 검색을 통해 운영 가시성을 높인다.
- 감사, 보고, 운영 점검에 필요한 기준 정보를 일관되게 제공한다.
- 관리자가 연계 대상 시스템과 관리 대상을 추가, 수정할 수 있는 확장 가능한 운영 구조를 제공한다.
- 일반 과제 등록 프로세스와 관리 외 프로젝트의 간소 등록 프로세스를 모두 제공한다.
- 관리 외 프로젝트를 진행 중 관리 대상 프로젝트로 전환할 수 있는 운영 루트를 제공한다.
- 조직도와 인력 정보를 과제 관리의 기반 마스터로 관리할 수 있어야 한다.
- 연간 조직 개편에도 대응할 수 있도록 조직명, 조직 구조, 소속 이력을 유연하게 관리할 수 있어야 한다.
- 조직도와 인력 정보는 사내 인사 `DB` 를 기준 원천으로 삼아야 하며, 동시에 과제 운영을 위한 내부 표준 마스터를 별도로 유지할 수 있어야 한다.
- `Crowd` 는 `Jira`, `Confluence`, `Bitbucket`, `Bamboo` 에 한정된 계정 연계 체계로 취급하고, 필요 시 그 외 시스템은 별도의 계정관리 체계와 연계할 수 있어야 한다.
- 조직 또는 프로젝트 단위의 공유 캘린더를 통해 휴가, 출장 등 부재 일정을 공유하고 관리할 수 있어야 한다.
- 외부 `SCM`, `CI/CD`, 정적분석 도구의 실행 결과를 프로젝트 문맥 기준으로 연결 조회하고 일부 관리할 수 있는 내부 기능을 제공한다.
- `AI` 기반 자동 코드리뷰 작성 기능을 통해 변경 요약, 잠재 이슈, 확인 포인트를 코드리뷰 흐름 안에서 보조하고, 팀별 코딩룰 준수 여부를 함께 판단한다.
- `PR` 변경 범위 또는 사용자 지정 범위를 기준으로 `AI` 기반 단위테스트 작성과 테스트 수행을 보조한다.
- 코드리뷰 참여율 등 예정 지표 계산 시 `PR` 기간 중 확정 부재 상태인 잠재 리뷰어를 모수에서 자동 제외할 수 있어야 한다.
- 프로젝트별 소스 특성과 표준 템플릿을 바탕으로 `AI` 기반 `CI` 플랜 초안을 생성해 초기 파이프라인 구성을 보조한다.
- `AI` 에이전트가 시스템 운영, 사용, 유지보수, 개발 전반에서 활용될 수 있도록 `MCP` 또는 `CLI` 기반 도구 제공 체계를 마련한다.

## 4. 기대 효과

- 분산된 관리 정보를 단일 화면 또는 단일 API 기준으로 조회 가능
- 프로젝트 상태와 품질 지표에 대한 의사결정 속도 향상
- 수작업 보고서 작성과 데이터 취합 비용 절감
- 변경 이력과 배포 결과의 추적성 강화
- 시스템 간 연계 표준화에 따른 운영 일관성 확보
- 자체 `ALM` 프로토타입 개발을 위한 데이터와 사용자 흐름 기반 확보
- 개발자 중심 `UX` 개선을 통한 업무 전환 비용 절감

## 5. 제품 범위

### 포함 범위

- 외부 시스템 메타데이터 수집
- 과제 등록 및 관리 기준정보 정의
- 조직도 및 인력 기준정보 정의
- 공통 데이터 모델 정의 및 저장
- 프로젝트/시스템/산출물/배포/품질 현황 통합 조회
- 검색, 필터링, 대시보드, 기본 리포트
- 사용자, 역할, 권한 기반 접근 통제
- 조직도 관리와 조직 개편 이력 관리
- 인력 소속, 조직장, 과제 구성원 관리
- 연계 실패 및 동기화 상태 모니터링
- 테스트 관리 정보, 정적분석 결과, 단위테스트 리포트, `KPI` 지표 수집 및 연결
- 일반 과제 등록 프로세스와 관리 외 프로젝트 등록 프로세스 제공
- 관리 외 프로젝트의 관리 대상 전환 절차와 상태 관리
- 코드리뷰 상태, `AI` 자동 리뷰 초안, 코딩룰 준수 판단 결과, `AI` 기반 단위테스트 초안 및 실행 결과, `AI` 기반 `CI` 플랜 초안, 빌드 결과, 산출물 메타데이터, 정적분석 결과를 내부 화면에서 조회/관리하는 기능
- 연계 대상 시스템과 데이터 범위를 관리자가 확장할 수 있는 설정 구조
- Phase 2 자체 `ALM` 프로토타입을 위한 화면/기능 흐름 설계
- `AI` 에이전트 활용을 위한 `MCP`/`CLI` 도구 전략 수립

### 제외 또는 추후 검토 범위

- Phase 1에서 외부 시스템의 핵심 업무 기능 자체를 즉시 대체하는 것
- 모든 외부 시스템의 전 기능 복제
- 실시간 제어 자동화 전면 도입
- 조직 전체 프로세스 표준화의 즉시 완결

조직 및 인력 기준정보의 원천은 사내 인사 `DB` 가 되어야 한다. 변경이 발생하면 기본 흐름은 `인사 DB -> 시스템 내부 데이터 -> Crowd 또는 별도 계정관리 시스템` 순서여야 하며, 시스템은 중간 단계에서 변경 내용을 검토하고 프로젝트/과제/계정 영향 범위를 확인할 수 있어야 한다. 또한 관리자가 승인한 경우 관련 조직 정보와 연결된 프로젝트 조직 정보, 계정 연계 정보를 마이그레이션할 수 있는 기능이 필요하다. 다만 과제 운영, 승인 흐름, 역할 부여, 예외 처리까지 외부 시스템에 전적으로 의존하면 변경 대응성이 낮아질 수 있으므로, 내부에서는 과제 중심 운영에 맞게 정규화된 조직/인력 마스터를 따로 유지하고 원천 식별자와 내부 식별자를 함께 관리하는 방향이 바람직하다.

## 6. 주요 이해관계자

- 경영진/PMO: 프로젝트 현황과 성과를 종합 확인
- 개발팀: 변경, 빌드, 배포, 품질 상태 추적
- 운영팀: 배포 이력, 장애 영향 범위, 운영 기준 관리
- 품질/감사 조직: 추적성, 승인 이력, 보고 자료 확보
- 시스템 관리자: 연계 설정, 계정/권한, 데이터 품질 관리
- `AI` 에이전트 및 자동화 도구 사용자: 운영, 분석, 유지보수, 개발 지원 작업 수행

## 7. 개념적 대상 시스템

현재 파악된 대표 시스템은 다음과 같다.

- `Jira`
- `Bitbucket`
- `Bamboo`
- `Confluence`
- 테스트 관리 도구
- `KPI` 대시보드
- 코딩 룰 검사 및 리뷰 보조 도구
- 언어별 단위테스트 리포트 수집 체계

우선순위 기준은 다음과 같다.

- 1차 핵심 연계: `Jira`, `Bitbucket`, `Bamboo`
- 2차 핵심 연계: `Confluence`, 테스트 관리 도구, `KPI`/지표 관리 계열
- 후속 검토: 정적분석 및 기타 보조 도구

다만 연계 대상 시스템과 수집 대상 데이터는 고정하지 않는다. 통합 시스템 관리자가 연계 시스템 종류와 관리 대상을 추가, 수정, 비활성화할 수 있어야 한다.

필요 시 향후 보안 스캔, `ITSM`, `CMDB` 계열 시스템도 포함할 수 있다.

## 8. 단계별 제품 방향

### Phase 1. 통합 관리 기반 구축

- 현재 운영 중인 시스템 정보를 수집하고 표준 모델로 정규화한다.
- 과제를 중심으로 한 등록/관리 구조와 관리 외 프로젝트 등록 구조를 마련한다.
- `SCM`, `CI/CD`, 정적분석은 외부 전문 도구를 연계하고, 프로젝트, 요구사항, 문서, 형상, 빌드, 배포, 정적분석, 테스트, 리뷰 지표를 연결 조회한다.
- 조직도와 인력 관리 기준을 먼저 정립해 과제와 승인 흐름의 기준 마스터로 삼는다.
- 관리자 중심의 연계 설정과 데이터 범위 관리 기능을 제공한다.
- `KPI` 및 언어별 단위테스트 리포트를 수집해 지표화한다.
- 코드리뷰, `AI` 자동 리뷰 초안, `AI` 기반 `CI` 플랜 초안, 빌드/산출물 메타데이터, 정적분석 결과를 통합 시스템 내부 화면에서 조회하고 일부 관리할 수 있는 기능을 우선 제공한다.

### Phase 2. 자체 ALM 프로토타입

- Phase 1의 데이터와 사용자 시나리오를 바탕으로 자체 `ALM` 프로토타입을 구축한다.
- 현행 `Jira`, `Bitbucket`, `Bamboo`, `Confluence` 등의 장점을 참고하되 개발자 편의성을 최우선으로 설계한다.
- 통합 조회를 넘어 일부 핵심 업무 흐름을 시스템 내부에서 직접 처리할 수 있는 기반을 마련한다.

## 9. 제품 설계 원칙

- 개발자 편의성 우선: 기존 도구의 익숙한 흐름을 참고하되 더 적은 클릭과 더 빠른 탐색을 지향한다.
- 관리자 확장성: 연계 대상과 관리 대상 데이터를 코드 수정 없이 확장 가능해야 한다.
- 추적성 우선: 요구사항, 변경, 빌드, 배포, 품질, 테스트 결과가 자연스럽게 연결되어야 한다.
- 실행 계층 분리: `SCM`, `CI/CD`, 정적분석 실행 엔진은 외부 도구를 우선 활용하고, 내부 시스템은 연결, 조회, 관리 경험을 강화해야 한다.
- `AI` 친화성: 운영, 검색, 분석, 유지보수, 개발 작업을 지원하는 `MCP`/`CLI` 도구를 함께 제공해야 한다.
- 리뷰 보조 자동화: `AI` 자동 코드리뷰는 사람 리뷰를 보조하고 리뷰 우선순위 식별을 돕는 방향으로 제공해야 한다.
- 규칙 기반 판단: `AI` 자동 코드리뷰는 팀별 코딩룰을 기준으로 준수 여부를 판단하고, 판단 근거가 된 규칙을 함께 보여줘야 한다.
- 테스트 보조 자동화: `AI` 단위테스트 기능은 변경 범위 또는 사용자 지정 범위를 기준으로 테스트 작성과 실행을 보조하되, 최종 채택 여부는 사용자가 검토할 수 있어야 한다.
- 설정 보조 자동화: `AI` 기반 `CI` 플랜 초안은 템플릿 기반 초기 구성을 보조하고, 최종 실행 전 사람이 검토하는 방향으로 제공해야 한다.

## 10. 핵심 성공 기준

- 최소한의 수작업으로 주요 운영 데이터를 수집할 수 있어야 한다.
- 외부 시스템 간 데이터 연결 규칙이 명확해야 한다.
- 과제 등록부터 변경, 빌드, 테스트, 문서, 품질 연결까지 일관된 흐름이 제공되어야 한다.
- 사용자별 필요한 정보를 빠르게 찾을 수 있어야 한다.
- 조직 개편 이후에도 과거 과제와 현재 조직 구조를 함께 추적할 수 있어야 한다.
- 동기화 실패와 데이터 불일치를 식별할 수 있어야 한다.
- 보고와 감사 대응에 활용 가능한 수준의 이력성과 신뢰성을 제공해야 한다.
- 관리자 스스로 연계 대상을 조정할 수 있어야 한다.
- 개발자가 기존 환경보다 더 낮은 탐색 비용으로 필요한 정보를 찾을 수 있어야 한다.
- `AI` 에이전트가 실질적으로 활용 가능한 운영/개발 도구가 제공되어야 한다.

## 11. 가정 및 미정 사항

- 현재 대상 조직, 사용자 수, 보안 정책은 추가 확정이 필요하다.
- 실제 테스트 관리 도구, 코드 리뷰 보조 도구, 단위테스트 수집 방식의 구체 제품과 인터페이스는 추가 분석이 필요하다.
- 코드리뷰와 산출물 관리 기능을 내부에서 어느 수준까지 직접 관리할지는 추가 결정이 필요하다.
- `AI` 자동 리뷰 결과의 저장 범위, 승인 흐름 반영 방식, 오탐 처리 기준은 추가 정의가 필요하다.
- 팀별 코딩룰의 관리 주체, 버전 관리 방식, 예외 허용 범위는 추가 정의가 필요하다.
- `AI` 기반 단위테스트 생성 범위, 테스트 실행 권한, 실패 시 재시도/수정 흐름은 추가 정의가 필요하다.
- `AI` 기반 `CI` 플랜 초안의 템플릿 구조, 자동 결정 규칙, 최종 확정 워크플로우는 추가 정의가 필요하다.
- `Jenkins` 와 `Bamboo Specs` 를 함께 지원할 때 공통 템플릿과 도구별 출력 형식을 어떻게 분리할지는 추가 정의가 필요하다.
- 실시간 연계가 필요한 데이터와 배치 기반으로 충분한 데이터의 구분은 추후 결정이 필요하다.
- 온프레미스, 클라우드, 하이브리드 중 어떤 배포 모델을 택할지는 미정이다.
- Phase 2에서 어떤 기능부터 자체 `ALM`으로 전환할지는 현행 사용 패턴 분석 후 확정한다.

## 12. 문서 연계 기준

- 본 문서는 프로젝트의 배경과 방향을 설명하는 상위 개요 문서다.
- 개념 수준의 요구와 범위 정의는 `CRS` 문서에서 관리한다.
- 세부 기능, 비기능, 연계, 운영 요구는 `SRS` 문서에서 관리한다.
- 개요 문서의 방향이 바뀌면 관련 `CRS`와 `SRS`를 함께 검토한다.
