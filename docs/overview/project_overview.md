# 프로젝트 개요

- 문서 목적: 통합 중앙 관리 시스템의 배경, 목표, 범위, 기대 효과를 정리한다.
- 범위: 프로젝트 개념 정의와 상위 수준의 제품 방향
- 대상 독자: 프로젝트 스폰서, 기획자, 개발자, 운영자, 아키텍트
- 상태: draft
- 최종 수정일: 2026-04-04
- 관련 문서: `docs/requirements/system_crs.md`, `docs/requirements/system_srs.md`

## 1. 배경

조직 내에서는 ALM, 형상관리 시스템, 문서관리 시스템, CI/CD, 정적 분석 및 품질 분석 도구 등 여러 시스템이 각각 독립적으로 운영되는 경우가 많다. 이로 인해 프로젝트 현황, 산출물 상태, 변경 이력, 품질 지표, 배포 결과, 승인 이력 등이 여러 도구에 분산되고, 동일한 관리 정보를 서로 다른 화면과 기준으로 반복 확인해야 하는 문제가 발생한다.

이 프로젝트는 이러한 분산 정보를 한 곳에 모아 중앙에서 관리할 수 있는 통합 관리 시스템을 구축하는 것을 목표로 한다.

## 2. 문제 정의

현재와 같은 분산 운영 환경에서는 다음 문제가 발생할 가능성이 높다.

- 프로젝트 상태를 종합적으로 파악하기 어렵다.
- 시스템별 데이터 기준과 식별자가 달라 추적성이 약해진다.
- 변경, 승인, 배포, 품질 상태를 연결해서 보기 어렵다.
- 운영 보고서와 감사 대응을 위해 수작업 데이터 수집이 발생한다.
- 권한 관리와 표준 프로세스를 일관되게 적용하기 어렵다.

## 3. 프로젝트 목표

본 시스템의 1차 목표는 여러 운영 시스템의 핵심 정보를 수집, 정규화, 연결, 조회할 수 있는 중앙 관리 플랫폼을 제공하는 것이다.

세부 목표:

- 프로젝트, 애플리케이션, 산출물, 변경, 배포, 품질 정보를 통합 조회한다.
- 외부 시스템별로 흩어진 식별자와 메타데이터를 표준 모델로 정규화한다.
- 변경에서 빌드, 배포, 품질, 문서까지 추적 가능한 연결 구조를 제공한다.
- 대시보드와 검색을 통해 운영 가시성을 높인다.
- 감사, 보고, 운영 점검에 필요한 기준 정보를 일관되게 제공한다.

## 4. 기대 효과

- 분산된 관리 정보를 단일 화면 또는 단일 API 기준으로 조회 가능
- 프로젝트 상태와 품질 지표에 대한 의사결정 속도 향상
- 수작업 보고서 작성과 데이터 취합 비용 절감
- 변경 이력과 배포 결과의 추적성 강화
- 시스템 간 연계 표준화에 따른 운영 일관성 확보

## 5. 제품 범위

### 포함 범위

- 외부 시스템 메타데이터 수집
- 공통 데이터 모델 정의 및 저장
- 프로젝트/시스템/산출물/배포/품질 현황 통합 조회
- 검색, 필터링, 대시보드, 기본 리포트
- 사용자, 역할, 권한 기반 접근 통제
- 연계 실패 및 동기화 상태 모니터링

### 제외 또는 추후 검토 범위

- 외부 시스템의 핵심 업무 기능 자체를 대체하는 것
- 모든 외부 시스템의 전 기능 복제
- 실시간 제어 자동화 전면 도입
- 조직 전체 프로세스 표준화의 즉시 완결

## 6. 주요 이해관계자

- 경영진/PMO: 프로젝트 현황과 성과를 종합 확인
- 개발팀: 변경, 빌드, 배포, 품질 상태 추적
- 운영팀: 배포 이력, 장애 영향 범위, 운영 기준 관리
- 품질/감사 조직: 추적성, 승인 이력, 보고 자료 확보
- 시스템 관리자: 연계 설정, 계정/권한, 데이터 품질 관리

## 7. 개념적 대상 시스템

초기 가정 기준으로 다음 유형의 시스템이 연계 대상이 될 수 있다.

- ALM 또는 업무관리 도구
- 형상관리 시스템
- 문서관리 시스템
- CI/CD 파이프라인 및 배포 시스템
- 정적 분석, 테스트, 코드 품질 도구
- 필요 시 보안 스캔, ITSM, 형상DB(CMDB) 계열 시스템

구체적인 제품군과 인터페이스는 추후 확정이 필요하다.

## 8. 핵심 성공 기준

- 최소한의 수작업으로 주요 운영 데이터를 수집할 수 있어야 한다.
- 외부 시스템 간 데이터 연결 규칙이 명확해야 한다.
- 사용자별 필요한 정보를 빠르게 찾을 수 있어야 한다.
- 동기화 실패와 데이터 불일치를 식별할 수 있어야 한다.
- 보고와 감사 대응에 활용 가능한 수준의 이력성과 신뢰성을 제공해야 한다.

## 9. 가정 및 미정 사항

- 현재 대상 조직, 사용자 수, 연계 대상 제품, 보안 정책은 확정되지 않았다.
- 실시간 연계가 필요한지, 배치 기반으로 충분한지는 추후 결정이 필요하다.
- 온프레미스, 클라우드, 하이브리드 중 어떤 배포 모델을 택할지는 미정이다.
- 요구사항 우선순위는 인터뷰와 현행 시스템 분석 후 조정될 수 있다.

## 10. 문서 연계 기준

- 본 문서는 프로젝트의 배경과 방향을 설명하는 상위 개요 문서다.
- 개념 수준의 요구와 범위 정의는 `CRS` 문서에서 관리한다.
- 세부 기능, 비기능, 연계, 운영 요구는 `SRS` 문서에서 관리한다.
- 개요 문서의 방향이 바뀌면 관련 `CRS`와 `SRS`를 함께 검토한다.
