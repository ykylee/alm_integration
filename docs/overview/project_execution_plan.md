## 1. 계획서 개요

본 문서는 현재까지 정리된 기안서, `CRS`, `SRS`를 기준으로 프로젝트를 실제 착수 가능한 수준으로 준비하기 위한 실행 계획 초안이다. 일정과 인력은 아직 확정되지 않았으므로, 단계와 산출물 중심의 계획 구조를 우선 제시한다.

## 2. 추진 목표

- 과제 등록과 관리 기능을 중심으로 외부 연계 흐름을 구성한다.
- 일반 과제 등록 프로세스와 관리 외 프로젝트 등록 프로세스를 분리한다.
- 관리 외 프로젝트를 진행 중 관리 대상 프로젝트로 전환할 수 있는 운영 경로를 마련한다.
- 조직도와 인력 관리 기능을 과제 관리의 기반 마스터로 확립한다.
- 우선 연계 대상 시스템의 운영 정보를 통합 조회 가능한 수준으로 연결한다.
- 표준 데이터 모델과 식별자 매핑 체계를 확립한다.
- 역할 기반 대시보드, 검색, 추적성 조회 기능의 `MVP`를 정의한다.
- 운영 관리 기능과 연계 실패 대응 체계를 마련한다.
- `AI` 보조 기능의 도입 후보와 적용 조건을 구체화한다.

## 3. 추진 범위 기준

### 우선 범위

- 1차 핵심 연계: `Jira`, `Bitbucket`, `Bamboo`
- 2차 핵심 연계: `Confluence`, 테스트 관리 도구, `KPI`/지표 관리 계열
- 통합 조회 대상: 요구사항, 문서, 형상, 빌드, 배포, 테스트, 품질
- 운영 관리 대상: 연계 상태, 동기화 이력, 권한, 예외 관리
- `AI` 검토 대상: 코드리뷰 초안, 코딩룰 점검, 단위테스트 보조, `CI` 플랜 초안

### 후속 확장 범위

- 정적분석 계열과 기타 `SDLC` 보조 도구 연계 확대
- 자체 `ALM` 프로토타입 기능 설계
- `MCP`/`CLI` 운영 도구 구체화

## 5.1 인력 운영 원칙

- 인력 투입은 실제 진행 상황, 우선 연계 대상 수, 병행 과제 상황에 따라 조정될 수 있다.
- 따라서 본 계획서는 고정 인원 수보다 역할별 필요 역량과 단계별 투입 집중도를 기준으로 관리한다.
- 구현 단계에서 연계 대상 수가 증가하면 개발 및 검증 리소스가 추가로 필요할 수 있다.
- 상세 투입 계획은 단계 착수 전 재산정하고, 주요 마일스톤 종료 시점마다 조정 여부를 다시 판단한다.

## 5.2 목표 시점 기준 인력 검토

`Phase 1`을 2026년 8월까지, `Phase 2` 프로토타입을 2026년 10월까지 완료하려면 현재 초안보다 압축된 추진이 필요하다. 이 일정은 가능하긴 하지만, 다음 전제가 충족되어야 한다.

- `Phase 1` 범위를 엄격히 `MVP`로 제한해야 한다.
- 분석, 표준 모델 정의, 연계 설계, 화면 설계를 일부 병렬로 진행해야 한다.
- 우선 연계 대상은 반드시 순차 우선순위를 정해 1차/2차로 나눠야 한다.
- `AI` 보조 기능은 `Phase 1`의 필수 완료 조건이 아니라 선택 또는 제한적 파일럿으로 관리해야 한다.

### 최소 인력 구성

| 역할 | 권장 수준 | 비고 |
| --- | --- | --- |
| PM/기획 | `1` | 범위 통제와 의사결정 조율 전담 필요 |
| 아키텍트/기술 리드 | `1` | 표준 모델, 연계 구조, 기술 기준 총괄 |
| 백엔드/플랫폼 개발 | `2~3` | 공통 플랫폼, 연계, 조회 기능 분담 |
| 프론트엔드 | `1` | 대시보드, 검색, 추적성 화면 전담 |
| 운영/인프라/보안 지원 | `0.5~1` | 전담이 아니어도 주기적 투입 필요 |
| QA 또는 검증 지원 | `0.5~1` | 시범 운영 전후 집중 투입 가능 |

해석:

- 최소 구성은 `약 5.0~8.0 FTE` 수준의 범위형 추정으로 보는 편이 적절하다.
- 다만 이는 역할 겸임이 가능하고 우선 연계 대상을 강하게 제한했을 때 기준이다.

### 권장 인력 구성

| 역할 | 권장 수준 | 비고 |
| --- | --- | --- |
| PM/기획 | `1` | 이해관계자 조율과 일정 관리 |
| 아키텍트/기술 리드 | `1` | 구조 의사결정과 품질 기준 유지 |
| 백엔드/플랫폼 개발 | `3~4` | 공통 기능과 연계 모듈 병렬 개발 |
| 프론트엔드 | `1~2` | 사용자 화면과 운영 화면 병행 |
| 운영/인프라/보안 | `1` | 인증, 권한, 배포, 감사 대응 |
| QA/검증 | `1` | 연계 검증과 시범 운영 검증 |

해석:

- 권장 구성은 `약 7.0~10.0 FTE` 수준의 범위형 추정이다.
- 이 수준이면 `Phase 1`과 `Phase 2` 준비 일부를 8월 이전부터 겹쳐서 진행할 여지가 생긴다.

### 일정 준수 관점의 판단

- 최소 인력 구성으로도 목표 달성은 가능하지만, 1차 핵심 연계를 `Jira`, `Bitbucket`, `Bamboo` 중심으로 제한하고 2차는 `Confluence`, 테스트 관리, 지표 관리 수준까지만 다뤄야 한다.
- 권장 인력 구성이면 `Phase 1`을 8월까지 완료한 뒤 9~10월 동안 `Phase 2` 프로토타입을 구현할 현실성이 높아진다.
- 반대로 1차와 2차 대상을 구분하지 않고 정적분석이나 기타 보조 도구까지 동시에 깊게 다루려 하면 현재 목표 일정은 과도하게 빡빡해진다.

## 8.1 일정 관점 핵심 리스크

- 2026년 8월 `Phase 1` 완료 목표는 분석 종료 후 설계에 들어가는 순차형 방식으로는 위험하다.
- 따라서 2026년 4월부터 6월 사이에는 현행 분석과 표준 모델 설계를 완전 순차로 두지 말고 병렬로 운영해야 한다.
- 2026년 9월부터 10월까지 `Phase 2` 프로토타입을 진행하려면 `Phase 1` 종료 전부터 프로토타입 범위를 미리 선정해 두어야 한다.
- 일정 준수 실패의 가장 큰 원인은 범위 증가이므로, 월별 범위 재승인 절차가 필요하다.

## 9. 후속 보완 필요 사항

- 데이터 보관 및 감사 정책 확정
- `AI` 기능별 운영 정책과 책임 범위 확정

## 10. 문서 활용 방식

- 착수 전에는 범위 합의와 선결 조건 확인 문서로 사용한다.
- 상세 일정 수립 전에는 단계와 산출물 기준의 계획 템플릿으로 사용한다.
- 이후 확정 정보가 생기면 본 문서를 기준으로 세부 WBS와 일정표를 확장한다.
