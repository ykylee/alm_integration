## 1. 한 줄 제안

분산된 `SDLC` 운영 정보를 하나의 표준 모델과 화면으로 통합해 가시성, 추적성, 운영 통제력을 높이고, 장기적으로 자체 `ALM` 프로토타입으로 확장할 수 있는 기반을 확보하는 프로젝트다.

## 2. 왜 지금 필요한가

현재 `Jira`, `Bitbucket`, `Bamboo`, `Confluence`, 테스트 관리 도구, `KPI` 대시보드, `Coverity` 등 여러 시스템에 요구사항, 변경, 빌드, 배포, 품질 정보가 분산되어 있다. 그 결과 다음 문제가 반복된다.

- 프로젝트 상태를 한 번에 보기 어렵다.
- 변경과 배포, 품질 간 추적성이 약하다.
- 보고와 감사 대응을 위해 수작업 데이터 취합이 필요하다.
- 개발자가 여러 도구를 오가며 맥락을 맞춰야 한다.

## 3. 무엇을 하려는가

### Phase 1

- 일반 과제는 표준 과제 등록 프로세스를 통해 등록한다.
- 관리 외 프로젝트는 간소한 별도 등록 프로세스로 최소 정보만 등록한다.
- 필요 시 관리 외 프로젝트를 관리 대상 프로젝트로 전환할 수 있는 루트를 제공한다.
- 외부 시스템 메타데이터를 수집, 정규화, 연결한다.
- 프로젝트, 요구사항, 문서, 형상, 빌드, 배포, 테스트, 품질 정보를 통합 조회한다.
- 연계 상태, 동기화 실패, 권한, 기본 보고 기능을 제공한다.

### Phase 2

- Phase 1에서 확보한 데이터와 사용 패턴을 기반으로 자체 `ALM` 프로토타입을 검토한다.
- 개발자 중심 `UX`와 내부 관리 기능을 강화한다.

## 4. 핵심 추진 원칙

- 과제 중심: 등록된 과제가 연계와 추적의 중심 단위가 되어야 한다.
- 외부 전문 도구의 실행 계층은 유지하고 통합과 연결을 우선한다.
- 관리자가 연계 대상과 수집 범위를 확장할 수 있어야 한다.
- 개발자 편의성을 우선한 탐색 구조를 지향한다.
- `AI`는 코드리뷰, 코딩룰 점검, 단위테스트, `CI` 플랜 초안 작성 등 보조 역할로 도입한다.

## 5. 기대 효과

- 분산 정보의 단일 조회 체계 확보
- 요구사항부터 변경, 빌드, 배포, 품질까지 추적성 강화
- 운영 보고 및 감사 대응 비용 절감
- 데이터 표준화 기반 운영 통제력 향상
- 향후 자체 `ALM` 전환 가능성 검증

## 6. 이번 단계에서 필요한 의사결정

- Phase 1 1차 핵심 연계 대상 `Jira`, `Bitbucket`, `Bamboo` 확정
- Phase 1 2차 연계 대상 `Confluence`, 테스트 관리, 지표 관리 범위 확정
- 시범 적용 조직과 프로젝트 범위 확정
- 목표 배포 모델 확정
- 인증 및 권한 모델 기준 확정
- `AI` 보조 기능의 우선 도입 범위 확정

## 7. 현재 기준의 보완 필요 항목

- 정량 효과 목표
- 예산 및 인력 계획
- 상세 일정과 마일스톤
- 운영 및 보안 정책
- `AI` 적용 및 승인 기준

## 7.1 일정 검토 의견

- `Phase 1` 개발 목표를 2026년 8월, `Phase 2` 프로토타입 목표를 2026년 10월로 두는 것은 가능하다.
- 다만 이 경우 `Phase 1`은 `Jira`, `Bitbucket`, `Bamboo` 중심의 `MVP` 범위로 통제해야 하며, `Confluence`, 테스트 관리, 지표 관리는 2차 범위로 두는 편이 안전하다.
- 권장 인력 수준은 대략 `7.0~10.0 FTE` 범위이며, 최소 구성으로도 가능하지만 범위 통제 강도가 훨씬 높아진다.

## 8. 보고 시 권장 메시지

- 이 프로젝트는 도구 교체가 아니라 정보 통합과 운영 표준화가 1차 목적이다.
- Phase 1은 빠른 통합 효과를 내는 `MVP` 중심으로 추진하고, Phase 2는 검증 결과를 바탕으로 판단한다.
- 투자 판단에 앞서 우선 연계 대상, 시범 범위, 보안 기준만 확정되면 상세 계획 수립이 가능하다.
