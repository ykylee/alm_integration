## 발표 구성 원칙

- 첫 3장 안에 `왜 필요한가`, `무엇을 할 것인가`, `무엇을 결정해야 하는가`가 보여야 한다.
- 발표 본문은 10장 내외를 권장한다.
- 세부 요구사항보다 사업 필요성과 투자 판단 근거를 먼저 배치한다.
- 인력은 확정 배치가 아니라 단계별 조정 가능 전제라는 점을 명시한다.

## 권장 목차

### 1. 프로젝트 제안 배경

- 현재 운영 환경과 분산 문제
- 보고, 추적, 운영 통제의 한계

### 2. 해결하려는 문제

- 왜 기존 구조만으로는 한계가 있는지
- 현업과 개발 측의 비효율 요약

### 3. 제안 개요

- 통합 중앙 관리 플랫폼 개념
- Phase 1과 Phase 2 방향

### 4. 추진 범위

- 포함 범위
- 제외 범위
- 우선 연계 대상 시스템

### 5. 기대 효과

- 가시성
- 추적성
- 운영 효율
- 향후 자체 `ALM` 가능성

### 6. 추진 전략

- 통합 우선
- 단계적 내재화
- 개발자 중심 `UX`
- `AI` 보조 기능 방향

### 7. 실행 계획 요약

- 주요 단계
- 마일스톤
- 시범 운영 방식

### 8. 예산 및 인력 접근 방식

- 고정 인원 배치가 아닌 단계별 가변 투입 원칙
- 예산 항목 구조
- 상세 예산은 범위 확정 후 재산정 예정

### 9. 리스크 및 대응

- 연계 복잡도
- 데이터 품질
- 보안 정책
- `AI` 결과 신뢰성

### 10. 의사결정 요청 사항

- 우선 연계 대상
- 시범 범위
- 배포 모델
- 보안/권한 기준
- 다음 단계 착수 승인

## 발표 시 유의사항

- `도구 대체`보다 `정보 통합과 운영 표준화`를 먼저 강조한다.
- Phase 2는 확정 계획이 아니라 Phase 1 검증 이후 판단할 확장 방향으로 설명한다.
- 인력은 실제 진행에 따라 변경될 수 있으므로, 역할별 필요 역량과 투입 원칙 중심으로 설명한다.
