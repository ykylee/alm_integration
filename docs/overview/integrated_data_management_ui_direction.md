## 1. 방향 전환 배경

현재 정적 프로토타입은 역할 기반 화면 구조를 빠르게 설명하는 데는 유효했지만, 실제 운영 기준에서 “데이터 관리 작업면”으로 쓰기에는 맞지 않는다. 특히 시스템 관리자가 다뤄야 하는 데이터는 일반 운영 화면과 성격이 다르고, 관리 대상도 더 명시적으로 분리되어야 한다.

따라서 다음 방향으로 재정의한다.

- 일반 업무용 화면과 시스템 관리자용 데이터 관리 UI 를 완전히 분리한다.
- 통합 데이터 관리 UI 는 시스템 관리자만 접속 가능한 별도 `URL` 공간으로 둔다.
- 이 UI 의 핵심 목적은 “운영 지표 보기”가 아니라 “관리 대상 데이터를 종류별로 조회, 보정, 검토, 반영”하는 것이다.

## 2. 기본 원칙

### 2.1 시스템 관리자 전용

- 인증 후 `system_admin` 권한이 있는 사용자만 접근 가능해야 한다.
- 일반 사용자, 프로젝트리더, 조직장, `SE`, 테스트 담당자는 이 UI 를 볼 수 없어야 한다.
- 읽기와 변경 권한도 다시 분리할 수 있어야 하지만, 1차 기준은 시스템 관리자 전용 단일 작업면으로 본다.

### 2.2 관리 대상 중심 구조

탐색의 기준을 “역할”이나 “업무 단계”가 아니라 “관리 대상 데이터 종류”로 둔다. 사용자는 어느 데이터 묶음을 관리하려는지 먼저 선택하고, 그 안에서 목록, 상세, 보정, 반영 작업을 수행해야 한다.

### 2.3 외부 시스템 단위 격리

외부 연결 시스템 데이터는 하나의 화면에 합쳐 보여주기보다, 시스템별로 작업 문맥을 분리해야 한다.

- `Jira`
- `Bitbucket`
- `Bamboo`
- `Confluence`
- 향후 추가 시스템

같은 “외부 시스템 데이터”라도 원천 구조, 오류 패턴, 재처리 기준이 다르므로 별도 관리 화면이 필요하다.

### 2.4 조직과 인력은 독립 관리 축

조직과 인력은 단순 기준정보가 아니라 다른 모든 도메인 데이터의 기준 축이다. 따라서 외부 시스템 데이터와 같은 레벨에서 별도 관리 영역으로 둔다.

- 조직 관리는 조직 구조와 코드, 상태, 상하위 관계, 반영 이력을 관리한다.
- 인력 관리는 조직 기준으로 소속된 인력을 목록/상세/보정 형태로 관리한다.

## 3. 1차 정보구조

권장 최상위 구조:

- 자체 `ALM` 데이터
- 외부 연결 시스템 데이터
- 조직 관리
- 인력 관리

현재 프로토타입 기준으로는 별도 `센터 홈` 또는 `대시보드` 를 두지 않고, 시스템 관리자가 관리 대상 화면으로 직접 진입하는 구성을 우선한다. 세부 관리는 각 관리 대상 화면에서 바로 수행한다.

### 3.1 자체 `ALM` 데이터

이 영역은 내부 표준 모델 기준 데이터를 직접 관리하는 화면이다.

관리 대상 예시:

- `project`
- `work_item`
- `work_item_status_history`
- `work_item_hierarchy`
- `iteration`, 향후 `release`, `milestone`
- 내부 표준 참조/매핑 결과

핵심 목적:

- 내부에 적재된 표준 데이터의 현재 상태를 직접 확인한다.
- 원천과 무관하게 현재 내부 데이터가 어떤 모습인지 관리한다.
- 필요 시 수동 보정 또는 운영 조치의 기준 화면으로 사용한다.

### 3.2 외부 연결 시스템 데이터

이 영역은 외부 원천 시스템 데이터를 시스템별로 구분해 관리하는 화면이다.

권장 구조:

- 외부 연결 시스템 데이터
  - `Jira`
  - `Bitbucket`
  - `Bamboo`
  - `Confluence`

각 시스템별 하위 화면에서 다뤄야 하는 것:

- 연결 상태
- 최근 수집/동기화 실행 이력
- 원시 적재 데이터
- 표준화 결과 요약
- 식별자 매핑 상태
- 시스템별 오류와 보정 필요 항목

이렇게 해야 “외부 시스템 항목 별로 별도 관리” 요구를 만족한다.

### 3.3 조직 관리

조직 관리는 조직 마스터 전용 작업면으로 둔다.

관리 대상:

- 조직 코드
- 조직명
- 상위 조직
- 활성/비활성 상태
- 유효기간
- 변경 이력

핵심 화면:

- 조직 목록
- 조직 상세
- 조직 구조 트리/계층 보기
- 변경 이력 및 영향 보기
- 등록/수정/삭제 액션
- 조직 구성원 관리 진입점

프로토타입 검증 단계에서는 `사업부 -> 팀 -> 그룹 -> 파트` 구조의 더미 데이터 생성 기능을 함께 두어, 다단계 계층과 조직별 구성원 관리 흐름을 빠르게 확인할 수 있게 한다.

조직 관리 화면의 시각 구조는 단순 트리 데모보다 `디렉터리 + 선택 상세 인스펙터 + 액션 레일 + 감사 이력` 구성이 더 적합하다. 이는 실제 엔터프라이즈 관리자 콘솔에서 공통적으로 보이는 운영형 구조이며, 레퍼런스 조사 내용은 [organization_admin_ui_reference_research.md](./organization_admin_ui_reference_research.md) 에 정리한다.

### 3.4 인력 관리

인력 관리는 반드시 “조직별 리스트” 중심으로 설계한다.

권장 구조:

- 조직 선택
- 선택 조직 소속 인력 목록
- 인력 상세
- 소속 변경/상태 변경 이력

핵심 필터:

- 조직 코드
- 재직 상태
- 사번
- 이름
- 직군

현재 1차 구현 기준으로 인력 관리는 조직별 리스트 화면에서 다음 액션을 지원하는 방향을 따른다.

- 조직 구성원 등록
- 다른 조직으로 이동
- 비활성화 처리

## 4. 권장 화면 구조

### 4.1 레이아웃 원칙

- 좌측: 관리 대상 네비게이션
- 상단: 현재 관리 영역, 필터, 최근 동기화 상태
- 본문 좌측: 목록 또는 트리
- 본문 우측: 상세/비교/변경 이력
- 하단 또는 보조 영역: 운영 액션 체크리스트와 영향 범위 요약

이 구조는 시스템 관리자에게 가장 중요한 “목록에서 하나를 선택하고, 상세/이력/조치를 확인하는 흐름”에 맞는다.

### 4.6 운영 판단 패널

각 관리 화면에는 단순 목록 외에 아래 패널을 함께 둔다.

- 대표 상세: 현재 목록에서 대표 레코드 하나를 기준으로 상태를 빠르게 파악
- 운영 액션: 다음에 수행할 점검, 보정, 재실행, 영향 확인 절차를 체크리스트로 제시
- 영향 범위: 조직, 인력, 도메인 참조, 후속 파이프라인에 미치는 범위를 요약

이 패널은 실제 편집 기능이 아직 없더라도 운영자가 “지금 무엇을 봐야 하고 다음에 무엇을 해야 하는지”를 빠르게 판단하게 해준다.

### 4.2 자체 `ALM` 데이터 화면

권장 서브플로우:

- `/admin/data/alm/projects`
- `/admin/data/alm/work-items`
- `/admin/data/alm/work-items/:id`
- `/admin/data/alm/plans`

### 4.3 외부 연결 시스템 데이터 화면

권장 서브플로우:

- `/admin/data/external`
- `/admin/data/external/jira`
- `/admin/data/external/bitbucket`
- `/admin/data/external/bamboo`
- `/admin/data/external/confluence`

### 4.4 조직 관리 화면

권장 서브플로우:

- `/admin/data/organizations`
- `/admin/data/organizations/:organization_code`

### 4.5 인력 관리 화면

권장 서브플로우:

- `/admin/data/workforce`
- `/admin/data/workforce?organization_code=...`
- `/admin/data/workforce/:employee_number`

## 5. 1차 사용자 시나리오

### 5.1 시스템 관리자가 내부 `ALM` 데이터 상태를 점검한다

1. `자체 ALM 데이터` 메뉴로 이동한다.
2. `project` 또는 `work_item` 목록을 연다.
3. 상태/조직/담당자 기준으로 필터링한다.
4. 상세 화면에서 내부 데이터와 참조된 조직/인력을 확인한다.

### 5.2 시스템 관리자가 외부 `Jira` 연계 데이터를 점검한다

1. `외부 연결 시스템 데이터 > Jira` 로 이동한다.
2. 최근 `sync-runs` 와 원시 적재 목록을 확인한다.
3. 보류 또는 실패 항목을 선택해 상세 원인을 확인한다.
4. 필요 시 표준화/매핑 상태를 따라가며 운영 조치를 결정한다.

### 5.3 시스템 관리자가 조직 마스터를 관리한다

1. `조직 관리` 메뉴로 이동한다.
2. 조직 트리 또는 목록에서 특정 조직을 선택한다.
3. 조직 상세, 상위 조직, 상태, 유효기간을 확인한다.
4. 변경 시 영향 받는 데이터 범위를 함께 본다.

### 5.4 시스템 관리자가 특정 조직 소속 인력을 관리한다

1. `인력 관리` 메뉴로 이동한다.
2. 조직 필터에서 특정 조직을 선택한다.
3. 선택 조직 소속 인력 목록을 확인한다.
4. 인력 상세에서 사번, 재직 상태, 직군, 주 소속, 참조 데이터를 확인한다.

## 6. 현재 프로토타입에 대한 판단

현재 `organization.html`, `admin.html` 은 운영 API 연결 실험에는 유효했지만, 장기적으로 유지할 UI 구조로는 적합하지 않다.

- 역할 기반 설명용 구조와 데이터 관리용 구조가 서로 다르다.
- 시스템 관리자 전용 데이터 관리 작업면이 별도 공간으로 분리되지 않았다.
- 외부 시스템별 관리 문맥이 독립되어 있지 않다.
- 조직/인력 관리가 기준정보 전용 화면으로 충분히 독립되지 않았다.

따라서 이후 UI 작업은 기존 정적 프로토타입을 점진 수정하는 방식보다, 본 문서를 기준으로 정보구조부터 다시 설계하는 것이 맞다.

## 7. 다음 단계 제안

- 본 문서를 기준으로 관리자 전용 데이터 관리 UI 의 `IA` 와 라우트 구조를 확정한다.
- `자체 ALM` 데이터, `외부 연결 시스템 데이터`, `조직 관리`, `인력 관리` 각각의 상세 요구사항을 `SRS` 수준으로 내린다.
- 이후 프론트엔드는 기존 프로토타입을 폐기하거나 참고용으로만 두고, 새 관리 UI 를 별도 앱/영역으로 구현한다.
