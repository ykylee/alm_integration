## 1. 프로젝트 개요

본 과제는 `Jira`, `Bitbucket`, `Bamboo`, `Confluence`, 테스트 관리 도구, `KPI` 대시보드, 코드리뷰 및 코딩룰 보조 도구, 언어별 단위테스트 리포트 수집 체계 등 여러 운영 시스템에 분산된 정보를 하나의 표준 모델과 화면에서 통합 관리하기 위한 프로젝트다. 우리 시스템은 과제 등록과 관리 기능을 중심으로 하며, 등록된 과제가 여러 시스템 정보를 연결하는 핵심 단위가 된다.

초기 단계에서는 이기종 시스템의 메타데이터를 수집, 정규화, 연결, 조회하는 중앙 관리 플랫폼을 구축하고, 이후 축적된 데이터와 사용자 흐름을 기반으로 자체 `ALM` 프로토타입으로 확장하는 것을 목표로 한다.

## 2. 추진 배경 및 필요성

현재 운영 환경에서는 요구사항, 형상, 빌드, 배포, 문서, 정적분석, 테스트, 품질 지표가 각각 다른 시스템에 저장된다. 이 구조는 각 도구의 전문성을 활용할 수 있다는 장점이 있지만, 실제 프로젝트 운영 관점에서는 다음 한계를 만든다.

- 프로젝트 상태를 종합적으로 파악하기 어렵다.
- 요구사항, 변경, 빌드, 배포, 품질 간 추적성이 약하다.
- 보고와 감사 대응을 위해 수작업 데이터 취합이 반복된다.
- 시스템별 식별자와 상태 체계가 달라 표준화와 통제가 어렵다.
- 개발자가 여러 도구를 오가며 맥락을 맞춰야 하므로 생산성이 떨어진다.

따라서 본 과제는 개별 도구를 즉시 대체하는 데 목적을 두기보다, 분산된 운영 정보를 연결하고 표준화해 가시성과 통제력을 높이는 방향으로 추진할 필요가 있다.

## 3. 추진 목적

- 분산된 운영 정보를 하나의 기준으로 통합 조회할 수 있는 체계를 마련한다.
- 과제 중심 등록/관리 구조를 통해 운영 흐름의 시작점을 일원화한다.
- 조직도와 인력 관리를 기반 마스터로 두어 과제 구성원과 승인 체계를 일관되게 관리한다.
- 프로젝트, 변경, 산출물, 배포, 품질, 테스트 간 연결 구조를 확보한다.
- 외부 시스템별 식별자와 메타데이터를 표준 데이터 모델로 정규화한다.
- 보고, 감사, 운영 점검에 필요한 정보의 일관성과 신뢰성을 높인다.
- 향후 자체 `ALM` 프로토타입으로 확장할 수 있는 기반을 확보한다.
- 개발자 중심 `UX`와 `AI` 보조 기능을 통해 운영 및 개발 효율을 높인다.

## 4. 추진 범위

### 포함 범위

- 외부 시스템 메타데이터 수집 및 동기화
- 일반 과제 등록 프로세스 제공
- 관리 외 프로젝트의 간소 등록 프로세스 제공
- 관리 외 프로젝트를 관리 대상 프로젝트로 전환하는 루트 제공
- 조직도 관리와 인력 관리 기능 제공
- 조직 개편 이력과 소속 변경 대응 구조 제공
- 표준 식별 체계와 공통 데이터 모델 관리
- 프로젝트, 요구사항, 문서, 형상, 빌드, 배포, 품질, 테스트 정보의 통합 조회
- 검색, 필터링, 대시보드, 기본 보고 기능
- 연계 상태, 오류, 동기화 결과 관리
- 사용자, 역할, 권한 기반 접근 통제
- 테스트 관리 정보, 단위테스트 리포트, 정적분석 결과, `KPI` 지표 수집
- `AI` 기반 코드리뷰 초안, 코딩룰 준수 판단, 단위테스트 초안/실행 보조, `CI` 플랜 초안 생성 기능 검토
- 향후 자체 `ALM` 프로토타입을 위한 사용자 흐름 및 화면 방향 정의

### 제외 범위 또는 후속 검토 범위

- Phase 1에서 외부 `SCM`, `CI/CD`, 정적분석 도구의 핵심 실행 기능을 즉시 대체하는 것
- 모든 외부 시스템의 전 기능 복제
- 조직 전체 프로세스의 일괄 표준화 완결
- 전사 실시간 제어 자동화의 전면 도입

## 6. 추진 전략

### 전략 1. 통합 우선

초기에는 외부 전문 도구의 실행 계층을 유지하고, 통합 시스템은 과제 중심의 등록, 조회, 연결, 표준화, 일부 관리 경험 강화에 집중한다.

### 전략 2. 단계적 내재화

Phase 1에서 확보한 데이터 모델과 사용 패턴을 기반으로, Phase 2에서 자체 `ALM` 프로토타입을 점진적으로 설계한다.

### 전략 3. 개발자 편의성 우선

기존 도구의 익숙한 흐름은 참고하되, 더 적은 탐색 단계와 더 빠른 맥락 파악을 목표로 설계한다.

### 전략 4. 관리자 확장성 확보

연계 대상 시스템과 수집 데이터 범위를 코드 수정 없이 관리자가 추가, 수정, 비활성화할 수 있는 구조를 우선 고려한다.

### 전략 4-1. 조직 변경 대응성 확보

조직명, 상하위 구조, 조직장, 소속 인력 정보가 연간 개편에 따라 언제든 바뀔 수 있으므로, 식별자 기반 관리와 유효 기간 기반 이력 관리 구조를 우선 고려한다.

### 전략 5. AI 보조 기능 내재화

`AI`는 사람을 대체하기보다 코드리뷰 초안 작성, 코딩룰 점검, 단위테스트 보조, `CI` 플랜 초안 제안 등 판단 보조 역할에 우선 적용한다.

## 7. 단계별 추진 계획

### Phase 1. 통합 중앙 관리 기반 구축

- 일반 과제 등록 프로세스와 관리 외 프로젝트 등록 프로세스를 구분 설계
- 과제를 기준으로 외부 시스템 데이터 연결 구조 정의
- 연계 대상 시스템 식별 및 우선순위 정의
- 표준 데이터 모델과 식별자 매핑 체계 정의
- 통합 조회, 검색, 대시보드, 보고 화면 구성
- 연계 상태 및 동기화 운영 기능 구축
- 테스트/정적분석/품질 지표 수집 체계 정비
- `AI` 보조 기능의 적용 후보 시나리오 구체화

### Phase 2. 자체 ALM 프로토타입 확장

- Phase 1 데이터와 사용자 흐름 기반 핵심 기능 선정
- 내부 코드리뷰/산출물/품질 관리 경험 고도화
- 일부 업무 흐름의 자체 처리 프로토타입 검증
- `MCP`/`CLI` 기반 `AI` 에이전트 활용 도구 구체화

## 8. 기대 효과

- 여러 시스템에 분산된 정보를 단일 기준에서 조회할 수 있다.
- 요구사항부터 변경, 빌드, 배포, 품질까지 추적성이 강화된다.
- 운영 보고와 감사 대응에 필요한 데이터 취합 비용이 줄어든다.
- 연계 기준과 상태 체계가 표준화되어 운영 통제력이 높아진다.
- 개발자가 여러 도구를 오가며 확인하던 작업을 줄일 수 있다.
- 장기적으로 자체 `ALM` 전환 가능성을 검증할 수 있다.

## 9. 추진 체계 초안

### 필요 역할

- 프로젝트 스폰서: 사업 우선순위와 투자 판단
- PM/기획: 범위 관리, 이해관계자 조율, 요구사항 정제
- 아키텍트: 표준 모델, 연계 구조, 확장 전략 설계
- 개발팀: 플랫폼 구현, 연계 모듈, 조회 기능, 운영 도구 구현
- 운영/보안 담당: 인증, 권한, 감사, 배포 정책 검토
- 현업 대표: 보고, 조회, 추적, 감사 관점 요구 검증

### 운영 방식 제안

- 초기에는 우선 연계 대상과 핵심 조회 시나리오를 좁게 선정해 `MVP` 범위로 추진한다.
- 이후 사용자 피드백과 운영 데이터를 바탕으로 단계적으로 대상 시스템과 기능을 확장한다.

## 10. 일정 초안

현재는 상세 일정 산정에 필요한 조직 규모, 우선 연계 대상, 인력 투입 계획이 확정되지 않아 아래와 같이 마일스톤 수준으로만 제안한다.

1. 착수 및 현행 분석
2. 표준 데이터 모델 및 연계 정책 정의
3. 우선 연계 시스템 구현
4. 통합 조회/대시보드/운영 기능 구현
5. 시범 운영 및 보완
6. Phase 2 전환 범위 재평가

상세 일정은 연계 대상 수, API 제약, 보안 검토 범위, 배포 환경 제약을 확인한 뒤 별도 수립이 필요하다.

## 11. 리스크 및 대응 방향

| 리스크 | 설명 | 대응 방향 |
| --- | --- | --- |
| 연계 복잡도 증가 | 시스템별 API, 인증, 데이터 품질 차이로 구현 난이도가 높을 수 있음 | 우선순위 기반 연계, 공통 어댑터 구조, 파일럿 연계 선행 |

| 운영 정책 불확실성 | 권한, 감사, 보관 주기, 보안 정책이 미정일 수 있음 | 초기 단계에서 보안/운영 요구를 별도 확정 |
| AI 결과 신뢰성 | 자동 리뷰나 테스트 초안의 오탐, 누락 가능성 존재 | 사람 검토 전제, 근거 표시, 예외 처리 흐름 설계 |
| 범위 확대 위험 | 통합 플랫폼이 전면 대체 프로젝트로 확대될 수 있음 | Phase 기준 범위 통제, MVP 중심 의사결정 |

## 13. 보완 필요 사항

현재 문서만으로는 방향성과 상위 범위는 설명할 수 있으나, 실제 승인용 계획서로 고도화하려면 아래 정보가 추가로 필요하다.

- 정량 목표: 도입 전후 리드타임, 보고 소요시간, 데이터 취합 시간, 결함/품질 지표 개선 목표
- 대상 범위: 시범 적용 부서, 프로젝트 수, 사용자 수, 연계 대상 시스템 수
- 예산/인력: 참여 인력 규모, 역할별 투입률, 외주/내부 개발 여부, 인프라 예산
- 일정: 분기 또는 월 단위 상세 일정, 마일스톤별 완료 기준
- 운영 정책: 인증/권한 모델, 감사 로그 범위, 데이터 보관 주기, 장애 대응 기준
- 기술 정책: 연계 방식 우선순위 API/DB/File, 배치/이벤트 기준, 표준 모델 관리 책임
- AI 정책: 사용 모델, 승인 절차, 결과 저장 범위, 오탐 처리 원칙, 보안 검토 기준

## 14. 기안서 활용 가이드

- 경영진 보고용으로는 2장, 3장, 8장, 11장, 12장을 중심으로 요약본을 만들면 된다.
- 실무 계획서로 확장하려면 본 문서를 기준으로 `CRS`, `SRS`를 세부 부속 문서로 연결하면 된다.
- 제안요청 또는 내부 투자심의 자료로 발전시킬 때는 예산, 일정, 인력, 정량 효과를 별도 표와 수치로 보강해야 한다.
