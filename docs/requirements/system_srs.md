- 문서 목적: 통합 중앙 관리 시스템의 상세 요구사항을 `SRS` 형태로 정리한다.

- 대상 독자: 기획자, 개발자, 아키텍트, 운영자, 테스트 담당자

## 1. 문서 작성 전제

본 문서는 현재 정의된 `CRS`를 바탕으로 작성한 초기 `SRS` 초안이다. Phase 1 통합 관리 플랫폼과 Phase 2 자체 `ALM` 프로토타입까지를 고려한 상위 요구를 담고 있으며, 실제 상세 요구사항은 현행 시스템 분석, 이해관계자 인터뷰, 보안 및 운영 정책 검토를 통해 구체화되어야 한다.

## 2. CRS 추적 기준

- `CR-001` ~ `CR-003`: 통합 대상 관리, 기준정보 관리, 데이터 수집 및 정규화, 테스트/품질 지표 관리
- `CR-004`, `CR-005`: 통합 조회, 검색, 추적성, 대시보드, 보고
- `CR-006` ~ `CR-008`: 운영 관리, 예외 관리, 사용자 및 권한 관리, 관리자 설정
- `CR-009` ~ `CR-011`: 연계 확장성, 성능, 안정성, 보안, Phase 2 확장 준비
- `CR-012`, `CR-013`: 사용성, `AI` 에이전트 연동, 운영 도구 제공

## 3. 핵심 사용자

- 관리자: 시스템 설정, 연계 관리, 권한 관리 담당
- 프로젝트 관리자: 프로젝트별 상태, 산출물, 일정, 품질 현황 확인
- 개발/운영 담당자: 변경, 빌드, 배포, 장애 관련 정보 조회
- 품질/감사 담당자: 추적성, 승인 이력, 준수 여부 확인
- 경영진/PMO: 요약 대시보드와 보고서 확인
- `AI` 에이전트 활용 담당자: 운영 자동화, 검색, 분석, 유지보수, 개발 지원 수행

## 4. 기능 요구사항

### 4.1 통합 대상 및 기준정보 관리

`FR-001` 시스템은 연계 대상 외부 시스템 목록을 등록, 수정, 비활성화할 수 있어야 한다.

`FR-002` 시스템은 외부 시스템별 접속 정보, 인증 방식, 동기화 주기, 상태 정보를 관리할 수 있어야 한다.

`FR-003` 시스템은 프로젝트, 애플리케이션, 조직, 담당자, 환경 등 공통 기준정보를 관리할 수 있어야 한다.

`FR-004` 시스템은 외부 시스템별 고유 식별자와 내부 표준 식별자 간 매핑 정보를 관리할 수 있어야 한다.

`FR-005` 시스템은 관리자가 연계 대상 시스템 유형과 수집 대상 데이터 범위를 코드 수정 없이 추가 또는 수정할 수 있는 설정 구조를 제공해야 한다.

### 4.2 데이터 수집 및 동기화

`FR-006` 시스템은 외부 시스템에서 프로젝트, 이슈/요구사항, 형상 정보, 문서 메타데이터, 빌드/배포 결과, 산출물 메타데이터, 분석 지표를 수집할 수 있어야 한다.

`FR-007` 시스템은 `Jira`, `Confluence`, `Gitea`, `Jenkins`, `Bamboo`, `Coverity` 와 같은 대표 시스템 연계를 우선 지원할 수 있어야 한다.

`FR-008` 시스템은 테스트 관리 도구, `KPI` 대시보드, 코딩 룰 검사/리뷰 보조 도구, 언어별 단위테스트 리포트 수집 체계와 연계할 수 있어야 한다.

`FR-009` 시스템은 배치 동기화를 지원해야 하며, 필요 시 이벤트 기반 또는 수동 동기화도 지원할 수 있어야 한다.

`FR-010` 시스템은 마지막 동기화 시각, 성공/실패 여부, 처리 건수, 오류 메시지를 기록해야 한다.

`FR-011` 시스템은 중복 데이터, 누락 데이터, 매핑 실패를 식별할 수 있어야 한다.

`FR-012` 시스템은 동기화 실패 시 재처리 기능 또는 재시도 정책을 제공해야 한다.

### 4.3 통합 조회 및 검색

`FR-013` 시스템은 프로젝트 단위로 요구사항, 소스 변경, 문서, 빌드, 배포, 품질, 테스트 정보를 통합 조회할 수 있어야 한다.

`FR-014` 시스템은 다중 조건 검색과 필터링을 지원해야 한다.

`FR-015` 시스템은 외부 시스템 출처와 수집 시점이 표시된 상세 조회 화면을 제공해야 한다.

`FR-016` 시스템은 프로젝트, 시스템, 기간, 상태, 담당자 기준으로 요약 현황을 조회할 수 있어야 한다.

`FR-017` 시스템은 주요 엔터티 간 연관관계를 따라 탐색할 수 있어야 한다.

`FR-018` 시스템은 단위테스트 결과, 정적분석 결과, 테스트 관리 현황, 코드 리뷰 품질 신호를 함께 비교 조회할 수 있어야 한다.

`FR-018A` 시스템은 외부 `SCM` 또는 리뷰 연계 정보 기반으로 코드리뷰 대상 변경, 리뷰 상태, 주요 코멘트를 프로젝트 문맥에서 조회할 수 있어야 한다.

`FR-018B` 시스템은 외부 `CI/CD` 결과와 연결된 빌드 산출물 메타데이터를 프로젝트, 버전, 배포 대상 기준으로 조회할 수 있어야 한다.

`FR-018C` 시스템은 외부 정적분석 도구 결과를 심각도, 상태, 컴포넌트, 프로젝트 기준으로 필터링해 조회할 수 있어야 한다.

`FR-018D` 시스템은 코드 변경, 리뷰 메타데이터, 빌드/테스트/정적분석 신호를 바탕으로 `AI` 자동 코드리뷰 초안을 생성할 수 있어야 한다.

`FR-018E` 시스템은 `AI` 자동 코드리뷰 초안에 대해 변경 요약, 잠재 이슈, 추가 확인 포인트, 참고 근거를 구조화해 표시할 수 있어야 한다.

`FR-018E-1` 시스템은 `AI` 자동 코드리뷰 수행 시 프로젝트 또는 개발팀에 지정된 코딩룰 규칙셋을 조회하고 적용할 수 있어야 한다.

`FR-018E-2` 시스템은 `AI` 자동 코드리뷰 결과에 대해 코딩룰별 준수 여부, 위반 의심 항목, 관련 코드 위치를 구조화해 표시할 수 있어야 한다.

`FR-018E-3` 시스템은 `AI` 자동 코드리뷰 결과에 어떤 코딩룰 규칙셋과 버전이 적용되었는지 식별할 수 있어야 한다.

`FR-018J` 시스템은 `PR/MR` 변경 범위를 기준으로 관련 단위테스트의 존재 여부와 실행 여부를 점검할 수 있어야 한다.

`FR-018K` 시스템은 `PR/MR` 변경 범위에 대해 적절한 단위테스트가 없다고 판단될 경우 `AI` 기반 단위테스트 초안을 생성할 수 있어야 한다.

`FR-018L` 시스템은 사용자가 지정한 파일, 디렉터리, 모듈 등 범위를 기준으로 `AI` 기반 단위테스트 초안을 생성할 수 있어야 한다.

`FR-018M` 시스템은 생성된 단위테스트 초안에 대해 대상 범위, 가정, 테스트 시나리오, 확인이 필요한 항목을 구조화해 표시할 수 있어야 한다.

`FR-018N` 시스템은 생성된 단위테스트를 실행하고 성공/실패, 실패 원인, 재생성 필요 여부를 표시할 수 있어야 한다.

`FR-018F` 시스템은 프로젝트 저장소의 언어, 빌드 도구, 컴파일러, 디렉터리 구조, 설정 파일 특성을 분석해 `CI` 플랜 초안 생성을 위한 입력 정보를 식별할 수 있어야 한다.

`FR-018G` 시스템은 미리 정의된 `stage`/`job` 템플릿을 바탕으로 프로젝트 특성에 맞는 `CI` 플랜 초안을 자동 생성할 수 있어야 한다.

`FR-018H` 시스템은 `AI` 기반 `CI` 플랜 초안에 대해 선택된 템플릿, 자동 결정된 변수, 사용자 확인이 필요한 항목을 구조화해 표시할 수 있어야 한다.

`FR-018I` 시스템은 공통 `CI` 플랜 초안을 `Jenkins` 와 `Bamboo Specs` 등 복수의 대상 형식으로 변환하거나 연결할 수 있어야 한다.

### 4.4 추적성 및 이력 관리

`FR-019` 시스템은 요구사항에서 변경, 빌드, 배포, 문서, 품질 결과, 테스트 결과까지 연결 관계를 조회할 수 있어야 한다.

`FR-020` 시스템은 데이터 변경 이력과 동기화 이력을 보관해야 한다.

`FR-021` 시스템은 승인 또는 상태 변경 이력을 감사 관점에서 확인할 수 있어야 한다.

`FR-021A` 시스템은 코드리뷰 상태, 산출물 상태, 정적분석 조치 상태의 변경 이력을 보관하거나 추적할 수 있어야 한다.

### 4.5 대시보드 및 보고

`FR-022` 시스템은 역할별 대시보드를 제공해야 한다.

`FR-023` 시스템은 프로젝트 진행 현황, 배포 현황, 품질 현황, 테스트 현황, 연계 상태를 요약해서 보여줄 수 있어야 한다.

`FR-024` 시스템은 조회 결과를 보고용 형식으로 내보낼 수 있어야 한다.

`FR-025` 시스템은 언어별 단위테스트 리포트를 수집해 지표화하고 추세를 조회할 수 있어야 한다.

`FR-026` 시스템은 외부 `KPI` 대시보드 또는 내부 지표 화면에서 활용할 수 있는 품질 지표 집계를 제공해야 한다.

`FR-026A` 시스템은 정적분석 결과와 테스트 결과, 빌드 상태를 결합한 품질 현황 요약을 제공할 수 있어야 한다.

### 4.6 사용자 및 권한 관리

`FR-027` 시스템은 사용자, 역할, 권한을 관리할 수 있어야 한다.

`FR-028` 시스템은 메뉴, 기능, 데이터 범위 수준의 접근 통제를 지원해야 한다.

`FR-029` 시스템은 관리자 전용 설정 기능과 일반 조회 기능을 분리해야 한다.

### 4.7 알림 및 예외 관리

`FR-030` 시스템은 동기화 실패, 품질 기준 초과, 배포 이상 등 주요 이벤트에 대해 알림을 제공할 수 있어야 한다.

`FR-031` 시스템은 예외 항목을 식별하고 조치 상태를 관리할 수 있어야 한다.

`FR-031A` 시스템은 정적분석 결과와 산출물, 코드리뷰 항목에 대해 예외 또는 조치 상태를 내부에서 관리할 수 있어야 한다.

`FR-031B` 시스템은 `AI` 자동 코드리뷰 초안에 대해 수용, 반려, 무시, 재생성 등의 처리 상태를 관리할 수 있어야 한다.

`FR-031B-1` 시스템은 코딩룰 위반 판단 결과에 대해 예외 승인, 규칙 오탐 처리, 재평가 요청 상태를 관리할 수 있어야 한다.

`FR-031C` 시스템은 `AI` 기반 `CI` 플랜 초안에 대해 승인 대기, 수정 필요, 확정, 재생성 등의 처리 상태를 관리할 수 있어야 한다.

`FR-031D` 시스템은 `AI` 기반 단위테스트 초안에 대해 생성됨, 검토 필요, 실행 완료, 수정 필요, 채택, 폐기 상태를 관리할 수 있어야 한다.

### 4.8 AI 에이전트 및 운영 도구 지원

`FR-032` 시스템은 운영자와 개발자가 주요 조회, 연계 관리, 상태 점검 작업을 자동화할 수 있도록 `MCP` 또는 `CLI` 도구를 제공해야 한다.

`FR-033` 시스템은 `AI` 에이전트가 사용할 수 있도록 주요 엔터티 조회, 동기화 실행, 상태 확인 기능에 대한 일관된 인터페이스를 제공해야 한다.

`FR-034` 시스템은 향후 자체 `ALM` 프로토타입 기능과 연계 도구가 동일한 표준 인터페이스를 사용할 수 있도록 설계되어야 한다.

`FR-035` 시스템은 `AI` 에이전트 또는 내부 `AI` 서비스가 코드리뷰 초안을 생성할 수 있도록 변경 집합, 품질 신호, 관련 메타데이터를 안전하게 전달하는 인터페이스를 제공해야 한다.

`FR-035-1` 시스템은 `AI` 코드리뷰 수행 시 코딩룰 규칙셋, 언어별 규칙, 팀별 예외 규칙을 함께 전달할 수 있어야 한다.

`FR-036` 시스템은 `AI` 에이전트 또는 내부 `AI` 서비스가 `CI` 플랜 초안을 생성할 수 있도록 프로젝트 분석 결과, 템플릿 정의, 필수 변수 정보를 안전하게 전달하는 인터페이스를 제공해야 한다.

`FR-037` 시스템은 `Bamboo` 연계 시 `Bamboo Specs` 기반으로 관리 가능한 계획 구조를 고려해 초안 또는 출력 결과를 제공할 수 있어야 한다.

`FR-038` 시스템은 `AI` 에이전트 또는 내부 `AI` 서비스가 단위테스트 초안을 생성하고 실행 판단을 보조할 수 있도록 변경 범위, 코드 문맥, 기존 테스트 현황, 언어별 테스트 규칙을 전달할 수 있어야 한다.

## 5. 비기능 요구사항

### 5.1 성능

`NFR-001` 일반 조회 화면은 목표 응답시간 기준을 정의해야 하며, 초기 가정으로 주요 조회는 3초 이내를 목표로 한다.

`NFR-002` 대량 동기화 수행 시 운영 서비스 조회 성능에 과도한 영향을 주지 않아야 한다.

### 5.2 가용성 및 안정성

`NFR-003` 시스템은 연계 실패가 발생해도 전체 서비스가 중단되지 않도록 격리되어야 한다.

`NFR-004` 실패한 동기화 작업은 재처리 가능해야 하며, 운영자가 실패 원인을 확인할 수 있어야 한다.

### 5.3 보안

`NFR-005` 시스템은 사용자 인증과 권한 검증을 수행해야 한다.

`NFR-006` 외부 시스템 접속 정보와 자격 증명은 안전하게 저장되어야 한다.

`NFR-007` 중요 조회 및 관리 행위에 대해 감사 로그를 남겨야 한다.

### 5.4 확장성

`NFR-008` 새로운 외부 시스템 유형이 추가될 때 기존 구조를 크게 변경하지 않고 연계할 수 있어야 한다.

`NFR-009` 프로젝트 수, 사용자 수, 수집 데이터량 증가에 대응 가능한 구조여야 한다.

### 5.5 사용성

`NFR-010` 역할별로 필요한 정보에 빠르게 접근할 수 있는 UI 구조를 제공해야 한다.

`NFR-011` 데이터 출처와 최신성, 상태값 의미를 사용자가 명확히 이해할 수 있어야 한다.

`NFR-012` 자체 `ALM` 프로토타입과 통합 조회 화면은 현재 사용 중인 도구들의 익숙한 흐름을 참고하되, 개발자 편의성을 최우선 기준으로 설계되어야 한다.

`NFR-013` 주요 화면은 개발자가 여러 도구를 오가지 않고도 맥락을 빠르게 파악할 수 있도록 탐색 단계 수를 최소화해야 한다.

### 5.6 AI 에이전트 활용성

`NFR-014` `AI` 에이전트용 인터페이스는 명령 구조와 반환 형식이 예측 가능해야 한다.

`NFR-015` `MCP`/`CLI` 도구는 운영과 개발 자동화에 사용할 수 있을 만큼 인증, 권한, 감사 기준을 충족해야 한다.

`NFR-016` `AI` 자동 코드리뷰 결과는 사람 작성 리뷰와 구분 가능해야 하며, 생성 시각과 사용한 입력 범위를 확인할 수 있어야 한다.

`NFR-017` `AI` 자동 코드리뷰는 사용자가 오탐 가능성을 인지할 수 있도록 주의 문구, 근거, 확신 수준 또는 이에 준하는 설명 정보를 제공할 수 있어야 한다.

`NFR-017-1` 코딩룰 기반 `AI` 리뷰 결과는 어떤 규칙을 근거로 판단했는지 추적 가능해야 하며, 규칙셋 변경 시 재평가 가능성을 고려해야 한다.

`NFR-018` `AI` 기반 `CI` 플랜 초안은 자동 생성 결과와 사람이 확정한 최종 설정을 구분해 표시할 수 있어야 한다.

`NFR-019` `AI` 기반 `CI` 플랜 초안은 어떤 템플릿과 프로젝트 분석 결과를 근거로 생성되었는지 추적 가능해야 한다.

`NFR-020` `AI` 기반 단위테스트 생성 결과는 생성 시각, 적용 범위, 사용한 입력 문맥, 실행 결과를 추적 가능해야 한다.

`NFR-021` `AI` 기반 단위테스트 실행 기능은 프로젝트별 실행 권한과 자원 사용 제한을 고려해야 한다.

## 6. 연계 요구사항

`IR-001` 외부 시스템별 인증 방식은 API Token, OAuth, 계정 기반 인증 등 복수 방식을 수용할 수 있어야 한다.

`IR-002` 연계 방식은 API, 파일, DB 연계 등 복수 인터페이스를 고려해야 한다.

`IR-003` 연계 대상별 데이터 수집 주기와 범위를 독립적으로 설정할 수 있어야 한다.

`IR-004` 외부 시스템 장애나 API 제한에 대응하기 위한 재시도, 지연, 예외 처리 정책이 필요하다.

`IR-005` 외부 시스템 원본 식별자를 유지하여 원본 시스템으로 역추적할 수 있어야 한다.

`IR-006` 단위테스트 리포트 수집은 언어 또는 빌드 도구별 포맷 차이를 수용할 수 있어야 한다.

`IR-007` `SCM`, `CI/CD`, 정적분석 계열 연계는 원본 시스템 링크를 유지해 통합 시스템에서 외부 상세 화면으로 이동할 수 있어야 한다.

`IR-008` 코드리뷰, 산출물, 정적분석 결과 관리 기능은 외부 원본 데이터와 내부 관리 상태를 함께 다룰 수 있는 매핑 구조를 가져야 한다.

`IR-009` `AI` 자동 코드리뷰 입력에는 변경 diff, 파일 메타데이터, 관련 빌드/테스트/정적분석 결과를 선택적으로 포함할 수 있어야 한다.

`IR-009-1` `AI` 자동 코드리뷰 입력에는 코딩룰 규칙셋, 언어별 규칙 매핑, 프로젝트별 예외 규칙을 포함할 수 있어야 한다.

`IR-010` `AI` 기반 `CI` 플랜 초안 입력에는 저장소 구조, 빌드 설정 파일, 언어/프레임워크 식별 정보, 템플릿 정의를 포함할 수 있어야 한다.

`IR-011` `CI` 연계는 `Jenkins` 와 `Bamboo` 를 동시에 지원할 수 있어야 하며, `Bamboo` 의 경우 `Bamboo Specs` 와의 정합성을 고려해야 한다.

`IR-012` `AI` 기반 단위테스트 생성 입력에는 변경 diff, 대상 파일, 관련 소스 문맥, 기존 테스트 파일, 언어별 테스트 프레임워크 정보를 포함할 수 있어야 한다.

## 7. 운영 및 관리 요구사항

`OR-001` 운영자는 연계 상태, 실패 내역, 처리 현황을 모니터링할 수 있어야 한다.

`OR-002` 운영자는 코드 수정 없이 일부 설정값을 관리할 수 있어야 한다.

`OR-003` 배포 환경별 설정 분리가 가능해야 한다.

`OR-004` 시스템 점검, 장애, 연계 중단 시 운영 공지 또는 상태 표시가 가능해야 한다.

`OR-005` 운영자는 연계 대상 시스템과 수집 대상 데이터 모델을 추가, 수정, 비활성화할 수 있어야 한다.

`OR-006` 운영자는 `AI` 에이전트용 `MCP`/`CLI` 도구의 사용 범위와 권한을 관리할 수 있어야 한다.

`OR-007` 운영자는 외부 `SCM`, `CI/CD`, 정적분석 도구별 연계 정책과 내부 관리 기능의 적용 범위를 구분해 설정할 수 있어야 한다.

`OR-008` 운영자는 `AI` 자동 코드리뷰의 적용 대상 프로젝트, 트리거 조건, 결과 저장 정책, 사용자 노출 범위를 설정할 수 있어야 한다.

`OR-008-1` 운영자 또는 권한 있는 관리자는 팀별 코딩룰 규칙셋을 등록, 수정, 버전 관리하고 프로젝트 또는 저장소에 매핑할 수 있어야 한다.

`OR-009` 운영자는 `AI` 기반 `CI` 플랜 초안 생성에 사용할 템플릿 세트, 자동 결정 규칙, 프로젝트별 예외 설정, 승인 절차를 관리할 수 있어야 한다.

`OR-010` 운영자는 프로젝트별로 `Jenkins`, `Bamboo`, 또는 복수 `CI` 연계 조합 중 적용 대상을 선택하고 관리할 수 있어야 한다.

`OR-011` 운영자는 `AI` 기반 단위테스트 기능의 적용 대상 프로젝트, 범위 지정 가능 수준, 실행 권한, 결과 저장 정책을 설정할 수 있어야 한다.

## 8. 데이터 및 거버넌스 요구사항

`DG-001` 공통 관리 대상의 표준 데이터 모델이 정의되어야 한다.

`DG-002` 데이터 품질 규칙과 필수값 검증 기준이 필요하다.

`DG-003` 시스템은 데이터 출처, 수집 시각, 변환 여부를 식별할 수 있어야 한다.

`DG-004` 보관 주기와 이력 관리 정책이 정의되어야 한다.

`DG-005` 용어 정의와 상태 코드 체계가 표준화되어야 한다.

`DG-006` 외부 시스템별 지표를 공통 `KPI`와 품질 지표 체계로 매핑할 수 있어야 한다.

`DG-007` 코드리뷰, 산출물, 정적분석 결과에 대해 원본 상태와 내부 관리 상태를 분리해 표현할 수 있는 표준 속성이 필요하다.

`DG-008` `AI` 자동 코드리뷰 결과를 저장할 경우 생성 모델, 입력 범위, 생성 시각, 처리 상태, 사용자 피드백을 추적할 수 있는 표준 속성이 필요하다.

`DG-008-1` 코딩룰 규칙셋은 규칙 ID, 설명, 적용 언어, 적용 범위, 버전, 예외 규칙, 활성 상태를 식별할 수 있어야 한다.

`DG-008-2` 코딩룰 기반 `AI` 리뷰 결과는 적용 규칙셋 버전, 위반 의심 규칙 ID, 판단 근거, 예외 승인 여부를 추적할 수 있어야 한다.

`DG-009` `AI` 기반 `CI` 플랜 초안을 저장할 경우 사용한 템플릿 버전, 분석된 프로젝트 특성, 자동 결정 변수, 승인 상태를 추적할 수 있는 표준 속성이 필요하다.

`DG-010` `CI` 플랜 초안은 공통 모델과 대상 도구별 출력 모델 간의 매핑 관계를 추적할 수 있어야 한다.

`DG-011` `AI` 기반 단위테스트 결과는 대상 범위, 생성된 테스트 파일 또는 초안, 실행 결과, 실패 로그, 사용자 채택 여부를 추적할 수 있어야 한다.

## 10. 오픈 이슈

- 테스트 관리 도구, 리뷰 보조 도구, 단위테스트 리포트 수집 대상의 우선순위는 무엇인가
- 프로젝트 단위 표준 식별자를 무엇으로 정의할 것인가
- 실시간 연계가 필요한 데이터와 배치로 충분한 데이터는 무엇인가
- 권한 모델을 조직 기준으로 할지 프로젝트 기준으로 할지
- 보고서와 감사 대응에 필요한 필수 지표는 무엇인가
- 원본 데이터 저장과 정규화 데이터 저장의 범위를 어디까지 가져갈 것인가
- 내부에서 직접 관리할 코드리뷰, 산출물, 정적분석 후속 조치의 범위를 어디까지 설정할 것인가
- `AI` 자동 코드리뷰의 기본 트리거를 `PR/MR` 생성 시점, 업데이트 시점, 수동 실행 중 무엇으로 둘 것인가
- `AI` 자동 코드리뷰 결과를 공식 리뷰 코멘트로 게시할지, 내부 초안으로만 유지할지
- 팀별 코딩룰을 자유 서술형 문서, 구조화 규칙셋, 또는 두 방식 혼합 중 무엇으로 관리할 것인가
- 코딩룰 위반 판단 결과를 권고 수준으로 둘지, 품질 게이트와 연동할지
- `PR` 변경 범위 기준 단위테스트 필요 여부를 어떤 규칙으로 판단할 것인가
- 사용자 지정 범위를 파일, 디렉터리, 모듈, 패키지 중 어디까지 지원할 것인가
- `AI` 가 생성한 단위테스트를 자동 커밋 후보로 둘지, 초안으로만 둘지
- `CI` 템플릿의 표준 단위를 프로젝트 유형, 언어, 빌드 도구, 배포 방식 중 무엇으로 둘 것인가
- `AI` 기반 `CI` 플랜 초안 결과를 `Jenkinsfile`, `Bamboo Specs`, 또는 동등한 설정으로 직접 변환할지, 내부 초안 단계로만 유지할지
- Phase 2에서 자체 `ALM`으로 우선 전환할 핵심 업무 흐름은 무엇인가
- `AI` 에이전트용 `MCP`/`CLI` 도구의 우선 제공 범위는 어디까지인가
