## 1. 기본 원칙

- 원천 조직 정보는 사내 인사 `DB` 변경을 기준으로 판단한다.
- 프로젝트와 과제는 `현재 조직 정보` 와 필요 시 `당시 조직 정보` 를 분리해 관리한다.
- 단순 표시값 변경은 최대한 자동 반영하되, 승인 체계나 책임 조직에 영향을 주는 변경은 검토 후 반영한다.
- 종료된 프로젝트와 보존 이력은 현재 조직 갱신과 별도로 다뤄야 한다.

## 2. 갱신 대상 구분

- 활성 프로젝트
- 종료 전환 중 프로젝트
- 종료 또는 보관 프로젝트
- 활성 과제
- 종료 또는 보존 과제
- 승인 대기 중 엔터티

| --- | --- | --- | --- | --- | --- |

## 4. 프로젝트 상태별 처리 기준

### 4.1 활성 프로젝트

- 현재 조직 정보는 최신 조직 기준으로 갱신 대상이 된다.
- 주관 조직, 참여 조직, 승인 조직이 모두 영향 범위에 포함될 수 있다.
- 승인 흐름이 열려 있는 경우 반영 전 영향 검토가 필요하다.

### 4.2 종료 전환 중 프로젝트

- 기본적으로 갱신 대상에 포함하되, 종료 승인 또는 보고 절차에 미치는 영향을 검토해야 한다.
- 필요 시 `갱신 보류` 상태로 분리할 수 있어야 한다.

### 4.3 종료 또는 보관 프로젝트

- 기본 원칙은 당시 조직 정보 유지다.
- 현재 조직 정보는 조회 편의를 위해 병기할 수 있으나 자동 치환은 지양한다.

## 5. 과제 상태별 처리 기준

### 5.1 활성 과제

- 현재 조직 참조 정보는 갱신 대상이다.
- 과제 책임자, 승인권자, 주관 조직이 함께 영향 분석 대상이 된다.

### 5.2 종료 또는 보존 과제

- 당시 조직 정보는 유지해야 한다.
- 현재 조직 정보는 선택적으로 병기할 수 있다.

## 6. 예외 처리 기준

- 법적 또는 감사 목적의 보존 데이터는 당시 조직 정보 변경 금지
- 조직 분리 시 대상 조직을 자동 확정하기 어려운 경우 수동 지정 필수
- 승인권자 공백이 발생하면 조직 갱신 완료 전에 대체 승인권자 지정 필요
- 외부 연계 시스템에서 조직 변경 수용이 불가능한 경우 내부 데이터만 우선 반영 가능

## 7. 운영자가 확인해야 할 체크리스트

- 해당 변경이 단순 명칭 변경인지 구조 변경인지
- 활성 프로젝트와 종료 프로젝트 중 무엇이 영향 받는지
- 과제 책임자 또는 승인권자 변경이 필요한지
- `Crowd` 또는 별도 계정계 후속 연계가 필요한지
- 예외 보존 대상이 포함되어 있는지

## 8. 권장 화면 또는 배치 기능

- 영향 프로젝트 목록 조회
- 영향 과제 목록 조회
- 자동 갱신 추천 목록
- 수동 지정 필요 목록
- 예외 제외 목록
- 마이그레이션 실행 이력 조회

## 9. 오픈 이슈

- 종료 프로젝트에도 현재 조직 병기를 기본 제공할지
- 조직 분리 시 자동 추천 기준을 조직장, 인력 구성, 프로젝트 속성 중 무엇으로 둘지
- 예외 제외 항목의 재검토 주기를 둘지
