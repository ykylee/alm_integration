# 조직 변경 감지 및 마이그레이션 프로세스

- 문서 목적: 인사 `DB` 의 조직 또는 인력 변경이 발생했을 때 시스템이 이를 감지하고, 관리자가 영향 범위를 검토한 뒤 시스템 데이터와 연관 계정 정보를 마이그레이션하는 절차를 정의한다.
- 범위: 변경 감지, 영향 분석, 관리자 검토, 내부 데이터 갱신, 계정계 후속 연계, 예외 처리
- 대상 독자: 운영자, 시스템 관리자, PMO, 아키텍트, 개발자

## 1. 프로세스 개요

본 프로세스는 조직 및 인력 기준정보의 원천이 사내 인사 `DB` 라는 전제를 따른다. 변경 반영의 기본 순서는 다음과 같다.

1. 인사 `DB` 변경 감지
2. 시스템 내부 조직/인력 데이터 반영 검토
3. 프로젝트 및 과제의 조직 참조 정보 갱신
4. `Crowd` 또는 별도 계정관리 시스템 후속 연계

`Crowd` 는 `Jira`, `Confluence`, `Bitbucket`, `Bamboo` 에 대한 계정 연계 체계로 한정하며, 조직 정보의 원천이 아니다.

## 2. 변경 유형

- 조직명 변경
- 상위 조직 변경
- 조직 통합
- 조직 분리
- 조직 폐지
- 조직장 변경
- 인력 소속 변경
- 인력 상태 변경

## 3. 단계별 절차

### 3.1 변경 감지

- 시스템은 배치 또는 수동 실행을 통해 인사 `DB` 의 변경분을 수집한다.
- 변경분 수집 결과에는 변경 유형, 대상 ID, 변경 전 값, 변경 후 값, 감지 시각이 포함되어야 한다.
- 감지 직후에는 자동 확정하지 않고 `검토 대기` 상태로 적재할 수 있어야 한다.

### 3.2 영향 범위 분석

- 시스템은 변경 대상과 연결된 조직, 인력, 프로젝트, 과제, 승인권자, 계정 연계를 식별해야 한다.
- 영향 분석 결과에는 최소한 다음 항목이 포함되어야 한다.
- 영향 조직 수
- 영향 인력 수
- 영향 프로젝트 수
- 영향 과제 수
- 영향 승인 라우팅 수
- 영향 `Crowd` 또는 외부 계정 연계 수

### 3.3 관리자 검토

- 관리자는 변경 유형과 영향 범위를 확인한 뒤 처리 방식을 결정해야 한다.
- 가능한 처리 방식은 다음과 같다.
- 전체 승인 후 일괄 반영
- 항목별 검토 후 선택 반영
- 일부 예외 제외 후 반영
- 반려 또는 보류
- 조직장, 승인권자, 주관 조직 변경은 일반 속성보다 높은 검토 우선순위를 가져야 한다.

### 3.4 시스템 내부 데이터 반영

- 승인된 변경은 내부 조직/인력 마스터에 먼저 반영되어야 한다.
- 내부 반영 시 변경 이력, 반영자, 반영 시각, 자동/수동 여부를 기록해야 한다.
- 조직 식별자가 변경되지 않는 단순 명칭 변경은 참조 무결성을 유지한 채 표시값만 갱신할 수 있어야 한다.
- 조직 통합/분리처럼 구조가 바뀌는 경우에는 신규 조직, 종료 조직, 이관 관계를 함께 기록해야 한다.

### 3.5 프로젝트 및 과제 조직 정보 갱신

- 내부 조직 마스터 반영 후, 관련 프로젝트와 과제의 현재 조직 참조 정보 갱신 대상을 산출해야 한다.
- 갱신 방식은 일괄 갱신, 수동 검토, 예외 제외 중 선택할 수 있어야 한다.
- 과거 이력 보존이 필요한 엔터티는 당시 조직 정보와 현재 조직 정보를 함께 유지해야 한다.

### 3.6 계정계 후속 연계

- 내부 데이터 반영이 완료된 후 `Crowd` 또는 별도 계정관리 시스템으로 후속 연계를 수행할 수 있어야 한다.
- `Crowd` 후속 연계는 `Jira`, `Confluence`, `Bitbucket`, `Bamboo` 계정 매핑에 한정한다.
- 계정계 연계 실패가 발생해도 내부 조직/인력 반영 결과는 별도로 보존되어야 한다.

## 4. 상태 초안

- `detected`: 인사 `DB` 변경 감지 완료
- `analyzed`: 영향 범위 분석 완료
- `review_pending`: 관리자 검토 대기
- `approved`: 내부 반영 승인
- `migrating_internal`: 시스템 내부 데이터 반영 중
- `migrating_project_refs`: 프로젝트/과제 조직 정보 갱신 중
- `migrating_accounts`: 계정계 후속 연계 중
- `completed`: 전체 처리 완료
- `partial_completed`: 일부 예외 또는 일부 실패 포함 완료
- `rejected`: 반영 반려
- `blocked`: 추가 확인 필요

## 5. 관리자 확인 화면에 필요한 정보

- 변경 대상 식별자
- 변경 유형
- 변경 전/후 값
- 영향 프로젝트 목록
- 영향 과제 목록
- 영향 조직장/승인권자 목록
- 영향 계정 연계 목록
- 권장 처리 방식
- 예외 처리 필요 항목

## 6. 예외 처리 원칙

- 조직 통합 또는 분리 시 사람이 최종 이관 대상을 확인해야 한다.
- 이미 종료된 프로젝트나 과거 보존 전용 과제는 자동 갱신에서 제외할 수 있어야 한다.
- 승인권자 공백이 생기는 경우에는 반영 완료 전에 대체 승인권자 지정이 필요하다.
- 계정계 연계 실패는 내부 반영 자체를 되돌리는 기준이 아니라 별도 후속 조치 대상으로 분리해야 한다.

## 7. 감사 및 이력 요구

- 변경 감지 시각
- 분석 실행 시각
- 검토자와 승인자
- 내부 반영 시각
- 프로젝트 조직 정보 갱신 시각
- 계정계 연계 실행 시각
- 실패 사유와 재시도 이력

## 8. 오픈 이슈

- 변경 감지 주기를 일 단위로 둘지, 더 짧은 배치로 둘지
- 조직 통합/분리 시 자동 이관 추천 규칙을 둘지
- 계정계 후속 연계 실패 시 자동 재시도 횟수를 어떻게 둘지
