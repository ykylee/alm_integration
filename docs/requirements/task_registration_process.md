- 대상 독자: 기획자, PM, 운영자, 개발자, 시스템 관리자

## 1. 문서 개요

우리 시스템은 과제를 중심으로 동작한다. 따라서 모든 관리 대상 업무는 과제 등록 절차를 통해 시스템 안으로 들어와야 한다. 다만 아직 정식 관리 범위에 포함되지 않은 프로젝트도 최소한의 정보는 등록할 수 있어야 하므로, 관리 외 프로젝트를 위한 간소 등록 절차를 별도로 둔다.

## 2. 등록 대상 구분

### 2.1 일반 과제

- 정식 관리 대상으로 운영되는 업무
- 요구사항, 형상, 빌드, 테스트, 문서, 품질 지표를 체계적으로 연결해야 하는 대상
- 승인된 등록 절차를 거쳐 과제 번호와 관리 범위를 부여받는 대상

### 2.2 관리 외 프로젝트

- 아직 정식 관리 범위에 포함되지 않은 프로젝트
- 최소 식별과 조회 목적의 간소 등록만 수행하는 대상
- 진행 중 필요성이 높아지면 관리 대상 프로젝트로 전환될 수 있는 대상

## 3. 일반 과제 등록 프로세스

### 3.1 목적

- 관리 대상 업무를 표준 절차로 시스템에 등록한다.
- 등록 시점부터 외부 연계와 추적의 기준점을 확보한다.

### 3.2 등록 단계

1. 등록 요청
2. 기본 정보 입력
3. 관리 대상 여부 검토
4. 승인
5. 과제 생성
6. 연계 설정 착수

### 3.3 필수 입력 항목 초안

- 과제명
- 과제 유형
- 담당 조직
- 요청자
- 목표 일정 또는 계획 기간
- 적용 대상 시스템 또는 제품
- 우선 연계 필요 시스템
- 관리 대상 사유
- 보안 또는 운영 등급

### 3.4 승인 결과

- 승인: 관리 대상 과제로 생성
- 보완 요청: 필수 정보 보완 후 재검토
- 반려: 등록 불가 또는 관리 외 프로젝트 등록으로 유도

### 3.5 등록 후 상태 초안

- `requested`
- `under_review`
- `approved`
- `rejected`
- `active`
- `closed`

## 4. 관리 외 프로젝트 등록 프로세스

### 4.1 목적

- 정식 관리 대상은 아니지만 최소 식별과 조회가 필요한 프로젝트를 빠르게 등록한다.
- 이후 관리 대상 전환 가능성을 열어 둔다.

### 4.2 등록 단계

1. 간소 등록 요청
2. 최소 정보 입력
3. 등록 적합성 확인
4. 관리 외 프로젝트 생성

### 4.3 최소 입력 항목 초안

- 프로젝트명
- 담당 조직 또는 담당자
- 관련 시스템
- 프로젝트 성격 요약
- 관리 외 등록 사유

### 4.4 등록 후 상태 초안

- `light_registered`
- `monitoring`
- `transition_requested`
- `converted`
- `closed`

## 5. 두 등록 프로세스의 차이

| 구분 | 일반 과제 등록 | 관리 외 프로젝트 등록 |
| --- | --- | --- |
| 목적 | 정식 관리 대상 편입 | 최소 식별 및 조회 |
| 입력 수준 | 상세 필수 정보 | 최소 정보 |
| 승인 수준 | 공식 승인 필요 | 간소 확인 또는 제한 승인 |
| 연계 범위 | 본격 연계 대상 | 제한적 조회 또는 최소 연계 |
| 후속 흐름 | 즉시 관리 체계 적용 | 필요 시 관리 대상 전환 |

## 6. 연계 시작 기준

- 일반 과제는 승인 이후 우선 연계 시스템 연결 작업에 착수한다.
- 관리 외 프로젝트는 기본적으로 최소 조회 수준만 허용한다.
- 관리 외 프로젝트가 관리 대상 프로젝트로 전환되기 전에는 깊은 연계나 자동화 적용 범위를 제한할 수 있다.

## 7. 오픈 이슈

- 일반 과제 등록 승인 주체를 조직장, PMO, 시스템 관리자 중 어디로 둘 것인가
- 관리 외 프로젝트의 간소 등록 권한을 누구까지 허용할 것인가
- 간소 등록 상태에서도 허용할 조회 범위와 연계 범위를 어디까지 둘 것인가
