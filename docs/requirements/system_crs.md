# 통합 중앙 관리 시스템 CRS

- 문서 목적: 과제의 컨셉과 상위 수준의 고객 요구를 `CRS` 형태로 정리한다.
- 범위: 프로젝트 배경, 목표, 범위, 핵심 개념, 사용자, 상위 요구
- 대상 독자: 기획자, 프로젝트 관리자, 아키텍트, 주요 이해관계자
- 상태: draft
- 최종 수정일: 2026-04-04
- 관련 문서: `docs/overview/project_overview.md`, `docs/requirements/system_srs.md`

## 1. 문서 개요

본 문서는 여러 운영 시스템의 정보를 중앙에서 관리하는 통합 시스템에 대한 컨셉 수준 요구를 정의한다. 상세한 기능 명세나 제약 조건은 `SRS`에서 다룬다.

## 2. 배경 및 문제 인식

조직 내 ALM, 형상관리, 문서관리, CI/CD, 분석 도구가 각각 분리되어 운영되면 프로젝트 현황, 변경 이력, 문서 상태, 배포 결과, 품질 지표가 도구별로 분산된다. 이로 인해 전체 상태 파악, 추적성 확보, 보고서 작성, 감사 대응, 운영 통제가 어려워진다.

## 3. 비전

본 시스템은 여러 이기종 시스템의 관리 정보를 수집하고 표준화하여 하나의 기준 화면과 데이터 모델로 제공하는 중앙 관리 플랫폼을 지향한다.

## 4. 목표

- 분산된 운영 정보를 통합 조회할 수 있는 단일 기준을 제공한다.
- 프로젝트, 변경, 산출물, 배포, 품질 간 관계를 연결해 추적성을 높인다.
- 외부 시스템별 메타데이터를 표준 모델로 정규화한다.
- 운영, 보고, 감사, 의사결정에 필요한 정보 접근성을 높인다.

## 5. 범위

### 포함 범위

- 외부 시스템 메타데이터 수집
- 공통 기준정보와 표준 식별 체계 관리
- 통합 조회, 검색, 대시보드, 보고 기능
- 동기화 상태 및 연계 실패 관리
- 사용자, 역할, 권한 기반 접근 통제

### 제외 또는 추후 검토 범위

- 외부 시스템의 고유 업무 기능 자체를 대체하는 것
- 모든 외부 시스템 전 기능의 복제
- 전체 조직 업무 프로세스의 즉시 표준화 완결
- 전사적 실시간 자동 제어 범위의 전면 도입

## 6. 핵심 개념

- 통합 관리 대상: 프로젝트, 애플리케이션, 산출물, 변경, 문서, 빌드, 배포, 품질 지표
- 연계 대상 시스템: ALM, 형상관리, 문서관리, CI/CD, 분석 도구, 필요 시 보안/ITSM/CMDB 계열
- 표준 모델: 외부 시스템의 식별자와 속성을 내부 표준 구조로 매핑한 공통 데이터 모델
- 추적성: 요구사항에서 변경, 빌드, 배포, 문서, 품질 결과까지 이어지는 관계 조회 능력

## 7. 주요 사용자

- 경영진/PMO: 프로젝트 상태와 성과를 요약해서 확인
- 프로젝트 관리자: 프로젝트 진행, 산출물, 품질, 이슈 상태 관리
- 개발/운영 담당자: 변경, 빌드, 배포, 장애, 품질 정보 추적
- 품질/감사 담당자: 승인 이력, 추적성, 준수 여부 확인
- 시스템 관리자: 연계 설정, 권한, 데이터 품질 관리

## 8. 상위 요구

### 8.1 정보 통합

- `CR-001` 시스템은 여러 외부 도구의 핵심 관리 정보를 통합 조회할 수 있어야 한다.
- `CR-002` 시스템은 외부 시스템별 데이터 차이를 내부 표준 기준으로 정규화할 수 있어야 한다.

### 8.2 추적성과 가시성

- `CR-003` 시스템은 프로젝트, 변경, 문서, 배포, 품질 간 연결 관계를 보여줄 수 있어야 한다.
- `CR-004` 시스템은 역할별로 필요한 요약 정보와 상세 정보를 제공할 수 있어야 한다.

### 8.3 운영 통제

- `CR-005` 시스템은 연계 상태, 오류, 동기화 결과를 관리할 수 있어야 한다.
- `CR-006` 시스템은 권한과 접근 범위를 통제할 수 있어야 한다.

### 8.4 확장성

- `CR-007` 시스템은 새로운 외부 시스템이 추가되더라도 연계 구조를 확장할 수 있어야 한다.
- `CR-008` 시스템은 데이터량과 사용자 수 증가에 대응 가능한 방향으로 설계되어야 한다.

## 9. 성공 기준

- 운영자가 여러 시스템을 직접 오가지 않고 주요 상태를 파악할 수 있어야 한다.
- 데이터 출처와 최신성, 연결 관계가 사용자에게 명확히 보여야 한다.
- 보고와 감사 대응에 활용 가능한 수준의 이력성이 제공되어야 한다.
- 새로운 연계 대상을 추가할 때 구조적 재작업이 최소화되어야 한다.

## 10. 가정 및 미정 사항

- 실제 연계 대상 제품과 인터페이스 방식은 확정되지 않았다.
- 사용자 수, 조직 구조, 보안 정책, 배포 환경은 추가 분석이 필요하다.
- 실시간 연계와 배치 연계의 경계는 세부 설계 단계에서 정해야 한다.

## 11. SRS 연계 기준

- `CR-001`, `CR-002` 는 `SRS`의 통합 대상 관리, 데이터 수집, 통합 조회 요구로 세분화한다.
- `CR-003`, `CR-004` 는 `SRS`의 추적성, 대시보드, 검색 요구로 세분화한다.
- `CR-005`, `CR-006` 는 `SRS`의 운영 관리, 권한, 감사, 예외 관리 요구로 세분화한다.
- `CR-007`, `CR-008` 는 `SRS`의 확장성, 안정성, 연계 아키텍처 요구로 세분화한다.
