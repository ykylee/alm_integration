## 1. 문서 개요

본 문서는 기존 `SDLC` 인프라를 통합하고, 이후 자체 `ALM` 프로토타입으로 확장하려는 과제에 대한 컨셉 수준 요구를 정의한다. 상세한 기능 명세나 제약 조건은 `SRS`에서 다룬다.

## 2. 배경 및 문제 인식

현재 운영 환경에는 `Jira`, `Confluence`, `Gitea`, `Jenkins`, `Bamboo`, `Coverity`, 테스트 관리 도구, `KPI` 대시보드, 코딩 룰 검사 및 리뷰 보조 도구, 언어별 단위테스트 리포트 수집 체계가 공존한다. 이로 인해 프로젝트 현황, 변경 이력, 문서 상태, 배포 결과, 테스트 결과, 품질 지표가 도구별로 분산되고 전체 흐름 파악과 추적성 확보가 어렵다.

## 3. 비전

본 시스템은 두 단계 비전을 가진다.

- Phase 1: 여러 이기종 시스템의 관리 정보를 수집하고 표준화하여 하나의 기준 화면과 데이터 모델로 제공하는 중앙 관리 플랫폼 구축
- Phase 2: Phase 1의 통합 정보와 사용자 흐름을 기반으로, 현재 사용 중인 도구의 일부 관리 기능을 점진적으로 내재화할 수 있는 자체 `ALM` 프로토타입 구축

## 4. 목표

- 분산된 운영 정보를 통합 조회할 수 있는 단일 기준을 제공한다.
- 프로젝트, 변경, 산출물, 배포, 품질 간 관계를 연결해 추적성을 높인다.
- 외부 시스템별 메타데이터를 표준 모델로 정규화한다.
- 운영, 보고, 감사, 의사결정에 필요한 정보 접근성을 높인다.
- 관리자가 연계 대상 시스템과 관리 대상을 직접 추가, 수정할 수 있는 운영 유연성을 제공한다.
- 개발자 편의성을 최우선으로 고려한 자체 `ALM` 프로토타입 방향을 구체화한다.
- `SCM`, `CI/CD`, 정적분석 실행 계층은 외부 전문 도구를 우선 활용하고, 코드리뷰, 빌드/산출물, 정적분석 결과 관리 경험은 내부에서 강화한다.
- 코드리뷰 과정에서 `AI` 가 자동 리뷰 초안을 작성해 개발자가 위험 변경과 확인 포인트를 빠르게 파악할 수 있도록 한다.
- `AI` 자동 코드리뷰는 개발팀 또는 프로젝트에서 지정한 코딩룰을 기준으로 준수 여부를 판단할 수 있어야 한다.
- `AI` 는 `PR` 변경 범위 또는 사용자가 지정한 범위를 기준으로 단위테스트 존재 여부를 점검하고, 필요 시 단위테스트 초안 작성과 테스트 수행을 보조할 수 있어야 한다.
- 프로젝트 초기 설정 단계에서 `AI` 가 소스코드 언어, 컴파일러, 구성 특성을 분석해 표준 템플릿 기반 `CI` 플랜 초안을 작성하고, 필요 시 `Jenkins` 또는 `Bamboo Specs` 형식으로 연결할 수 있도록 한다.
- 시스템 운영, 사용, 유지보수, 개발 전반에서 `AI` 에이전트를 활용할 수 있도록 `MCP` 또는 `CLI` 도구 제공 체계를 마련한다.

## 5. 범위

### 포함 범위

- 외부 시스템 메타데이터 수집
- 공통 기준정보와 표준 식별 체계 관리
- 통합 조회, 검색, 대시보드, 보고 기능
- 동기화 상태 및 연계 실패 관리
- 사용자, 역할, 권한 기반 접근 통제
- 테스트 관리 정보, 단위테스트 리포트, 정적분석 결과, `KPI` 지표 수집 및 관리
- 연계 시스템 유형과 대상 데이터를 관리자가 설정 가능한 구조
- 자체 `ALM` 프로토타입을 위한 사용자 흐름과 화면 컨셉 정의
- `AI` 에이전트 연동을 위한 운영용 인터페이스와 도구 방향 정의

### 제외 또는 추후 검토 범위

- Phase 1에서 외부 시스템의 고유 업무 기능 자체를 즉시 대체하는 것
- 모든 외부 시스템 전 기능의 복제
- 전체 조직 업무 프로세스의 즉시 표준화 완결
- 전사적 실시간 자동 제어 범위의 전면 도입

## 6. 핵심 개념

- 통합 관리 대상: 프로젝트, 애플리케이션, 산출물, 변경, 문서, 빌드, 배포, 품질 지표
- 연계 대상 시스템: `Jira`, `Confluence`, `Gitea`, `Jenkins`, `Bamboo`, `Coverity`, 테스트 관리 도구, `KPI` 대시보드, 코딩 룰 검사/리뷰 보조 도구, 단위테스트 리포트 수집 체계, 필요 시 보안/`ITSM`/`CMDB` 계열
- 표준 모델: 외부 시스템의 식별자와 속성을 내부 표준 구조로 매핑한 공통 데이터 모델
- 추적성: 요구사항에서 변경, 빌드, 배포, 문서, 품질 결과까지 이어지는 관계 조회 능력
- 관리자 확장성: 연계 대상 시스템과 수집 데이터 범위를 관리자가 직접 조정할 수 있는 능력
- 실행 계층 분리: 외부 전문 도구는 실행과 원본 관리를 담당하고, 통합 시스템은 조회, 연결, 일부 관리 기능을 담당하는 구조
- `AI` 친화 운영: 사람뿐 아니라 `AI` 에이전트도 시스템 운영과 분석 업무를 수행할 수 있도록 인터페이스를 제공하는 방향

## 7. 주요 사용자

- 경영진/PMO: 프로젝트 상태와 성과를 요약해서 확인
- 프로젝트 관리자: 프로젝트 진행, 산출물, 품질, 이슈 상태 관리
- 개발/운영 담당자: 변경, 빌드, 배포, 장애, 품질 정보 추적
- 품질/감사 담당자: 승인 이력, 추적성, 준수 여부 확인
- 시스템 관리자: 연계 설정, 권한, 데이터 품질 관리
- `AI` 에이전트 활용 담당자: 운영 자동화, 분석, 유지보수, 개발 지원용 도구 활용

## 8. 상위 요구

### 8.1 정보 통합

- `CR-001` 시스템은 여러 외부 도구의 핵심 관리 정보를 통합 조회할 수 있어야 한다.
- `CR-002` 시스템은 외부 시스템별 데이터 차이를 내부 표준 기준으로 정규화할 수 있어야 한다.
- `CR-003` 시스템은 테스트 관리 정보, 단위테스트 리포트, 정적분석 결과, `KPI` 정보를 함께 연결 관리할 수 있어야 한다.

### 8.2 추적성과 가시성

- `CR-004` 시스템은 프로젝트, 변경, 문서, 배포, 품질, 테스트 간 연결 관계를 보여줄 수 있어야 한다.
- `CR-005` 시스템은 역할별로 필요한 요약 정보와 상세 정보를 제공할 수 있어야 한다.

### 8.3 운영 통제

- `CR-006` 시스템은 연계 상태, 오류, 동기화 결과를 관리할 수 있어야 한다.
- `CR-007` 시스템은 권한과 접근 범위를 통제할 수 있어야 한다.
- `CR-008` 시스템은 관리자가 연계 대상 시스템과 데이터 범위를 추가, 수정, 비활성화할 수 있어야 한다.

### 8.4 확장성과 진화

- `CR-009` 시스템은 새로운 외부 시스템이 추가되더라도 연계 구조를 확장할 수 있어야 한다.
- `CR-010` 시스템은 데이터량과 사용자 수 증가에 대응 가능한 방향으로 설계되어야 한다.
- `CR-011` 시스템은 Phase 2에서 자체 `ALM` 프로토타입으로 확장할 수 있는 방향으로 설계되어야 한다.
- `CR-011A` 시스템은 `SCM`, `CI/CD`, 정적분석 실행 계층을 외부 전문 도구와 연계하는 구조를 우선 지원해야 한다.
- `CR-011B` 시스템은 코드리뷰, 빌드/산출물 메타데이터, 정적분석 결과와 같은 개발 맥락 정보의 내부 조회 및 관리 기능을 확장할 수 있어야 한다.
- `CR-011C` 시스템은 코드 변경 정보와 품질 신호를 바탕으로 `AI` 자동 코드리뷰 초안을 생성하고 제공할 수 있어야 한다.
- `CR-011C-1` 시스템은 `AI` 자동 코드리뷰 수행 시 프로젝트 또는 팀별 코딩룰 규칙셋을 적용해 준수 여부를 판단할 수 있어야 한다.
- `CR-011F` 시스템은 `PR` 변경 범위 또는 사용자 지정 범위를 기준으로 `AI` 기반 단위테스트 초안을 생성하고 테스트 수행을 보조할 수 있어야 한다.
- `CR-011D` 시스템은 프로젝트 소스 특성과 표준 템플릿을 바탕으로 `AI` 기반 `CI` 플랜 초안을 생성하고 제공할 수 있어야 한다.
- `CR-011E` 시스템은 `CI` 플랜 초안을 공통 모델로 관리하면서 `Jenkins` 와 `Bamboo Specs` 같은 복수의 외부 `CI` 설정 형식으로 연결할 수 있어야 한다.

### 8.5 사용자 경험과 AI 활용

- `CR-012` 시스템은 기존 사용 도구의 `UX`를 참고하되 개발자 편의성을 최우선으로 고려해야 한다.
- `CR-013` 시스템은 운영, 사용, 유지보수, 개발 전반에서 `AI` 에이전트가 활용될 수 있도록 `MCP` 또는 `CLI` 도구를 제공할 수 있어야 한다.
- `CR-014` 시스템은 `AI` 자동 코드리뷰가 사람 리뷰를 보조하는 방향으로 동작하며, 결과의 출처와 한계를 사용자에게 명확히 보여줄 수 있어야 한다.
- `CR-016` 시스템은 코딩룰 기반 `AI` 리뷰 결과에 대해 적용된 규칙셋과 판단 근거를 사용자에게 명확히 보여줄 수 있어야 한다.
- `CR-015` 시스템은 `AI` 기반 `CI` 플랜 초안이 템플릿 기반 제안이라는 점과 자동 결정 근거를 사용자에게 명확히 보여줄 수 있어야 한다.
- `CR-017` 시스템은 `AI` 기반 단위테스트 보조 결과에 대해 적용 범위, 생성된 테스트, 실행 결과, 사용자 확인 필요 항목을 명확히 보여줄 수 있어야 한다.

## 9. 성공 기준

- 운영자가 여러 시스템을 직접 오가지 않고 주요 상태를 파악할 수 있어야 한다.
- 데이터 출처와 최신성, 연결 관계가 사용자에게 명확히 보여야 한다.
- 보고와 감사 대응에 활용 가능한 수준의 이력성이 제공되어야 한다.
- 새로운 연계 대상을 추가할 때 구조적 재작업이 최소화되어야 한다.
- 개발자 입장에서 기존 도구 조합보다 더 빠르게 필요한 맥락을 찾을 수 있어야 한다.
- `AI` 에이전트 활용 시 운영과 개발 지원 작업의 자동화 가능성이 확보되어야 한다.

## 10. 가정 및 미정 사항

- 테스트 관리 도구, 리뷰 보조 도구, 단위테스트 수집 체계의 구체 제품과 인터페이스 방식은 추가 확정이 필요하다.
- 코드리뷰와 산출물 관리 기능의 내부 제공 범위는 추가 분석이 필요하다.
- `AI` 자동 코드리뷰의 공식 리뷰 반영 범위와 책임 분담 방식은 추가 분석이 필요하다.
- 팀별 코딩룰 규칙셋의 등록, 변경, 예외 처리 책임 주체는 추가 분석이 필요하다.
- `AI` 기반 단위테스트 생성과 실행 기능의 적용 범위, 자동 반영 수준, 실패 처리 책임 주체는 추가 분석이 필요하다.
- `AI` 기반 `CI` 플랜 초안의 승인 책임, 템플릿 관리 주체, 자동 결정 범위는 추가 분석이 필요하다.
- `CI` 공통 템플릿과 `Jenkins`/`Bamboo Specs`별 출력 규칙의 분리 기준은 추가 분석이 필요하다.
- 사용자 수, 조직 구조, 보안 정책, 배포 환경은 추가 분석이 필요하다.
- 실시간 연계와 배치 연계의 경계는 세부 설계 단계에서 정해야 한다.
- Phase 2에서 우선 대체할 자체 `ALM` 기능 범위는 현행 사용 패턴 분석 후 확정한다.

## 11. SRS 연계 기준

- `CR-001` ~ `CR-003` 은 `SRS`의 통합 대상 관리, 데이터 수집, 테스트/품질 지표 관리 요구로 세분화한다.
- `CR-004`, `CR-005` 는 `SRS`의 추적성, 대시보드, 검색 요구로 세분화한다.
- `CR-006` ~ `CR-008` 은 `SRS`의 운영 관리, 권한, 관리자 설정 요구로 세분화한다.
